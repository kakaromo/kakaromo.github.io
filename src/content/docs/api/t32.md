## T32 Config CRUD

Base path: `/api/t32`

| Method | Path | 설명 |
|--------|------|------|
| GET | `/configs` | 전체 T32 설정 목록 (DTO, 서버 이름 해석 포함) |
| POST | `/configs` | T32 설정 추가 |
| PUT | `/configs/{id}` | T32 설정 수정 (비밀번호 빈 문자열이면 기존 값 유지) |
| DELETE | `/configs/{id}` | T32 설정 삭제 (cascade로 매핑도 삭제) |

### 요청 본문 (POST/PUT)

```json
{
  "serverGroupId": 1,
  "jtagServerId": 5,
  "jtagUsername": "",
  "jtagPassword": "",
  "t32PcId": 6,
  "t32PcUsername": "admin",
  "t32PcPassword": "secret",
  "jtagCommand": "cd T32linker && echo yes | python ./tmux_lib.py -n 1 -t {tentacle_num} -s {slot}",
  "jtagSuccessPattern": "Measured Voltage.*Register Value",
  "t32RemoteHost": "192.168.1.60",
  "t32RemotePort": 50551,
  "t32PortCheckCommand": "20000",
  "t32StartCommand": "C:\\T32\\bin\\windows64\\t32mp64.exe -c C:\\T32\\config.t32",
  "coreScriptsJson": "[{\"controller\":\"S5E9945\",\"canaryRelPath\":\"00_BUILD\\\\00_SIMULATOR\\\\CANARY\",\"cores\":[{\"core\":\"H-Core\",\"cmmRelPath\":\"scripts/h_core.cmm\",\"optionalCommands\":\"\"}]}]",
  "dumpCommand": "cmd /c C:\\T32\\dump.bat {branch_path} {result_path}",
  "fwCodeLinuxBase": "/appdata/samsung/OCTO_HEAD/FW_Code",
  "fwCodeWindowsBase": "F:\\FW_Code",
  "resultBasePath": "/appdata/samsung/OCTO_HEAD/T32_dump",
  "resultWindowsBasePath": "F:\\T32_dump",
  "description": "Lab A T32 설정",
  "enabled": true,
  "assignedServerIds": [3, 4, 5]
}
```

:::note[gRPC 경로 vs legacy]
`t32RemoteHost` + `t32RemotePort`가 채워지면 Step 2·3을 **t32remote gRPC**로 수행합니다. 이때 `t32PortCheckCommand`는 명령이 아니라 **RCL 포트 번호**로 해석되고, Step 3은 `coreScriptsJson`에서 **controller(SoC)별 그룹**을 골라 그 그룹의 core별 cmm을 실행합니다. 비어 있으면 `dumpCommand`로 legacy SSH+bat 경로를 씁니다.
:::

### coreScriptsJson 포맷 (controller 그룹)

`coreScriptsJson`은 controller(SoC)마다 cmm 위치·Canary 경로가 달라 **controller 그룹** 배열로 저장합니다:

```json
[
  {
    "controller": "S5E9945",
    "canaryRelPath": "00_BUILD\\00_SIMULATOR\\CANARY",
    "cores": [
      { "core": "H-Core", "cmmRelPath": "scripts/h_core.cmm", "optionalCommands": "" },
      { "core": "Canary", "cmmRelPath": "scripts/canary.cmm", "optionalCommands": "" }
    ]
  }
]
```

| 키 | 설명 |
|----|------|
| `controller` | slot/HEAD의 controller 문자열과 매칭하는 키 (대소문자 무시). 비면 controller 없는 slot의 fallback 그룹 |
| `canaryRelPath` | Canary 폴더의 branch base 기준 상대경로. 비면 기본값 `00_BUILD\00_SIMULATOR\CANARY` |
| `cores[]` | 이 controller의 core별 cmm 스크립트 (`core`, `cmmRelPath`, `optionalCommands`) |

- **구 포맷 호환**: flat core 목록 `[{core, cmmRelPath, optionalCommands}]`도 그대로 동작 — controller 없는 단일 그룹으로 감싸짐. 스키마/마이그레이션 변경 없음(TEXT 컬럼).
- **그룹 선택**: dump 시 요청 본문의 `controller`로 그룹을 고름. 그룹이 하나면 자동 선택, controller가 없거나 안 맞으면 `controller-needed` SSE 이벤트로 사용자에게 선택을 요청.

### 필드 설명

| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| `serverGroupId` | Long | O | Lab 서버 그룹 ID |
| `jtagServerId` | Long | O | TMUX 서버 ID (portal_servers) |
| `jtagUsername` | String | | TMUX 전용 SSH 계정 (비워두면 서버 기본) |
| `jtagPassword` | String | | TMUX 전용 SSH 비밀번호 (비워두면 서버 기본) |
| `t32PcId` | Long | O | T32 PC 서버 ID (portal_servers) |
| `t32PcUsername` | String | | T32 PC 전용 SSH 계정 (비워두면 서버 기본) |
| `t32PcPassword` | String | | T32 PC 전용 SSH 비밀번호 (비워두면 서버 기본) |
| `jtagCommand` | String | | JTAG 연결 명령어 (`{tentacle}`, `{tentacle_num}`, `{slot}`) |
| `jtagSuccessPattern` | String | | Step 1 성공 판정 정규식 (선택) |
| `t32RemoteHost` | String | | t32remote.exe 호스트 (채우면 gRPC 경로) |
| `t32RemotePort` | Integer | | t32remote.exe 포트 (예: 50551) |
| `t32PortCheckCommand` | String | | gRPC 경로: RCL 포트 번호 / legacy: Attach 확인 명령 |
| `t32StartCommand` | String | | JTAG 변경 후 PowerView 재시작 명령 (PsExec) |
| `coreScriptsJson` | String | | gRPC 경로: controller 그룹 JSON `[{controller, canaryRelPath, cores:[...]}]` (구 flat 포맷 호환) |
| `dumpCommand` | String | | legacy(SSH+bat) 전용 (`{result_path}`, `{branch_path}`) |
| `fwCodeLinuxBase` | String | | Linux FW 코드 기본 경로 |
| `fwCodeWindowsBase` | String | | Windows FW 코드 기본 경로 |
| `resultBasePath` | String | | Linux 결과 저장 기본 경로 |
| `resultWindowsBasePath` | String | | Windows 결과 저장 기본 경로 |
| `description` | String | | 메모 |
| `enabled` | boolean | O | 활성화 여부 |
| `assignedServerIds` | Long[] | | 담당 텐타클 서버 ID 목록 |

## T32 Dump 실행

Base path: `/api/t32/dump`

### Config 확인

```
GET /check?serverId={id}
```

해당 서버에 T32 설정이 있는지 확인합니다. 컨텍스트 메뉴 활성화 판단용.

```json
{ "available": true, "configId": 1 }
```

### Dump 실행 (SSE)

```
POST /execute
Content-Type: application/json
→ text/event-stream;charset=UTF-8 (timeout: 600초)
```

### 요청 본문

```json
{
  "serverId": 5,
  "tentacleName": "T10",
  "slotNumber": 1,
  "fw": "Savona_V8",
  "branchPath": "/appdata/samsung/OCTO_HEAD/FW_Code/Savona/Savona_V8_P00RC28",
  "setLocation": "T10-1",
  "testToolName": "randwrite",
  "testTrName": "Savona_V8_TLC_512Gb_512GB_P00RC28",
  "historyId": 1234,
  "testType": "COMPATIBILITY",
  "source": "INTERNAL",
  "controller": "S5E9945"
}
```

| 필드 | 타입 | 설명 |
|------|------|------|
| `serverId` | Long | 텐타클 서버 ID (T32Config 매핑 조회용) |
| `tentacleName` | String | 텐타클 이름 (T10 등) |
| `slotNumber` | int | 슬롯 번호 |
| `fw` | String | FW 버전 (미사용 시 빈 문자열) |
| `branchPath` | String | FW 소스 폴더 Linux 경로 (Windows로 자동 변환) |
| `setLocation` | String | 슬롯 위치 (결과 폴더명 조합용) |
| `testToolName` | String | 테스트 도구명 (결과 폴더명 조합용) |
| `testTrName` | String | TR 이름 (결과 폴더명 조합용) |
| `historyId` | Long | 결과 매칭용 history ID (nullable) |
| `testType` | String | `COMPATIBILITY` / `PERFORMANCE` (nullable, 기본 COMPATIBILITY) |
| `source` | String | `INTERNAL` / `OUTSOURCED` (nullable, 기본 INTERNAL) |
| `controller` | String | slot/HEAD의 controller(SoC). gRPC 경로에서 `coreScriptsJson` 그룹 선택 (nullable) |

`historyId`/`testType`/`source`는 결과 ZIP을 history와 매칭해 MinIO에 기록할 때 사용합니다. 없으면 업로드는 하되 `historyId`를 NULL로 둡니다.

`controller`는 gRPC 경로에서 `coreScriptsJson`의 controller 그룹을 고르는 데 씁니다. 빈 문자열이면 백엔드가 자동 선택을 시도하고(그룹 1개면 그것), 모호하면 `controller-needed` 이벤트로 되묻습니다. controller를 골라 다시 `/execute`를 호출하면 됩니다.

### 결과 폴더명 조합

```
{datetime}_{setLocation}_{testToolName}_{testTrName}
```

예시: `20260623_14h05m22s_T10-1_randwrite_Savona_V8_TLC_512Gb_512GB_P00RC28`

빈 값은 건너뛰어 짧아질 수 있습니다. 날짜+시·분·초까지 포함해 같은 브랜치 반복 dump도 구분됩니다.

### SSE 이벤트

| 이벤트 | 데이터 | 시점 |
|--------|--------|------|
| `locked` | `{lockedBy: string, since: string}` | 다른 사용자가 이미 점유 중 (작업 없이 종료) |
| `controller-needed` | `{message: string, slotController: string, options: string[]}` | controller 그룹을 결정 못함 → 사용자가 선택 후 재실행 (이후 `done {failedStep:3}`) |
| `step-start` | `{step: number, name: string}` | 각 Step 시작 |
| `step-output` | `{step: number, line: string}` | 실시간 출력 라인 (SSH stdout / ProgressEvent) |
| `step-done` | `{step: number, status: "success"\|"failed", output: string}` | Step 완료 또는 실패 |
| `dump-progress` | `{step: 3, core: string, phase: string, status: string}` | Step 3 core별 진행 |
| `done` | `{success: boolean, resultPath: string, resultWindowsPath: string, failedStep?: number}` | 전체 완료 |
| `cancelled` | `{message: string}` | 사용자 중단 |
| `error` | `{message: string}` | 예외 오류 |

### Step별 성공 판정 (gRPC 경로)

| Step | 성공 조건 |
|------|-----------|
| 1 (JTAG) | `jtagSuccessPattern` 매칭 → "Measured Voltage" > 0 → exitCode=0 (우선순위) |
| 2 (Attach) | `attach`의 `t32_version` ≠ 빈값 AND `STATE.TARGET()` 정상 |
| 3 (Dump) | 선택된 controller 그룹의 core별 `ExecuteCommand` 전부 성공 |
| 4 (업로드) | (soft) MinIO 업로드 — 실패해도 dump는 성공 처리 |

legacy(SSH+bat): Step 2는 stdout "Down" 미포함, Step 3은 stdout "fail" 미포함(대소문자 무시).

### 타임아웃

| Step | 제한 시간 |
|------|-----------|
| Step 1 (JTAG) | 30초 |
| Step 2 (Attach) | 30초 |
| Step 3 (Dump) | 300초 (명령당) |
| SSE emitter 전체 | 600초 (10분) |

### Dump 중단

```
POST /cancel?serverId={id}
```

진행 중인 dump 워커를 interrupt해 단독 점유 lock·t32remote 세션을 즉시 정리합니다. SSE fetch abort만으로는 백엔드가 끊김을 늦게 감지하므로 명시적으로 호출합니다. 점유자(요청 사용자)가 일치할 때만 중단됩니다.

```json
{ "cancelled": true }
```

## T32 결과 조회

Base path: `/api/t32/results`

### 최신 결과 아티팩트

```
GET /latest?testType={type}&source={source}&historyId={id}
```

주어진 history의 최신 업로드 완료(`COMPLETED`) 결과 ZIP 1건을 반환합니다. 프론트는 응답의 `bucket`+`objectKey`로 기존 MinIO presigned 다운로드를 사용합니다.

| 파라미터 | 타입 | 필수 | 설명 |
|----------|------|------|------|
| `testType` | String | O | `COMPATIBILITY` / `PERFORMANCE` |
| `source` | String | | `INTERNAL` / `OUTSOURCED` (기본 INTERNAL) |
| `historyId` | Long | O | 매칭 history ID |

응답 (`T32ResultArtifactDto`):

```json
{
  "id": 42,
  "testType": "COMPATIBILITY",
  "source": "INTERNAL",
  "historyId": 1234,
  "bucket": "t32-results",
  "objectKey": "COMPATIBILITY/INTERNAL/1234/20260623_14h05m22s_T10-1_randwrite_....zip",
  "sizeBytes": 18234221,
  "resultDirName": "20260623_14h05m22s_T10-1_randwrite_...",
  "status": "COMPLETED",
  "uploadedAt": "2026-06-23 14:08:10"
}
```

COMPLETED 결과가 없으면(아직 UPLOADING / FAILED) **404**를 반환합니다.

## TC Pre-Command API

Base path: `/api/pre-commands`

### TC별 조회

```
GET /tc?source={source}&slotIndex={slotIndex}
```

```json
[
  { "tcId": 34, "preCommandId": 1, "preCommandName": "Init UFS" },
  { "tcId": 38, "preCommandId": 2, "preCommandName": "Hang for T32" }
]
```

### TC별 등록

```
POST /tc/assign
```

```json
{
  "preCommandId": 1,
  "source": "compatibility",
  "slotIndex": 3,
  "tcId": 34
}
```

### TC별 해제

```
POST /tc/unassign
```

```json
{
  "source": "compatibility",
  "slotIndex": 3,
  "tcId": 34
}
```

### 동작 규칙

- TC에 Pre-Command가 있으면 → 슬롯 Pre-Command 무시 (TC 우선)
- 슬롯 clear 시 → 해당 슬롯의 모든 TC Pre-Command 자동 삭제
- TC 삭제 시 → 해당 TC의 Pre-Command 자동 삭제
