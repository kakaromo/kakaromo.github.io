## 30초 요약

컨트롤러는 **세 줄의 조립 코드**가 거의 전부입니다.

1. `resultDataService.fetchResultData(historyId)` → 데이터 수집
2. `excelGrpcClient.generateExcel(...)` → Go 서비스 호출
3. `ResponseEntity<byte[]>` + Content-Disposition + xlsx MIME → 브라우저 다운로드

프론트는 **`window.location.href`로 직접 링크**만 열면 끝 — fetch + Blob 변환 없이 브라우저가 자동 다운로드.

## 왜 스트리밍이 아닌 byte[]인가

Go 서비스가 **완성된 xlsx 바이너리 통째로** gRPC response에 담아 보내므로, Spring은 이미 메모리에 로드된 byte[]를 그대로 응답. 스트리밍의 이점이 없음.

이상적으로 Go 서비스가 **chunked response**로 바꾸고 Java가 `StreamingResponseBody`로 받으면 메모리 피크가 낮아집니다. 하지만:

- 현재 xlsx가 수 MB 수준 → 메모리 부담 크지 않음
- 스트리밍으로 바꾸려면 proto에 `stream ExcelResponse` 변경 + 양쪽 구현
- 성능 병목으로 드러난 적 없음 → 미리 최적화 안 함

**현재 사용 패턴에 맞는 가장 단순한 구현**. 커지면 바꾸면 됨.

## 프론트 다운로드 — window.location.href

```svelte
<!-- routes/testdb/performance/history/[hisId]/+page.svelte -->
<Button onclick={() => window.location.href = `/api/performance-results/${hisId}/excel`}>
  <FileDown class="size-3 mr-1" /> Export All
</Button>
```

단 한 줄. 이 **"직접 링크"** 패턴이 가능한 이유:

- 응답에 `Content-Disposition: attachment` 헤더가 있음 → 브라우저가 네비게이션 대신 **다운로드로 해석**
- 현재 URL은 바뀌지 않음 (헤더가 해석을 막음)
- fetch + Blob 변환 같은 중간 단계 불필요 → **스피너 UI도 불필요**

T32 Dump의 ZIP 다운로드도 같은 패턴. **Content-Disposition 하나로 프론트 복잡도 제거**.

## 에러 응답은 어떻게 되는가

컨트롤러에 try-catch가 없습니다. 예외가 나면:

- `IllegalArgumentException("History not found: ...")` → 400
- `IllegalArgumentException("COLLECTING:...")` → 400 + 프론트가 prefix 감지해 "수집 중" 안내
- `StatusRuntimeException` (gRPC 실패) → 500
- 기타 Exception → 500

프론트는 `window.location.href`로 요청했기 때문에 **에러 응답도 다운로드 다이얼로그 대신 브라우저 화면**에 나타남. 이상적이지 않지만 Export 실패 빈도가 낮아 충분.

**개선 후보**: 프론트에서 먼저 `HEAD` 요청으로 상태 체크 후 OK면 링크 오픈. 현재는 단순성 우선.

## 여덟 L2의 "외부 서비스 응답 방식" 요약

| L2 | 외부 서비스 | 응답 방식 | 프론트 수신 |
|---|---|---|---|
| 슬롯 | Head TCP | JSON SSE | EventSource |
| 벤치마크 | Go Agent gRPC | JSON SSE (중계) | EventSource |
| 원격 | guacd | WebSocket binary | WebSocketTunnel |
| Metadata | Android adb | HTTP JSON | fetch 폴링 |
| T32 Dump | SSH × 2 | SSE step-* | fetch ReadableStream |
| Pre-Command | SSH | (없음) | (없음) |
| Log Browser | SSH or Local | text / byte[] | fetch |
| **Excel Export** | **Go gRPC unary** | **xlsx byte[]** | **window.location.href** |

**Excel은 가장 짧은 프론트 코드**로 가능한 L2. "작업이 짧고 결과가 파일"인 조합이 이 단순성을 가능케 함.
