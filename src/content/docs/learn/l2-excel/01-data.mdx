## 30초 요약

`PerformanceResultDataService.fetchResultData(historyId)`가 **모든 데이터를 한 record로 합칩니다**.

1. `PerformanceHistory` 조회 → tcId, trId, setId, logPath, slotLocation
2. `PerformanceTestCase` → `PerformanceParser` 체인 조회
3. `logPath` 유무로 **완료 TC** vs **RUNNING TC** 경로 분기
4. `LogBrowserService.readFileContent`로 원격 JSON 읽기
5. JSON 유효성 검증 · 실패 시 **trailing comma 제거 + bracket 닫기** 복구 시도
6. TR에서 `fw`, SetInfo에서 `setName`, history에서 `fileSystem` 추출
7. `ResultData` record 반환

**같은 메서드를 Excel API와 JSON API가 공유**. 이 재사용이 핵심.

## 왜 이 Service가 Controller와 분리되는가

원래 `PerformanceResultController`에 이 로직이 있었는데 **Excel API가 추가되면서 Service로 추출**되었습니다. 근거:

- **JSON 응답 API** (`PerformanceResultController.getResult(historyId)`) — 프론트가 차트 그릴 때
- **Excel 응답 API** (`ExcelExportController.exportExcel(historyId)`) — Export All 버튼

두 엔드포인트가 **정확히 같은 데이터**를 필요로 합니다. 차이는 응답 포맷(JSON vs xlsx)뿐.

**Controller에 로직이 있으면 복제 불가피** → Service로 올려 공유. Log Browser의 `LogBrowserService` 인터페이스처럼 **"비즈니스 데이터 조합"을 Service가 책임**지는 레이어링.

## COLLECTING 예외 — 도메인 시맨틱을 예외로

```java
throw new IllegalArgumentException("COLLECTING:" + remotePath);
```

메시지 prefix가 `COLLECTING:`이면 "아직 수집 중이니 잠시 후 다시 요청"이라는 도메인 상태. 프론트가 이 prefix를 감지해 **스피너 + "테스트 진행 중" 안내**로 처리.

이상적으로는 **별도 예외 클래스**(예: `TestCollectingException`)가 더 깔끔하지만, 간단한 prefix 관례로 시작해 운영하다 필요하면 리팩토링. **기술 부채 후보**로 명시하는 편이 좋음.
