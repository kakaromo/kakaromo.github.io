## 이 모듈에서 배울 것

"사용자가 성능 테스트 결과 페이지에서 **Export All** 버튼을 누르면, 백엔드가 어떻게 **Go 서비스**에 gRPC로 요청을 보내고 **.xlsx 바이너리**를 받아 브라우저에 돌려주는가?"

여덟 번째 L2. **외부 gRPC 서비스로 무거운 작업을 위임**하는 패턴을 정면으로 다룹니다. 벤치마크 L2와 같은 gRPC 축이지만:

- **벤치마크**: 긴 job의 **진행률 스트림** (gRPC Server Stream)
- **Excel**: 단일 요청 · 단일 응답 (gRPC Unary)

## 앞 L2들과의 대조

| 축 | 벤치마크 (Agent) | **Excel Export** |
|---|---|---|
| gRPC 패턴 | Server Stream | **Unary (단일 요청·응답)** |
| 외부 서비스 | Go Agent (디바이스 제어) | Go excel-service (xlsx 생성) |
| 호출 방식 | async StreamObserver | **blocking stub** |
| 응답 크기 | 진행률 JSON (작음) | **.xlsx 바이너리 (수 MB 가능)** |
| 브라우저 인터랙션 | SSE 실시간 중계 | **window.location.href 직접** (브라우저 자동 다운로드) |
| 지속 시간 | 수초~수시간 | 수초 |

| 축 | Log Browser | **Excel Export** |
|---|---|---|
| 재사용 방향 | 다른 L2가 `/files`·`/download-dir` 소비 | **같은 Data Service를 Excel API + JSON API가 공유** |
| 외부 의존 | 런타임 선택 (SSH/Local) | **항상 Go 서비스** |
| 데이터 변환 | 원본 그대로 | **DB 여러 엔티티 조합 + JSON 복구** |

**"재사용 레이어"라는 공통점**이 있지만 방향이 다름:
- Log Browser는 **하부(파일 시스템) 추상화**
- Excel Export는 **상부(비즈니스 데이터 조합)** 추상화

`PerformanceResultDataService`가 핵심 — Excel API와 JSON API가 **같은 로직을 공유**해 일관성을 보장.

## 왜 Go 서비스에 맡기는가

Java에 `excelize` 비슷한 라이브러리(Apache POI)가 있습니다. 왜 외부 서비스로?

- **라이선스·성능**: excelize(Go)는 Apache POI 대비 메모리 효율·차트 지원이 우수한 영역이 있음
- **책임 분리**: Excel 생성 로직 변경이 포털 배포와 분리됨 — 리포트 스타일만 고치고 싶을 때 Go 서비스만 배포
- **재사용**: 같은 Go 서비스를 다른 시스템(내부 대시보드 등)에서도 이용 가능
- **성능 격리**: 대용량 xlsx 생성이 포털 JVM 힙을 잡아먹지 않음

단점: **네트워크 의존**. Go 서비스가 죽으면 Excel 기능 전체 중단. 이 trade-off를 인지하고 선택.
