## 30초 요약

`ExcelGrpcClient`는 **30줄짜리 얇은 컴포넌트**입니다. Spring gRPC의 `GrpcChannelFactory`가 `application.yaml`의 `excel-service` 채널 설정을 읽어 `ManagedChannel`을 만들고, `BlockingStub`을 하나 보유합니다.

호출 메서드 하나: `generateExcel(...)` — proto 빌더로 Request 조립 + `stub.generateExcel(request)` 한 줄.

**벤치마크의 `AgentGrpcClient`와 대조**: Agent는 서버별로 채널을 동적 생성했지만 Excel은 **고정 Bean 하나** (단일 Go 서비스). 더 단순한 선택이 가능했던 이유.

## application.yaml 설정

```yaml title="application-dev.yaml"
spring:
  grpc:
    client:
      channels:
        excel-service:
          address: static://localhost:50052
```

- `excel-service` — 이름. Java 코드의 `createChannel("excel-service")`와 일치해야 함
- `static://` — 정적 주소 resolver (부하분산·DNS 없는 단순 연결)
- `localhost:50052` — Go 서비스 포트

프로덕션에서는 **호스트명·포트 바꾸면 됨**. Java 코드 변경 불필요.

## Spring gRPC vs 직접 ManagedChannel

Agent L2의 `AgentGrpcClient`는 **ManagedChannel을 수동 생성**했습니다:

```java
// AgentGrpcClient.java
this.channel = ManagedChannelBuilder.forAddress(host, port)
        .usePlaintext()
        .maxInboundMessageSize(256 * 1024 * 1024)
        .keepAliveTime(30, TimeUnit.SECONDS)
        // ...
```

Excel은 Spring gRPC의 **`@ConfigurationProperties` 기반 팩토리** 사용:

```java
// ExcelGrpcClient.java
this.stub = ExcelServiceGrpc.newBlockingStub(channels.createChannel("excel-service"));
```

두 방식의 trade-off:

| 축 | Agent (수동) | Excel (팩토리) |
|---|---|---|
| 설정 유연성 | 코드에서 세밀 제어 | yaml로 외부화 |
| 여러 서버 지원 | 동적 host:port 가능 | 빈당 1개 고정 |
| 보일러플레이트 | 많음 | 짧음 |
| 적합 케이스 | 런타임에 변경되는 엔드포인트 | 고정 단일 서비스 |

**Excel은 하나의 고정 서비스**라 팩토리 방식이 깔끔. Agent는 `AgentServer` DB 행마다 다른 host라 수동이 필요했음.

## maxInboundMessageSize 고려

Excel 파일이 커지면 기본 4MB 한도를 넘을 수 있습니다. `application.yaml`에 채널 옵션 추가로 늘림:

```yaml
spring:
  grpc:
    client:
      channels:
        excel-service:
          address: static://localhost:50052
          max-inbound-message-size: 64MB
```

**현재 MOVE에는 이 옵션이 없음** — 성능 결과가 수십 MB로 커지면 추가 필요. 기술 부채 후보로 인지.

## 왜 BlockingStub으로 충분한가

Excel 생성이 보통 **수 초 이내** 완료. Spring MVC 컨트롤러의 요청 스레드가 그 동안 대기해도 부담 없음. async stub + CompletableFuture 같은 구조는 오버엔지니어링.

**응답이 수십 초 이상 걸릴 수 있는 작업**(벤치마크 progress 스트림)은 async 필수. 호출 지연 특성에 맞는 선택.
