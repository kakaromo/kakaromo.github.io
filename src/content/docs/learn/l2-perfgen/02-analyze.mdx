## 30초 요약

`analyzeJson` 한 함수가 **최상위 shape 판정 → 재귀 flattenFields → override 적용 → AnalysisResult 조립**까지. Svelte 5 `$derived`로 **입력이 바뀌면 자동 재실행**.

두 지원 shape:

- **Case 1: object-of-arrays** — `{"SEQ":[...], "RAND":[...]}` — 최상위 키가 탭이 되고 각 배열의 첫 원소로 필드 추론
- **Case 2: array-of-objects** — `[{cycle:1, metrics:{...}}, ...]` — 중첩 object가 탭. 사이클/공통 필드는 최상위 primitive에서

## 왜 $derived 체인인가 — 명령형 재계산 회피

같은 기능을 명령형으로 쓰면:

```js
function onJsonChange() {
  const analysis = analyzeJson(jsonInput);
  updateTabs(analysis);
  updateFieldMapping(analysis);
  regenerateCode();
}
```

- 호출 순서 실수 가능 · 다중 트리거 필요 (role 변경·JSON 변경·tab 수정)
- 한 번의 override에 4곳 수동 업데이트

**$derived 체인으로 DAG 구성**:

```
jsonInput ─┐
           ├→ analysis ─→ mergedTabs ─→ generate ─→ code
overrides ─┘           └─ uniqueFields
           └──────────→ (preview도 같은 analysis)
```

- 노드 추가가 의존성만 선언하면 됨
- 다중 진입점 자동 처리
- "무언가 바뀌면 다 다시 계산"이 당연 — 실수 여지 적음

Svelte 5 runes의 강점이 **반응형 파이프라인에 정확히 들어맞음**.

## 직접 해보기

- [ ] `[{ "a": 1 }, { "a": 2 }]`를 넣고 shape 확인 — Case 2의 "no nested objects" 분기인지
- [ ] override로 stat 필드를 data로 바꾼 뒤 JSON의 해당 필드를 number에서 number[]로 수정 → role 유지 + type 갱신이 동시에 되는지
- [ ] 두 탭에 동일한 이름 필드를 넣고 override를 한 탭에만 적용 → 다른 탭에는 영향 안 가는지 (path prefix 덕분)
