## 30초 요약

6개 공유 타입이 전체 파이프라인의 **계약**:

- **`FieldRole`** — 5종(`tab` · `cycle` · `data` · `stat` · `ignore`) — 필드가 UI에서 어떤 역할인지
- **`FieldType`** — 5종(`number` · `number[]` · `string` · `null` · `object`) — JSON 원시 타입
- **`TopLevelShape`** — 3종(`object-of-arrays` · `array-of-objects` · `other`)
- **`FieldNode`** — path + key + type + sample + role (재귀 flatten 결과)
- **`TabInfo`** — 탭 하나의 메타 (key/label/yAxisUnit/fields)
- **`AnalysisResult`** — 파이프라인 2단계의 반환 값

## 왜 role을 5종으로 좁혔는가

필드가 UI에 나타날 수 있는 자리는 여러 개지만, **perf-content 패턴에서 의미 있는 자리는 5개**:

| Role | 코드 위치 |
|---|---|
| `tab` | 최상위 탭 바 (`{#each tabs}`) |
| `cycle` | 시리즈 라벨 (`Cycle ${entry.cycle}`) |
| `data` | 차트 Y 시리즈 (`entry.latency` 같은 배열) |
| `stat` | 통계 테이블 컬럼 |
| `ignore` | 생성 코드에서 제외 |

**6번째 role을 추가하려면** 생성 템플릿에도 새 섹션 필요 — 타입이 코드 구조와 1:1. "타입 시스템으로 generate의 분기 수를 고정" 하는 설계.

## "path 배열"이라는 선택

FieldNode.path가 **문자열이 아닌 배열**:

```typescript
path: ['metrics', 'latency']  // 좋음
path: 'metrics.latency'        // 나쁨 (왜?)
```

이유:

- **키에 `.`이 들어갈 수 있음** — `{ "a.b": 1 }` 같은 입력에 문자열 방식은 깨짐
- **접근이 자연스러움** — `for (key of path) obj = obj[key]` 루프
- **비교/해시가 필요하면 `pathStr()`** 으로 명시 변환

**배열이 일급이고 문자열은 파생** — 가장 정확한 표현을 일급으로 두는 원칙.

## override의 세 층 (경로별 · 탭별 · 분석 시 적용)

Perf Generator의 **보정 레이어링**:

1. **자동 role** — flattenFields가 타입과 키 이름으로 결정
2. **`fieldRoleOverrides`** — path 문자열 키로 덮어쓰기 (analyzeJson 내부에서 참조)
3. **`tabOverrides`** — 탭 key로 label/yAxisUnit 덮어쓰기 (mergedTabs $derived에서 적용)

**왜 분리**: role은 분석 시 반영되어야 "data였던 필드가 stat이 되면 차트에서 사라지는" 즉각 반응이 가능. tab 레이블은 분석 결과에 영향 안 주므로 **별도 계산 단계**. 동일 데이터라도 소비 시점이 달라 override 시점 분리.

## 직접 해보기

- [ ] `{ "a.b": [1,2,3] }` 같은 key에 `.` 포함 JSON을 넣어 → flattenFields가 path를 `[\"a.b\"]` (단일 원소)로 정확히 보존하는지 콘솔 로그
- [ ] role override를 건 field를 분석 중 제거한(JSON 수정) 후 다시 같은 경로로 추가 → 이전 override가 복원되는지
- [ ] `guessYAxisUnit`의 regex 3개가 겹치는 케이스(`seqTime`) 시도 → **첫 매치 승리** 순서 확인 (rand → seq → time)
