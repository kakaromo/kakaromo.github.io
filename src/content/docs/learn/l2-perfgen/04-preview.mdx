## 30초 요약

두 도우미 컴포넌트가 서로 다른 관점에서 **"생성 전 확인"** 을 돕습니다:

- **JsonTreeView** — 입력 JSON의 구조 시각화 (depth≤2 자동 펼침, 배열 첫 5개만)
- **PerfPreview** — generateCode 결과를 실제로 렌더링 (generate의 side-by-side 검증)

**왜 둘 다 필요한가**: 트리 뷰는 "내 JSON이 어떤 모양이지", 프리뷰는 "그래서 생성 코드가 뭘 만들지" 각기 다른 질문에 답. 한 화면에 섞지 않고 **탭으로 전환**해 인지 부담 절감.

## 왜 "depth 2 + array 5"인가

- **Depth 2**: 최상위 + 첫 중첩. 사용자가 보고 싶은 것은 **"어떤 탭/필드가 있는지"** — 그 수준이면 충분. 그 이상은 클릭으로 확장
- **Array 5**: 같은 shape의 반복이라 5개만 봐도 패턴 파악. 20개를 자동 펼치면 스크롤만 나옴

**"자동으로 어디까지 보여줄지"의 두 축**. 너무 적게(depth 1, array 1)면 매번 클릭, 너무 많이(depth ∞, array ∞)면 메모리 + 스크롤 문제. 경험적으로 이 숫자가 최적.

## 프리뷰-생성 일관성 — 두 구현, 같은 출력

Perf Generator의 위험 포인트는 **"프리뷰에서 잘 보였는데 복사 후 동작 다름"**. 방어책:

1. **같은 analysis record를 공유** — PerfPreview가 자체 분석 안 함
2. **같은 perf-content import** — PerfChart·DataTable·baseChartOption이 프리뷰와 생성 코드 둘 다 사용
3. **같은 role 휴리스틱 적용** — dataFields/statFields 분리가 두 곳에서 동일
4. **같은 xAxis/yAxis 빌드** — baseChartOption이 공통 기본값 제공

그럼에도 **1% 정도 차이 가능**:

- 프리뷰는 runtime data, 생성 코드는 props로 data 받음 → 타이밍 미세 차이
- 프리뷰는 개발 환경, 생성 컴포넌트는 사용처 환경 → CSS inherit 차이

실무 권장: **"프리뷰에서 OK면 95% 준비, 복사 후 한 번 렌더 테스트"**.

## PerfPreview가 생성 코드를 "흉내"하는 깊이

완전한 흉내는 아니고 **핵심만**:

| 요소 | 프리뷰 | 생성 코드 |
|---|---|---|
| 탭 전환 | ✅ | ✅ |
| 차트 렌더 | ✅ line/scatter 토글 | ✅ |
| 통계 테이블 | ✅ | ✅ |
| Excel 내보내기 | ❌ (의미 없음) | ✅ |
| raw data 테이블 | ❌ (공간 제약) | ✅ |
| yAxisMax prop | ❌ | ✅ |
| 사이드바 토글 | ❌ | ✅ |

**프리뷰는 "핵심이 되는가"만 확인**, 생성 코드는 "제품 수준으로 쓰이는가". 80/20 법칙.

## "Sample data" 배너

PerfPreview는 위에 **"이것은 생성 결과의 샘플 미리보기입니다"** 메시지를 표시. 왜?

- 사용자가 **real data를 보고 있다고 오해하기 쉬움** — 사실은 "생성 코드를 가상 실행한 결과"
- 값이 조금 이상해도 괜찮다는 신호
- 생성 흐름의 맥락을 계속 상기

UX의 **"과도한 신뢰 방지"** 패턴. 편리한 미리보기는 오해를 부르기 때문에 방어 멘트 필요.
