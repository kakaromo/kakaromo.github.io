## 30초 요약

저수준 바인딩의 뼈대는 **`API` 인터페이스 한 개**입니다(`internal/t32/api.go`). 두 빌드가 이걸 구현:

- **`api_cgo.go`** (`//go:build windows`) — 진짜 `t32api64.dll` 호출. cgo 헤더가 `T32_*` 함수를 forward-declare 하고 `-lt32api64` 로 링크.
- **`api_stub.go`** (`//go:build !windows`) — in-memory mock. 명령을 기록하고 `PRINT` 를 에코하며 가짜 변수/레지스터/bp 를 들고 있어, **맥/리눅스에서 컴파일·테스트** 가능.

그 위에 **`Worker`** 가 모든 호출을 **단일 goroutine** 으로 직렬화하고(t32api 는 thread-safe 가 아님), **`Session`** 이 단일 attach 수명을 관리합니다.

```go
type API interface {
    Config(node string, port, packLen int) error
    Init() error;  Attach(devID int) error;  Exit() error;  Ping() error
    Cmd(cmd string) error;  EvalExpression(expr string) (string, error)
    GetState() (PracticeRunState, error);  GetPracticeState() (PracticeState, error)
    GetMessage() (text string, mode int, err error)
    // ... ReadVariableString / ReadMemory / ReadRegisterByName / WriteBreakpoint / Stop / Step
}
```

## 왜 인터페이스 + 두 빌드인가

TRACE32 는 Windows 전용 DLL 입니다. 그대로 두면 **개발 머신(맥/리눅스)에서 빌드조차 안 됩니다.** `API` 인터페이스로 경계를 그으면:

- 상위 계층(server/session/worker/poller)은 **인터페이스에만 의존** → 어디서든 컴파일
- Windows 에서만 cgo 백엔드가 진짜 DLL 을 호출
- CI/단위테스트는 stub 으로 빠르게 (`server_test.go` 가 stub 백엔드로 핸들러 검증)

**의존성 역전의 교과서적 적용** — 플랫폼 종속 코드를 인터페이스 뒤로.

## Worker — 단일 goroutine 직렬화

`t32api` 는 thread-safe 가 아니므로 **모든 호출이 worker goroutine 하나**를 거칩니다.

```go
// 모든 t32api 호출은 Do()로 제출 → worker goroutine에서 순차 실행
worker.Do(ctx, func(api API) (any, error) {
    return nil, api.Cmd("Go.direct")
})
```

요청 채널(버퍼 32)로 들어온 작업을 한 goroutine 이 꺼내 실행하고 결과를 돌려줍니다. gRPC 핸들러는 여러 goroutine 에서 동시에 들어와도 **worker 가 줄을 세워** 경합을 없앱니다. `Cmd`/`EvalExpression`/`GetState`/`GetMessage` 등은 모두 이 `Do` 의 편의 래퍼.
