## 30초 요약

`api/v1/t32remote.proto`, package `t32remote.v1`, service `T32Remote`. RPC 26개를 **6묶음**으로:

| 묶음 | RPC | 형태 |
|---|---|---|
| **수명** | `Attach` · `Detach` · `Ping` | unary |
| **실행** | `RunScript` · `ExecuteCommand` | **server-stream** `ProgressEvent` |
| **읽기/평가** | `ReadVariable` · `ReadMemory` · `ReadRegister` · `Eval` | unary |
| **OS 명령** | `RunCommand` | unary |
| **다이얼로그** | `SetDialogValue` · `ClickDialogButton` · `GetDialogValue` · `RegisterDialogPolicy` | unary |
| **윈도우** | `OpenWindow` · `MoveWindow` · `CloseWindow` · `CaptureWindow` | unary |
| **브레이크포인트/실행제어** | `SetBreakpoint` · `DeleteBreakpoint` · `ListBreakpoints` · `Go`(stream) · `Break` · `Step` | unary/stream |

Portal dump 가 실제 쓰는 6개는 [개요](/learn/l2-t32remote/)에 정리. 나머지는 범용/향후용.

## 왜 ExecuteCommand는 스트리밍인가

`T32_Cmd` 는 PRACTICE 가 명령을 **수락한 순간** 리턴합니다 — 스크립트가 끝난 게 아닙니다. 그래서 단순 unary 면 "명령은 갔지만 결과는 모르는" 상태가 됩니다. t32remote 는:

1. worker goroutine 으로 `Cmd` 를 던지고
2. 그동안 broker 가 모아온 **AREA 출력·상태 변화** 를 스트림으로 계속 흘리고
3. PRACTICE 가 idle 로 돌아갈 때까지 기다린 뒤(`waitPracticeIdle`)
4. `Completed` 를 보냅니다.

이 흐름은 [4. 런타임](/learn/l2-t32remote/04-runtime/)의 `runWithProgress` 에서 코드로 봅니다.
