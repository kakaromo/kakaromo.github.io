## 왜 이 문서인가

L2 열아홉 모듈을 **축별로** 정리하면:

- **기능 축 11** (1-11): 도메인별 완결 모듈 (슬롯·벤치마크·원격·… + 🎯 Trace)
- **크로스-컷 2** (12-13): 요청·데이터 관점 (🔐 Auth · 📊 TestDB)
- **엔진 축 3** (14-16): 단방향 파이프라인 (🧬 BinMapper · ⚙ Perf Generator · 🔬 iotest — 순수 2 + 부작용 1)
- **Orchestration 축 1** (17): 🕸 Scenario Canvas — DAG 편집·실행 (축 개척)
- **관리 UI 패턴 2** (18-19): 🛠 Admin · 🖥 Server Admin — 공통 패턴 발견, 축 승격 유보

**5 축 후보 중 3 축 확정, 1 개척, 1 패턴**. 단일 분류로 완벽히 떨어지지 않는 L2도 있음 — Admin은 "Auth/TestDB의 UI 표면"으로도 해석 가능. Trace 는 기능 축이지만 "외부 Rust 서비스 + 컬럼 지향 + 바이너리 IPC" 로 **데이터 규모가 한 단계 위** 인 서브축을 연다.

이 문서의 쓰임:

1. **빠른 참고** — "벤치마크에서 SSE 중계를 어떻게 했더라?" 같은 교차 조회
2. **다음 L2를 쓸 때의 출발점** — 아래 체크리스트와 축별 분기표를 골조로 사용
3. **리뷰어 체크리스트** — 새 L2 PR이 기존 원칙을 따르는지 확인

## 한 줄 요약 열아홉 줄

- **슬롯**: Head TCP → StateStore version → SSE 1초 diff push → 브라우저 상시 스트림
- **벤치마크**: Run 버튼 → Spring gRPC 호출 → Agent 스트림 → SSE 재중계 → 완료까지 Job Card
- **원격**: 서버 선택 → WebSocket 열림 → Spring ↔ guacd TCP 릴레이 → 양방향 프레임
- **Metadata**: running 감지 → slot별 ScheduledExecutor → adb shell 4-way → HTTP 폴링으로 조회
- **T32 Dump**: Run 버튼 → JSch × 2 서버 → 4-Step Stepper (도메인 키워드 판정) → SSE step-* → ZIP
- **Pre-Command**: StateStore.updateSlots → onSlotStateChanged 훅 → TC/Slot 우선순위 → 비동기 SSH (실시간 스트림 없음)
- **Log Browser**: 인터페이스 + @ConditionalOnProperty 두 구현 (SSH · Local) → 다른 L2가 소비하는 재사용 레이어
- **Excel Export**: Data Service (JSON API와 공유) → Go gRPC Unary → xlsx byte[] → window.location.href 직접 다운로드
- **Bitbucket**: @Scheduled 5분 폴링 → REST PAT · 페이지네이션 → autoDownload 분기 → ZIP 스트리밍 + Zip Slip 방어 → T32 소비
- **MinIO**: MinioClient 싱글턴 → 얇은 서비스 래퍼 → multipart 업로드 + 2GB 압축 강제 → 폴더 ZIP 스트리밍 → Agent gRPC가 Go에서 직접 S3 업로드
- **🔐 인증·권한**: ADFS Hybrid Flow → id_token 파싱 → HttpSession 120분 → DB 기반 URL 인터셉터 + 17 권한 + UserHeadAccess (**요청 축 크로스-컷**)
- **📊 TestDB**: TR(FW)·TC(시나리오)·History(결과) 트리플 + logPath 포인터 + PerformanceResultDataService 단일 진입점 + RUNNING/COLLECTING 부분 복구 + Reparse FixedThreadPool(4) (**데이터 축 크로스-컷**)
- **🧬 BinMapper**: C struct 텍스트 → Lexer(15 Token) → Parser(재귀 하강, typedefMap/enumMap) → IR(`StructDefinition`) → BinaryReader(ByteBuffer + 6 분기 재귀) → `MappedResult` 트리 → Table/Hex/JSON 3뷰 (**엔진 축 · 백엔드 엔진 · 독립 devtool**)
- **⚙ Perf Generator**: JSON 샘플 → `analyzeJson`(shape 판정 + `flattenFields` 재귀 + role 휴리스틱) → AnalysisResult($derived) → Field Mapping 사용자 보정 → `generateCode` 문자열 템플릿 조립 → Svelte 소스 → perf-content 폴더에 paste (**엔진 축 · 프론트엔드 코드 생성기**)
- **🔬 iotest**: IOTestEditor에서 thread + commands 트리 조립 → IOTestPreset 엔티티(configJson MEDIUMTEXT) → generic `RunBenchmark` gRPC(tool=IOTEST, params.config=JSON) → Go Agent executor → thread별 goroutine → 실제 syscall(open/pread/pwrite/fsync) → SubscribeJobProgress 스트림 → SSE → IOTestProgressView (**엔진 축 · side-effect 엔진**)
- **🕸 Scenario Canvas**: 비주얼 DAG 편집(@xyflow/svelte) → 8 step + Loop/Condition 노드 → `canvasToProto` 직렬화(Y정렬 vs Kahn 위상 정렬) → 전용 `RunScenario` gRPC(steps+loops+edges+has_branching) → Go Agent → JobProgress 스트림 → ExecutionMiniCanvas 실시간 색상 (**Orchestration 축 개척**)
- **🛠 Admin 대시보드**: 단일 +page.svelte(2652줄) 18 탭 → AdminController(15+ Service 의존) → 17 권한 CRUD + UserHeadAccess + PermissionRequest 승인 워크플로(비관적 락 + 원자적 3 write) (**관리 UI 패턴 1호**)
- **🖥 Server Admin**: portal_servers(10 컬럼 + 포트 3종 + connectionType 비트마스크 + guacd null fallback) + ServerGroup(sortOrder) → Admin 내 Servers 탭 → 5 L2가 공유(원격·슬롯·Log Browser·Pre-Command·T32) (**관리 UI 패턴 2호**)
- **🎯 Trace Analysis**: 업로드 → @Async triggerParse → Rust `ProcessLogs` 스트림 → parquet(3 스키마 → 공용 10컬럼) → `GetChartData`(Arrow IPC) + `GetTraceStats`(JSON) → ECharts(기본) / Deck.gl WebGL(opt-in, 1M 60fps) + MinIO async range-GET reader(`/tmp` 경유 0) (**기능 축 11호 · 데이터 규모 서브축**)

MinIO는 **Log Browser와 짝을 이루는 두 번째 재사용 파일 레이어**. 파일 시스템 vs 오브젝트 스토리지의 축 대조가 선명.

Auth는 **다른 L2와 다른 축**: 설정·런타임 상태·이력이 다른 10 L2의 *가로지르는* 관점입니다. "누가 요청했는가"를 앞단에서 해결해주기 때문에 각 L2 컨트롤러는 권한 검사 코드 없이 비즈니스 로직만 남습니다.

TestDB는 **또 다른 축**: "결과를 어디에 쌓고, 누가 어떻게 꺼내는가"를 표준화. 6 L2가 TestDB 엔티티를 직접 참조, **단일 진입점(`fetchResultData`)으로 JSON API와 Excel Export가 같은 데이터를 공유**. DB에 메타만, 실제 JSON은 원격 FS — 이원 저장이 고유 축.

BinMapper는 **세 번째 축 "엔진"**: 다른 L2에 의존하지 않고 **입력→파이프라인→출력**만 수행하는 독립 devtool. stateless · side-effect 없음 · 한 번 돌고 버리는 pure function.

Perf Generator는 엔진 축 **두 번째 구성원** — 프론트엔드만으로 완결되는 **"코드를 만드는 코드"** 메타 도구. BinMapper(백엔드, struct→data)와 Perf Generator(프론트, JSON→소스코드)가 같은 축 안에서 **네 서브축**(환경·스케일·출력·재사용성) 변주를 보여줌.

iotest는 엔진 축 **세 번째 구성원** — 앞 두 엔진과 결정적 차이는 **side-effect**. 실제 Android 디바이스 파일시스템에 syscall 수행 → 디바이스 상태가 실행 후 바뀜. "엔진이라도 순수/부작용 구분이 있다"는 서브축이 확립됨.

🎯 Trace Analysis 는 기능 축 11번째이지만 **"데이터 규모가 한 단계 위"** 서브축을 연다. 앞 10 기능 L2 가 JSON 직렬화로 충분했다면, Trace 는 **컬럼 지향 parquet + 바이너리 IPC** 없이는 실용 불가. Excel Export (Go gRPC) / Agent (Go gRPC) 의 **Rust 확장판** 이지만 응답 포맷이 xlsx 나 JSON 이 아니라 **Arrow IPC 바이너리** 인 점이 결정적. 그래서 `@Async` (Agent 와 공유) · gRPC streaming (Agent 와 공유) · 외부 언어 서비스 (Excel 과 공유) 세 축을 그대로 쓰면서, `ProjectionMask` · `RowFilter` · `AsyncFileReader` · `ScatterplotLayer binary attribute` 네 개의 L3 개념을 새로 도입.

🕸 Scenario Canvas는 **Orchestration 축 개척** — 사용자가 여러 step을 **DAG로 편집**, 제어 흐름(Loop/Condition) 노드가 실행 경로 변경. 엔진 축과 다른 이유: IR이 선형이 아닌 그래프, 편집→저장→실행 3단계 (엔진은 입력→출력 1단계).

🛠 Admin 대시보드와 🖥 Server Admin은 **관리 UI 패턴** — 공통 패턴은 명확하지만 **축 승격 유보**. 이유: Admin이 Auth/TestDB의 UI 표면으로도 해석 가능, Server Admin이 Admin 안에 포함 — 독립 축이라기보다 **크로스-컷의 앞단**.

### 프론트 실시간 패턴 (공통)

```svelte
<script lang="ts">
  import { createHeadSlotStore } from '$lib/api/headSlotStore.svelte';
  const store = createHeadSlotStore('foo');
  $effect(() => {
    store.connect();
    return () => store.disconnect();
  });
</script>
```

**L2 중 실시간이 필요한 다섯은 모두 `$effect` 안에서 리소스를 획득하고 return cleanup으로 해제**합니다. 획득하는 리소스만 다릅니다.

| 🕸 Scenario Canvas | `EventSource` (per-step JobProgress) | `es.close()` |
| 🛠 Admin 대시보드 | **없음** (폴링 위주) | — |
| 🖥 Server Admin | **없음** (CRUD 응답만) | — |
| 🎯 Trace Analysis | `setInterval`(2s job list polling) + `AbortController`(chart/stats fetch) | `clearInterval()` + `controller.abort()` |

**19 L2 중 11개 무자원** — Trace 는 **"폴링 + fetch cancel"** 이라는 9번째 자원 보유 케이스. SSE/WebSocket 같은 persistent 스트림 없이, 짧은 polling 과 취소 가능한 fetch 만으로 진행 추적. 실시간 자원 보유 8개(슬롯·벤치마크·원격·Metadata·T32·Bitbucket·iotest·Scenario·**Trace**).

## 설계 결정의 공통 원칙

L2 19종을 읽으며 반복되는 판단 기준:

1. **"얼마나 오래 살아야 하는가"로 저장소를 가른다** — 분 미만이면 인메모리, 그 이상이면 DB.
2. **"얼마나 자주 바뀌는가"로 yaml vs DB를 가른다** — 빌드 단위면 yaml, 운영 중 바뀌면 DB.
3. **"빠른 감지자 + 느린 작업자"로 스레드풀을 분리한다** — Spring 기본 `@Scheduled` 풀이 I/O에 막히지 않게.
4. **프로토콜 변환은 한 곳에서만** — 컨트롤러가 겹겹이 감싸지 않도록, 한 진입점에서 단일 번역 레이어.
5. **에러 격리** — 한 item의 실패가 루프를 멈추지 않도록 try/catch 후 continue.
6. **cleanup은 finally/return** — 획득한 자원은 모든 경로에서 해제 보장.

- **L1 보강** — 학습 경로 최적화, 19 L2 지도 갱신, 신규 개발자 온보딩 트랙 명시화

어느 것을 다음에 쓸지는 "신규 개발자 온보딩에 가장 일찍 나타나는 모듈"로 고르면 됩니다.

- [T32 Dump L2](/learn/l2-t32/) — 5단계 템플릿이 단발 Stepper 도메인에 적합함을 증명.
- [Pre-Command L2](/learn/l2-precmd/) — 실시간 스트림이 없는 이벤트 훅 모듈에도 템플릿이 적용됨.
- [Log Browser L2](/learn/l2-logbrowser/) — 재사용 레이어(전략 패턴)에도 적용됨.
- [Excel Export L2](/learn/l2-excel/) — 가장 짧은 L2(4페이지). "작업 · 응답 · 끝" 패턴에도 템플릿이 쓰일 수 있음.
- [Bitbucket L2](/learn/l2-bitbucket/) — 외부 REST API 폴링 + FS 부작용. 주기·상태 머신·Zip Slip 방어가 한 L2에 공존.
- [MinIO L2](/learn/l2-minio/) — S3 호환 오브젝트 스토리지. Log Browser와 짝을 이루는 하부 재사용 레이어.
- [🔐 인증·권한 L2](/learn/l2-auth/) — 요청 축 크로스-컷. 컨트롤러 없이 인터셉터로만 권한이 해결되는 구조.
- [📊 TestDB L2](/learn/l2-testdb/) — 데이터 축 크로스-컷. DB 메타 + 원격 JSON 이원 저장 + 단일 진입점 DataService로 6 L2가 공유.
- [🧬 BinMapper L2](/learn/l2-binmapper/) — 엔진 축. Lexer + Parser + ByteBuffer로 컴파일러 파이프라인을 L2 템플릿 안에 포장.
- [⚙ Perf Generator L2](/learn/l2-perfgen/) — 엔진 축 두 번째. 프론트엔드만으로 `analyzeJson` + `generateCode` 파이프라인 — "코드를 만드는 코드" 메타 도구.
- [🔬 iotest L2](/learn/l2-iotest/) — 엔진 축 세 번째. syscall DSL 실행 엔진. "엔진이라도 순수/부작용 구분" 서브축 확립.
- [🕸 Scenario Canvas L2](/learn/l2-scenario/) — Orchestration 축 개척. @xyflow/svelte 비주얼 편집 + Kahn 위상 정렬 + has_branching 플래그.
- [🛠 Admin 대시보드 L2](/learn/l2-admin/) — 관리 UI 패턴 1호. 2652줄 단일 페이지 18 탭 + 비관적 락 승인 워크플로.
- [🖥 Server Admin L2](/learn/l2-serveradmin/) — 관리 UI 패턴 2호. 비트마스크 3 프로토콜 + null fallback + 5 L2 공용 인프라 관리.
- [🎯 Trace Analysis L2](/learn/l2-trace/) — 기능 축 11호 · 데이터 규모 서브축. Rust gRPC + Arrow IPC + parquet projection + WebGL binary attribute + MinIO range-GET async reader. 4 신규 L3 개념 도입([Apache Arrow](/learn/l3-concepts/data/apache-arrow/) · [Parquet ProjectionMask/RowFilter](/learn/l3-concepts/data/parquet-projection/) · [AsyncFileReader trait](/learn/l3-concepts/rust/async-file-reader/) · [Deck.gl OrthographicView + binary attribute](/learn/l3-concepts/webgl/deckgl-orthographic/)).
- [UFS Metadata 모니터링 L2](/learn/l2-metadata/) — 기능 축 (재작성). TC running 감지 자동 수집 + 7 command_type(tool/sysfs/keyvalue/raw/table/bitmap/binary) + BinMapper 재사용(binary) + f2fs 3 전용 뷰(Iostat/Heatmap/Bitmap) + `{userdata}` placeholder. "감지자 + 작업자 분리" 패턴의 모범.

> **참고**: 위 "축별 대조" 표 3종(트리거·전송·영속)에는 Orchestration(Scenario)·관리 UI(Admin·Server Admin) 3 모듈은 컬럼 수 제약으로 포함하지 않았습니다. 각 모듈의 상세는 위 "한 줄 요약 열아홉 줄"과 해당 L2 페이지를 참고하세요.
