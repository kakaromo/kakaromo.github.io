## 왜 이 문서인가

L2 열네 모듈을 순서대로 읽으면 **공통 패턴**이 점점 눈에 들어옵니다. 앞 10 L2는 기능 모듈, 열한 번째 Auth는 **요청 축 크로스-컷**, 열두 번째 TestDB는 **데이터 축 크로스-컷**, 열세 번째 BinMapper와 열네 번째 Perf Generator는 **엔진 축**(입력→파이프라인→출력 단방향 계산) 두 번째 구성원. **세 축 각각 2종 이상 확보**되어 축 자체가 패턴임이 확정됐습니다.

이 문서의 쓰임:

1. **빠른 참고** — "벤치마크에서 SSE 중계를 어떻게 했더라?" 같은 교차 조회
2. **다음 L2를 쓸 때의 출발점** — 아래 체크리스트와 축별 분기표를 골조로 사용
3. **리뷰어 체크리스트** — 새 L2 PR이 기존 원칙을 따르는지 확인

## 한 줄 요약 열네 줄

- **슬롯**: Head TCP → StateStore version → SSE 1초 diff push → 브라우저 상시 스트림
- **벤치마크**: Run 버튼 → Spring gRPC 호출 → Agent 스트림 → SSE 재중계 → 완료까지 Job Card
- **원격**: 서버 선택 → WebSocket 열림 → Spring ↔ guacd TCP 릴레이 → 양방향 프레임
- **Metadata**: running 감지 → slot별 ScheduledExecutor → adb shell 4-way → HTTP 폴링으로 조회
- **T32 Dump**: Run 버튼 → JSch × 2 서버 → 4-Step Stepper (도메인 키워드 판정) → SSE step-* → ZIP
- **Pre-Command**: StateStore.updateSlots → onSlotStateChanged 훅 → TC/Slot 우선순위 → 비동기 SSH (실시간 스트림 없음)
- **Log Browser**: 인터페이스 + @ConditionalOnProperty 두 구현 (SSH · Local) → 다른 L2가 소비하는 재사용 레이어
- **Excel Export**: Data Service (JSON API와 공유) → Go gRPC Unary → xlsx byte[] → window.location.href 직접 다운로드
- **Bitbucket**: @Scheduled 5분 폴링 → REST PAT · 페이지네이션 → autoDownload 분기 → ZIP 스트리밍 + Zip Slip 방어 → T32 소비
- **MinIO**: MinioClient 싱글턴 → 얇은 서비스 래퍼 → multipart 업로드 + 2GB 압축 강제 → 폴더 ZIP 스트리밍 → Agent gRPC가 Go에서 직접 S3 업로드
- **🔐 인증·권한**: ADFS Hybrid Flow → id_token 파싱 → HttpSession 120분 → DB 기반 URL 인터셉터 + 17 권한 + UserHeadAccess (**요청 축 크로스-컷**)
- **📊 TestDB**: TR(FW)·TC(시나리오)·History(결과) 트리플 + logPath 포인터 + PerformanceResultDataService 단일 진입점 + RUNNING/COLLECTING 부분 복구 + Reparse FixedThreadPool(4) (**데이터 축 크로스-컷**)
- **🧬 BinMapper**: C struct 텍스트 → Lexer(15 Token) → Parser(재귀 하강, typedefMap/enumMap) → IR(`StructDefinition`) → BinaryReader(ByteBuffer + 6 분기 재귀) → `MappedResult` 트리 → Table/Hex/JSON 3뷰 (**엔진 축 · 백엔드 엔진 · 독립 devtool**)
- **⚙ Perf Generator**: JSON 샘플 → `analyzeJson`(shape 판정 + `flattenFields` 재귀 + role 휴리스틱) → AnalysisResult($derived) → Field Mapping 사용자 보정 → `generateCode` 문자열 템플릿 조립 → Svelte 소스 → perf-content 폴더에 paste (**엔진 축 · 프론트엔드 코드 생성기**)

MinIO는 **Log Browser와 짝을 이루는 두 번째 재사용 파일 레이어**. 파일 시스템 vs 오브젝트 스토리지의 축 대조가 선명.

Auth는 **다른 L2와 다른 축**: 설정·런타임 상태·이력이 다른 10 L2의 *가로지르는* 관점입니다. "누가 요청했는가"를 앞단에서 해결해주기 때문에 각 L2 컨트롤러는 권한 검사 코드 없이 비즈니스 로직만 남습니다.

TestDB는 **또 다른 축**: "결과를 어디에 쌓고, 누가 어떻게 꺼내는가"를 표준화. 6 L2가 TestDB 엔티티를 직접 참조, **단일 진입점(`fetchResultData`)으로 JSON API와 Excel Export가 같은 데이터를 공유**. DB에 메타만, 실제 JSON은 원격 FS — 이원 저장이 고유 축.

BinMapper는 **세 번째 축 "엔진"**: 다른 L2에 의존하지 않고 **입력→파이프라인→출력**만 수행하는 독립 devtool. stateless · side-effect 없음 · 한 번 돌고 버리는 pure function.

Perf Generator는 엔진 축 **두 번째 구성원** — 프론트엔드만으로 완결되는 **"코드를 만드는 코드"** 메타 도구. BinMapper(백엔드, struct→data)와 Perf Generator(프론트, JSON→소스코드)가 같은 축 안에서 **네 서브축**(환경·스케일·출력·재사용성) 변주를 보여줌. iotest 엔진이 세 번째 후보로 대기.

### 프론트 실시간 패턴 (공통)

```svelte
<script lang="ts">
  import { createHeadSlotStore } from '$lib/api/headSlotStore.svelte';
  const store = createHeadSlotStore('foo');
  $effect(() => {
    store.connect();
    return () => store.disconnect();
  });
</script>
```

**L2 중 실시간이 필요한 다섯은 모두 `$effect` 안에서 리소스를 획득하고 return cleanup으로 해제**합니다. 획득하는 리소스만 다릅니다.

| L2 | 획득 리소스 | cleanup |
|---|---|---|
| 슬롯 | `EventSource` | `es.close()` |
| 벤치마크 | `EventSource`(job당 하나) | `es.close()` |
| 원격 | `Guacamole.Client` + keyboard/mouse | `client.disconnect() + keyboard.reset()` |
| Metadata | `setInterval(30s)` | `clearInterval()` |
| T32 Dump | `fetch reader` + `setInterval`(경과 타이머) | `reader.cancel() + clearInterval()` |
| Pre-Command | **없음** | — |
| Log Browser | **없음** (탐색·다운로드 fetch만) | — |
| Excel Export | **없음** (`window.location.href`) | — |
| Bitbucket | `fetch reader`(DETECTED 다운로드 SSE) | `reader.cancel()` |
| MinIO | **없음** (multipart 업로드 + 탐색 fetch) | — |
| 🔐 Auth | **없음** (`auth.svelte.ts` · `session.svelte.ts` 전역 store 주기적 갱신) | — (앱 라이프사이클과 같이 감) |
| 📊 TestDB | **없음** (REST 조회 + 차트 렌더만, Reparse 폴링은 `setInterval` 1회) | Reparse: `clearInterval()` |
| 🧬 BinMapper | **없음** (multipart POST 응답 렌더만) | — |
| ⚙ Perf Generator | **없음** (서버 호출 0 · 모든 연산 $derived) | — |

Perf Generator까지 무자원. **14 L2 중 9개는 실시간 자원 없음** — 필요 없는 건 안 쓴다는 원칙이 과반을 훨씬 넘김.

## 설계 결정의 공통 원칙

L2 14종을 읽으며 반복되는 판단 기준:

1. **"얼마나 오래 살아야 하는가"로 저장소를 가른다** — 분 미만이면 인메모리, 그 이상이면 DB.
2. **"얼마나 자주 바뀌는가"로 yaml vs DB를 가른다** — 빌드 단위면 yaml, 운영 중 바뀌면 DB.
3. **"빠른 감지자 + 느린 작업자"로 스레드풀을 분리한다** — Spring 기본 `@Scheduled` 풀이 I/O에 막히지 않게.
4. **프로토콜 변환은 한 곳에서만** — 컨트롤러가 겹겹이 감싸지 않도록, 한 진입점에서 단일 번역 레이어.
5. **에러 격리** — 한 item의 실패가 루프를 멈추지 않도록 try/catch 후 continue.
6. **cleanup은 finally/return** — 획득한 자원은 모든 경로에서 해제 보장.

## 아직 L2로 없는 후보

메모리에 기록된 기능들 중 템플릿에 잘 맞을 것:

- **iotest syscall 엔진** — I/O 테스트 실행. 엔진 축 세 번째 후보 (side-effect 있는 엔진)
- **Guacamole Admin/관리** — 원격 L2의 관리 UI 확장
- **Admin 대시보드 (17 권한 편집·유저 관리)** — Auth의 관리 UI 확장
- **SlotInfomation · SetInfomation 상세** — TestDB 크로스-컷의 하부 엔티티 상세

어느 것을 다음에 쓸지는 "신규 개발자 온보딩에 가장 일찍 나타나는 모듈"로 고르면 됩니다.

**검증된 사례 10건**:
- [T32 Dump L2](/learn/l2-t32/) — 5단계 템플릿이 단발 Stepper 도메인에 적합함을 증명.
- [Pre-Command L2](/learn/l2-precmd/) — 실시간 스트림이 없는 이벤트 훅 모듈에도 템플릿이 적용됨.
- [Log Browser L2](/learn/l2-logbrowser/) — 재사용 레이어(전략 패턴)에도 적용됨.
- [Excel Export L2](/learn/l2-excel/) — 가장 짧은 L2(4페이지). "작업 · 응답 · 끝" 패턴에도 템플릿이 쓰일 수 있음.
- [Bitbucket L2](/learn/l2-bitbucket/) — 외부 REST API 폴링 + FS 부작용. 주기·상태 머신·Zip Slip 방어가 한 L2에 공존.
- [MinIO L2](/learn/l2-minio/) — S3 호환 오브젝트 스토리지. Log Browser와 짝을 이루는 하부 재사용 레이어.
- [🔐 인증·권한 L2](/learn/l2-auth/) — 요청 축 크로스-컷. 컨트롤러 없이 인터셉터로만 권한이 해결되는 구조.
- [📊 TestDB L2](/learn/l2-testdb/) — 데이터 축 크로스-컷. DB 메타 + 원격 JSON 이원 저장 + 단일 진입점 DataService로 6 L2가 공유.
- [🧬 BinMapper L2](/learn/l2-binmapper/) — 엔진 축. Lexer + Parser + ByteBuffer로 컴파일러 파이프라인을 L2 템플릿 안에 포장.
- [⚙ Perf Generator L2](/learn/l2-perfgen/) — 엔진 축 두 번째. 프론트엔드만으로 `analyzeJson` + `generateCode` 파이프라인 — "코드를 만드는 코드" 메타 도구.
