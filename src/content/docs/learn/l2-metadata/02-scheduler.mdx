## 30초 요약

`MetadataMonitorService`가 **두 겹의 스케줄**을 관리합니다.

- **바깥(@Scheduled 5초)**: 슬롯 `testState`가 running으로 전환되었는지 확인. 상태 변화를 감지해 **개별 슬롯 수집을 시작/중지**.
- **안쪽(ScheduledExecutorService, 슬롯당 1개)**: 실제 수집 `monitorOnce()`를 슬롯별 주기(기본 5분)로 실행.

슬롯마다 `SlotMonitorContext`가 생성되고, 8개 스레드풀에서 병렬로 돈다.

## 기본값 OFF 설계

왜 새 슬롯은 기본이 OFF인가?

- **디바이스 I/O 추가 부하** — 테스트 자체 성능에 영향 가능
- **명령 중 실수** — 새로 추가된 타입이 잘못된 명령을 보내 디바이스가 응답 불능이 되면 **모든 슬롯이 동시에 영향**
- **디버깅 용도 우선** — 평상시에는 수집 없이, 필요할 때만 켜서 사용

관리자는 Admin UI에서 슬롯별 토글로 켭니다 — 단계 4의 REST API가 이 토글을 받아 `enabledSlots`에 반영.

## 왜 @Scheduled 기본 풀이 아닌 별도 Executor인가

Spring `@Scheduled` 기본은 **싱글 스레드**입니다. 8개 슬롯이 동시에 수집 중인데 하나라도 10초가 걸리면, 다음 5초 폴링이 밀려서 상태 변화를 놓칠 수 있습니다.

두 풀을 분리해:

- **@Scheduled 5초 폴링**은 항상 빠르게(수 ms).
- **Executor**에서 실제 I/O 작업을 흡수.

이 패턴은 SSE 중계·Heartbeat·다른 서비스에서도 반복적으로 나타나는 **"빠른 감지자 + 느린 작업자"** 분리입니다.
