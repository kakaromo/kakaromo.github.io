## 30초 요약

`monitorOnce(ctx)`가 주기마다 호출됩니다. 한 실행에서:

1. Context의 command 목록을 순회
2. 각 `commandType`에 맞는 executor 메서드 호출 (tool / sysfs / raw / keyvalue)
3. JSON 문자열을 파싱 → `time` 필드 추가 → 인메모리 리스트에 누적
4. SFTP로 VM에 `debug_<typeKey>.json` 저장

**전체가 `ReentrantLock`으로 보호**됩니다 — @Scheduled task와 `stopMonitoring`의 최종 수집이 겹치는 것을 막기 위해서.

## 왜 4가지 executor 메서드를 서비스 밖에 분리했나

`MetadataCommandExecutor`에 실제 실행 로직이 있고, 서비스는 **분기만** 합니다. 이 분리의 이득:

- **각 방식이 각자 복잡함**: sysfs는 여러 경로 읽고 regex 추출. keyvalue는 들여쓰기 기반 파싱. raw는 단순.
- **테스트 용이**: 서비스 테스트는 executor를 mock. executor 테스트는 입력 문자열만 갖고 단위 검증.
- **재사용**: 다른 도구(Admin의 1회성 명령 실행)가 같은 executor를 씀.

이 L2 튜토리얼은 executor 내부 구현(정규식·파싱)까지는 깊이 들어가지 않습니다 — 패턴만 알면 세부는 파일을 직접 열어 따라갈 수 있습니다.

## "time" 필드가 중요한 이유

수집된 JSON 하나하나가 **타임스탬프를 자체 보유**하게 되면서:

- 프론트가 과거 파일을 다시 읽어도 x축이 자연스러움
- 샘플 간 주기가 달라도(슬롯별 override) 각자의 time이 따로 기록됨
- **히트맵의 x축 = time, y축 = 배열 index, 값 = 숫자**라는 단순한 매핑이 가능

이 작은 결정이 단계 5의 프론트 히트맵을 단순하게 만듭니다.
