## 30초 요약

Metadata 모듈은 **3개 테이블**로 구성됩니다.

- **`ufs_metadata_types`** — "수집 대상이 되는 메타데이터 종류"의 카탈로그 (예: `pa-health`, `smart`, `wa`).
- **`ufs_metadata_commands`** — **한 타입 → 여러 명령**. 실제 adb shell로 무엇을 실행할지. `commandType` 컬럼이 **4가지 수집 방식**을 가른다.
- **`ufs_product_metadata`** — 제품(controller + nandType + cellType)별로 **어떤 타입을 수집해야 하는지** CSV 매핑.

전역 설정 `metadata.monitor.*`는 `MetadataMonitorProperties`에서 yaml로 바인딩.

## 4가지 commandType 비교

| 값 | 실행 방식 | 입력 예시 | 결과 처리 |
|---|---|---|---|
| `tool` | `adb shell <cmd>` | `./my-ufs-tool --json pa` | 출력이 JSON일 것으로 기대 |
| `sysfs` | `adb shell cat <paths>` | `/sys/block/sda/stat\n/sys/block/sda/size` (줄바꿈 구분) | 각 경로의 plaintext → 키별 JSON 객체로 조립 |
| `raw` | `adb shell <cmd>` | `dd if=/dev/sdaN bs=1 count=512 2>/dev/null \| xxd` | 출력 전체를 문자열 그대로 `{"value": "..."}`에 저장 |
| `keyvalue` | `adb shell <cmd>` | `./show-wa --verbose` | 들여쓰기·괄호 자동 파싱, 숫자는 number로 |

**프론트 히트맵은 JSON 모양이 무엇이든 동일하게 처리** — 단계 5에서 `flattenObject` + `classifyKeys`로 흡수합니다.

## UfsMetadataType — 타입 카탈로그 (간략)

```java
@Entity
@Table(name = "ufs_metadata_types")
public class UfsMetadataType {
    @Id @GeneratedValue Long id;
    String name;         // "PA Health"
    String typeKey;      // "pa-health" (UK, 프론트·파일명에 쓰임)
    String category;     // "SSD", "UFS" 등
    boolean enabled;
    String description;
}
```

`typeKey`가 핵심 식별자입니다. **프론트 히트맵 탭 이름**, **SFTP 파일 이름**(`debug_pa-health.json`), **REST API 경로** 등에서 모두 이 값을 씁니다.

## UfsProductMetadata — 제품별 매핑 (간략)

```java
@Entity
@Table(
  name = "ufs_product_metadata",
  uniqueConstraints = @UniqueConstraint(columnNames = {"controller","cellType","nandType","oem"})
)
public class UfsProductMetadata {
    String controller;       // 펌웨어 벤더
    String cellType;         // TLC, QLC ...
    String nandType;
    String oem;
    String metadataTypeIds;  // CSV: "1,3,5" → 지원하는 타입 ID들
}
```

왜 CSV? → **다대다 관계지만 조회 패턴이 "한 제품 → 타입 목록"** 뿐이라 중간 테이블을 생략했습니다. 쓰는 쪽에서 `parseTypeIds(csv)`로 풀고, 업데이트도 통째로 하므로 실용적 비용이 낮음.

## 왜 타입·명령을 DB에 두는가 (하드코딩하지 않고)

- **장비 추가가 코드 릴리스 없이 가능** — Admin UI에서 새 타입·명령을 넣으면 다음 수집부터 바로 적용.
- **운영 중 한 명령만 꺼도 됨** — `enabled=false`로 바꾸면 해당 타입 수집 중단.
- **장비별 명령 차이 흡수** — 같은 `pa-health`라도 펌웨어가 다르면 명령을 다르게 등록 가능 (현재는 다대일로 1:N, 장비 필터링은 `UfsProductMetadata`가 담당).

Hard-coded enum보다 **유연성과 운영 비용의 균형**이 좋습니다.
