## 30초 요약

REST 계층은 두 컨트롤러로:

1. **`MetadataController` (`/api/metadata/*`)** — 사용자용 15 endpoint: 타입 조회, 슬롯 상태, in-memory 시계열, 파일 다운로드, 토글/제외/주기 설정
2. **`MetadataAdminController` (`/api/admin/metadata/*`)** — Admin CRUD: Type/Command/ProductMapping 관리 + 중복 검증

`{userdata}` placeholder 치환:

- **1회 조회**: `readlink -f /dev/block/by-name/userdata` → `/dev/block/sda10` → basename `sda10`
- **저장**: `SlotMonitorContext.placeholders.put("userdata", "sda10")` — startMonitoring 시 1회만
- **치환**: `resolvePlaceholders(template, ctx)` 가 매 monitorOnce 때 `{userdata}` 를 실제 값으로

f2fs 경로는 디바이스마다 다른데(`sda10`, `sdc77`, `mmcblk0p13` 등) commandTemplate 에 하드코딩하면 제품별 매핑이 폭발적으로 증가 → placeholder 로 흡수.

## Admin API — `MetadataAdminController`

사용자용과 분리된 `/api/admin/metadata/*` 14 endpoint:

| 리소스 | CRUD | 특이점 |
|---|---|---|
| `/types` | GET/POST/PUT/DELETE | DELETE 시 연관 command/mapping 자동 정리 |
| `/commands` | GET/POST/PUT/DELETE + `/by-type/{typeId}` | **`commandType` 런타임 변경 가능** — 같은 명령을 sysfs→keyvalue 로 바꿔 실험 |
| `/product-mappings` | GET/POST/PUT/DELETE + `/by-type/{typeId}` | **UNIQUE `(controller, cell, nand, oem)`** — 중복 등록 시 409 Conflict |

중복 체크는 서비스 레이어:

```java
// MetadataTypeService.createProductMapping
// 정규화 — null/blank → ""
String c = nz(controller); String ct = nz(cellType);
String nt = nz(nandType);  String o = nz(oem);

// 이미 있는지 체크
if (productMetadataRepo.findExact(c, ct, nt, o).isPresent()) {
    throw new IllegalStateException("이미 등록된 제품 조합입니다");
}
```

프론트는 이 예외를 409 로 받아 "이미 등록된 제품 조합" 알림 — 사용자에게 직관적.

## Product mapping 와일드카드 매칭

DB NOT NULL DEFAULT '' 로 저장하고, **빈 문자열은 wildcard**:

```sql
-- 예시 데이터
INSERT INTO ufs_product_metadata (controller, cell_type, nand_type, oem, metadata_type_ids) VALUES
  ('', '', '', '', '1,2'),              -- 모든 제품에 타입 1,2
  ('CONTROLLER_A', '', '', '', '3'),    -- CONTROLLER_A 라면 타입 3도 추가
  ('CONTROLLER_A', 'TLC', '', '', '4'); -- CONTROLLER_A + TLC 에 타입 4도
```

`findByProduct(c, ct, nt, o)` 는 각 컬럼이 `= ''` OR `= ?` 인 row 들을 모두 반환 → `metadataTypeIds` CSV 를 합집합으로 모아 **타입 1, 2, 3, 4** 전부 수집.

## 왜 placeholder 를 1회만 조회하는가

`/dev/block/by-name/userdata` 는 **디바이스가 같은 한 불변**:

- 디바이스 연결 = 같은 파티션 레이아웃
- 부팅 후 바뀌지 않음 (partition table 자체가 변경되는 경우는 극단)

매 tick 마다 readlink 를 다시 치면:

- adb 왕복 1회 추가 — 주기 30초에서 수집 20개 명령이면 **20% 오버헤드**
- 디바이스 slowdown 시 전체 수집이 지연
- placeholder 값이 변할 이유가 없음

**"안 변하는 건 한 번만"** 원칙 — SlotMonitorContext 가 TC running 동안 살아 있으므로 그 수명에 바인딩.

## `{userdata}` 외의 placeholder 는?

현재는 `{userdata}` 하나만. 추가하고 싶다면:

1. `SlotMonitorContext.placeholders.put("newkey", value)` 를 startMonitoring 어딘가에서 populate
2. commandTemplate 에 `{newkey}` 쓰기 — resolvePlaceholders 가 자동 치환

예시 후보:
- `{ufs_partition}` — RPMB 등 다른 UFS LU 경로
- `{kernel_version}` — 커널 별 sysfs 경로 변주

추가 시 코드는 단순 — **설계 공간이 이미 열려 있음**.
