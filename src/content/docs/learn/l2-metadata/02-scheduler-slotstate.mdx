## 30초 요약

**두 종류의 타이머** 가 있습니다:

1. **`@Scheduled(5000ms)` checkSlotStateChanges()** — Spring 관리 루프. 모든 슬롯을 훑어 running 진입/이탈만 감지. **가볍게, 절대 안 막히게**
2. **슬롯별 `ScheduledFuture`** — 8 threads pool 에서 `scheduleAtFixedRate(monitorOnce, 0, intervalSec)`. 실제 수집 (SSH + adb shell 왕복 수 초). **한 슬롯 느려도 다른 슬롯 영향 없음**

`initslot` 명령이 도착하면 `clearSlot()` 이 enabledSlots/excludedTypes/slotIntervalSeconds 3 맵 + activeMonitors 를 모두 청소. VM 디스크의 JSON 파일은 **이력 보존 목적으로 유지**.

## 왜 "감지자 + 작업자" 분리

단일 `@Scheduled` 안에서 모든 슬롯의 수집을 돌리면:

- 슬롯 3 이 SSH 타임아웃(30초) 걸리는 동안 **슬롯 1, 2, 4... 전부 대기**
- `fixedDelay` 라 한 번 막히면 다음 tick 이 한참 뒤
- thread count 를 늘리고 싶어도 Spring 기본 `@Scheduled` executor 는 1 스레드

이 구조를 깨려면 **"가벼운 감지자 → 무거운 작업자로 디스패치"**:

- 감지자(`@Scheduled`): 상태 저장소(`HeadSlotStateStore`) 읽기 + 전이 감지만 — microseconds
- 작업자(`ScheduledExecutorService`): 슬롯별 fixedRate task — seconds-scale 걸려도 OK

같은 패턴이 TestDB Reparse, Agent Job, Head SSE 에도 사용됨 ([L2 공통 원칙 #3](/learn/l2-compare/#설계-결정의-공통-원칙)).

## `monitorOnce` 의 lock 보호

```java
if (!ctx.getMonitorLock().tryLock()) return;
```

`scheduleAtFixedRate` 는 이전 태스크가 아직 실행 중이어도 다음 tick 을 시작합니다. 주기가 10초인데 실제 수집이 15초 걸리면 **동시 실행 위험**. `tryLock` 으로 **이전이 끝나지 않았으면 이번 tick 은 skip** — 데이터 중복 방지.

`stopMonitoring` 의 최종 수집도 같은 lock 을 쓰므로 scheduler 와 종료 처리가 **race 없이 직렬화**.

## 주기 변경 (초 단위)

이전 버전은 `@Scheduled(fixedDelayString = "분")` 로 분 단위만 가능했습니다. 현재는:

```java
ScheduledFuture<?> future = executor.scheduleAtFixedRate(
    () -> monitorOnce(ctx), 0, intervalSec, TimeUnit.SECONDS);
```

- `PUT /api/metadata/slot/{tentacleName}/{slotNumber}/interval?intervalSeconds=30` 으로 슬롯마다 개별 설정
- 최소 10초 clamp (Controller 에서) — 네트워크/디바이스 보호
- **기존 future 를 cancel 하고 재등록하지 않음** — 맵 값만 갱신하고 다음 startMonitoring 부터 반영. 운영상 slot toggle(off → on) 로 즉시 재적용 가능

## 직접 해보기

- [ ] TC 시작 전 `SELECT * FROM portal_trace_jobs` 같은 방식으로 `enabledSlots` 상태 확인할 수는 없음(인메모리) — `GET /api/metadata/slot/.../enabled` 사용
- [ ] TC running 시 `GET /api/metadata/status` 로 activeMonitors 확인
- [ ] 수집 중에 `PUT .../interval?intervalSeconds=15` 로 주기 바꿔도 현재 future 에는 즉시 반영 안 됨 (off→on 재적용 필요)
