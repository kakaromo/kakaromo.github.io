## 30초 요약

프론트는 SSE/WebSocket 대신 **HTTP GET 폴링**으로 데이터를 가져옵니다.

- **GET `/api/metadata/slot/{tentacleName}/{slotNumber}`** — 상태 요약 (모니터링 여부, elapsed, 타입별 entry 수)
- **GET `/api/metadata/slot/{tentacleName}/{slotNumber}/{typeKey}`** — 특정 타입의 시계열 JSON 배열

수집은 이미 백엔드에서 자동이고, 프론트는 **"필요할 때 최신 스냅샷만 받아온다"** 입장. 실시간 필요도가 낮아 SSE를 쓸 이유가 없습니다.

## 왜 SSE가 아닌가

앞 세 L2는 모두 SSE/WebSocket을 썼습니다. 이 모듈만 HTTP 폴링을 쓰는 이유:

| 축 | SSE 모듈 (슬롯/벤치마크) | Metadata |
|---|---|---|
| 갱신 주기 | 1초 이하 | 수분(기본 5분) |
| 이벤트 수 | 초당 수십~수백 | 분당 몇 개 |
| 실시간성 요구 | 높음 | 낮음 |
| 연결 관리 복잡도 | 감수 필요 | 불필요 |

분 단위 주기를 위해 SSE를 상시 연결해 두는 것은 오버엔지니어링. 사용자가 대시보드를 열어둔 동안에만 30초 폴링으로 충분합니다 — 단계 5에서 봅니다.

## 엔드포인트 한 장 요약

이 튜토리얼에 나온 것 외에 Controller가 제공하는 다른 엔드포인트들:

| Method | Path | 용도 |
|---|---|---|
| GET | `/api/metadata/file` | VM에 저장된 JSON 파일 읽기 (MOVE 재시작 후에도 이력) |
| PUT | `/api/metadata/slot/{t}/{n}/enabled` | 슬롯별 수집 on/off 토글 |
| PUT | `/api/metadata/slot/{t}/{n}/interval` | 슬롯별 주기 변경 (최소 10초) |
| PUT | `/api/metadata/slot/{t}/{n}/excluded-types` | 타입별 제외 설정 |

프론트는 초기화 → 상태 조회 → 타입 선택 → 데이터 조회 순서로 호출합니다. 단계 5의 `MetadataDialog.svelte`가 이 조합을 사용.

## 슬롯 식별 규약 — tentacleName vs source

Head 내부에서는 슬롯 키가 `"source:slotIndex"` 형식이지만, **사용자에게는 `tentacleName`이 더 친숙**합니다. `source`가 Head 연결 이름(`compatibility-1`)이라면 `tentacleName`은 그 아래 묶인 설비 그룹 이름(`T1`).

컨트롤러는 `tentacleName`을 받아 내부적으로 `source`를 찾아주는 `findContextBySlot` 헬퍼를 씁니다 — **외부 경로와 내부 키 규약을 분리**해, 내부 구조가 바뀌어도 URL은 안정적.

## 왜 `monitoring: false`를 404가 아닌 200으로

직관적으로는 "모니터링 중이 아니면 존재하지 않음"이라 404가 맞아 보입니다. 하지만 MOVE는 **200 + `{monitoring: false}`** 로 보냅니다.

- 프론트가 **에러 처리 없이 `.monitoring` 필드만 체크**하면 됨
- 404는 "잘못된 경로" 같은 진짜 에러와 구분되기 어려움
- 슬롯은 존재하지만 아직 모니터링을 안 시작한 상태를 "200"이 더 정확히 표현

단, 타입 데이터(`/slot/{t}/{n}/{typeKey}`)는 **`findContextBySlot` 자체가 실패하면 404** — 완전히 없는 슬롯이면 진짜 에러로 간주.
