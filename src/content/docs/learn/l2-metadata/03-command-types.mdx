## 30초 요약

`MetadataMonitorService.monitorOnce()` 가 `cmd.getCommandType()` 으로 **switch 분기** → `MetadataCommandExecutor` 의 메서드 호출. 이 장에서는 **비바이너리 6가지** 를 다룹니다 (binary 는 다음 장):

| command_type | Executor 메서드 | 핵심 파싱 |
|---|---|---|
| `tool` | `executeCommand` | 출력을 그대로 반환 (JSON 가정) |
| `sysfs` | `executeSysfsRead` | regex 캡처 그룹 → keys 매핑 |
| `keyvalue` | `executeKeyValue` | 들여쓰기 스택 + 섹션 + 괄호 내 값 추출 |
| `raw` | `executeRaw` | 전체 출력을 `{basename: "..."}` 로 |
| `table` | `executeTable` | 3-column 섹션 (WRITE/READ/OTHER) |
| `bitmap` | `executeBitmap` | 라인 배열 `{format, path, lines: [...]}` |

모든 결과는 **`{...}` JSON 형태의 문자열** — monitorOnce 가 `ObjectMapper.readValue` 로 파싱해 `Map<String, Object>` 으로, `time` 필드 추가해 `CopyOnWriteArrayList` 에 누적.

## 왜 6 가지 분기가 필요한가

"전용 tool 로 JSON 만 받자" 라는 이상은 현실에 안 맞습니다. UFS 는:

- **업체마다 출력 포맷이 다름** — Samsung, SK hynix, Micron, WDC, …
- **같은 업체 내 펌웨어마다** 라인 수/섹션 헤더가 조금씩 다름
- **f2fs 가 제공하는 sysfs 파일** 은 f2fs 자체 구조라 업체 무관하지만, **human-readable** (예: 들여쓰기)
- **디바이스에 새 도구를 push 할 수 없는 경우** 가 있음 (production firmware, 읽기 전용)

여섯 분기는 이 다양성의 **합집합** — 각각 특정 상황에 최적:

- `tool`: 디바이스에 도구 push 가능 + 도구가 JSON 내줌 (이상적, 가장 깔끔)
- `sysfs`: 도구 push 없이 sysfs 라인 한 줄에서 여러 숫자 뽑기 (regex 로 단번에)
- `keyvalue`: f2fs status 처럼 구조화된 human-readable (코드 수정 없이 새 필드 자동 포함)
- `raw`: 파싱이 무의미한 큰 덩어리 — 원본 보관 + 프론트 후처리
- `table`: iostat_info 같은 고정 3-column 섹션 (WRITE/READ 축)
- `bitmap`: 수만 라인의 비트맵 — 라인 배열 보존해 프론트 Canvas 로 시각화

## 나머지 3 타입 요약 (`raw` / `table` / `bitmap`)

### `raw` — 파싱 불가능한 덩어리

```java
// 출력 전체를 key=basename 에 담아 JSON 화
{ "iostat_info": "Period: 1\nWRITE\n  app_buffered_data  12345 30 411\n..." }
```

테이블/비트맵을 **프론트에서** 파싱하고 싶거나, 디버깅용으로 원본을 그대로 보고 싶을 때. 파싱 비용 0, 스토리지 비용 크게 증가.

### `table` — f2fs iostat_info 3-column

```
WRITE
             app_buffered_data     12345  30    411
             app_direct_data           0   0      0
READ
             app_buffered_data    678901 120   5657
```

→

```json
{
  "WRITE": {
    "app_buffered_data": { "io_bytes": 12345, "count": 30, "avg_bytes": 411 },
    "app_direct_data":   { "io_bytes": 0,     "count": 0,  "avg_bytes": 0 }
  },
  "READ":  { "app_buffered_data": { ... } }
}
```

섹션 헤더(`WRITE`, `READ`, `OTHER`) 로 그룹핑, 라인은 공백 구분 4-token (이름 + 3 숫자). 차트에서 **섹션 × 3 컬럼** 구조를 그대로 활용.

### `bitmap` — 대용량 라인 배열

```
SECTION 0:
  row0 10101010...
  row1 11001100...
  ... 수천 라인
```

→

```json
{
  "segment_info": { "format": "segment_info", "path": "/sys/fs/f2fs/sda10/segment_info", "lines": ["row0 ...", "row1 ..."] }
}
```

JSON 파싱은 없고 **라인 배열 보존**. 프론트 `SegmentHeatmap` / `BitmapGridView` 가 Canvas 로 렌더 (6장).

## 직접 해보기

- [ ] sysfs command: `/sys/block/sda/stat | regex:(\d+)\s+\d+\s+(\d+)\s+\d+\s+(\d+) | keys:read_ios,read_sectors,write_ios` 등록 → in-memory 에서 3 key 가 나오는지
- [ ] keyvalue command: f2fs status 경로(`/sys/kernel/debug/f2fs/status`) 등록 → partition section prefix 가 붙는지
- [ ] `bitmap` command 로 `/sys/fs/f2fs/{userdata}/segment_info` 등록 → lines 배열 길이가 디바이스 segment 수와 일치하는지
- [ ] 일부러 틀린 경로로 `tool` command 등록 → adb 에러가 non-JSON 이라 skip 되는지 로그 확인
