## 이 모듈에서 배울 것

"사용자가 **슬롯에 UFS metadata 수집을 켜두면**, TC running 을 감지한 순간 Portal 이 자동으로 **해당 제품에 맞는 metadata 명령들** 을 찾아 **초 단위로 adb shell** 을 치고, UFS 공급사마다 출력 포맷이 전혀 달라서 **7가지 `command_type`** 으로 분기 파싱해 JSON 화하며, binary 덤프는 **BinMapper 모듈을 재사용**해 struct 로 매핑하고, f2fs 전용 3 뷰(Iostat·Heatmap·Bitmap)로 시각화한다."

스무 번째 L2입니다. [L2 19종 비교](/learn/l2-compare/)의 체크리스트를 따라 작성됐고, 이 모듈의 특이점은:

- **트리거가 TC running** — 사용자 RUN 버튼이 아니라 Head SlotState 변화를 **5초 폴링**으로 감지 (→ 자동 시작/종료)
- **7가지 `command_type` 분기** — UFS 컨트롤러·펌웨어·파일시스템마다 출력 포맷이 달라 **파싱 경로가 7개**. 새 포맷이 생기면 `command_type` 을 추가
- **binary 는 BinMapper 재사용** — adb pull → SFTP read → `binMapperService.parseFromBytes()` → `MappedResultFlattener` → flat JSON. BinMapper L2 를 **서비스 계층에서 호출하는 첫 소비자**
- **슬롯별 `ScheduledExecutorService(8 threads)`** — @Scheduled 는 "감지" 에만, 실제 수집은 슬롯 독립 태스크. 한 슬롯이 느려도 다른 슬롯이 밀리지 않음
- **`{userdata}` placeholder** — f2fs 파티션 block 이름(`sda10` 등) 이 디바이스마다 다르므로 `readlink /dev/block/by-name/userdata` 로 1회 조회 후 치환

## 앞 L2와의 차이

| 축 | 슬롯 | Metadata 모니터링 | T32 Dump | 🎯 Trace |
|---|---|---|---|---|
| 트리거 | Head TCP | **TC running 감지 (Head 폴링)** | Run+Confirm | 업로드 |
| 수명 | 페이지 이탈까지 | **TC running 지속 — 수십 분~수 시간** | 단발 수십 초 | 수 분 |
| 외부 시스템 | Head TCP | **SSH + adb shell (per-slot)** | SSH × 2 | Rust gRPC + MinIO |
| 분기 다양성 | - | **7 `command_type`** | 없음 (4 step 고정) | 3 trace type |
| 전송 | SSE | **HTTP polling (1초)** | SSE | polling + Arrow IPC |
| 영속 | 없음 | **DB 3 테이블 + in-memory + SFTP JSON** | FS+ZIP | DB + MinIO parquet |

가장 큰 특징은 "**TC 수명과 맞물린 자동 수집**" + "**분기 다양성**" . 사용자가 RUN 을 눌러 시작하는 게 아니라 TC 가 시작되면 자동으로 붙고, TC 가 끝나면 자동으로 마무리. 그리고 UFS 공급사별 파싱이 7가지로 분기됨.

## 왜 7가지 `command_type` 인가

UFS 는 제조사가 여럿이고 같은 제조사라도 펌웨어마다 sysfs/proc 출력이 다릅니다. **"전용 tool"** 만으로 가면 이상적이지만 현실은:

- 어떤 컨트롤러는 `adb shell tool` 로 JSON 을 바로 내주지만,
- 어떤 건 `cat /sys/block/sda/stat` 같은 **sysfs 라인** 을 읽어야 하고,
- 어떤 f2fs 상태는 **"key: value" 들여쓰기 형식** (human-readable) 이며,
- segment_info 같은 비트맵은 **라인 배열 그대로** 보존해야 시각화 가능,
- binary 덤프는 **C struct 로 매핑** 해야 읽힘.

하나의 파서로는 무리라서 `command_type` 을 필드로 두고 **파싱 경로를 분기**:

| command_type | 용도 | 파싱 |
|---|---|---|
| `tool` (기본) | 전용 tool 의 JSON 출력 | 그대로 파싱 |
| `sysfs` | sysfs/proc 파일 라인 | regex 캡처 → 키 매핑 |
| `keyvalue` | "key: value" 들여쓰기 | 스택 기반 depth + dot-notation |
| `raw` | 테이블·비트맵 덩어리 | 출력 전체를 value 에 저장 |
| `table` | f2fs iostat_info 3-column 섹션 | 섹션 + 3컬럼 파싱 |
| `bitmap` | f2fs segment_info 대용량 라인 | 라인 배열 보존 |
| `binary` | UFS 바이너리 덤프 | binmapper struct 매핑 → flatten |

새 UFS 세대가 나와도 **새 `command_type` 을 추가하면 됨** — Open/Closed 원칙을 DB 필드로 실현.

## 왜 `ScheduledExecutorService` 인가 (@Scheduled 아니고)

Spring `@Scheduled` 한 개만 쓰면 "모든 슬롯을 한 루프" 에서 돌게 됩니다. 한 슬롯이 SSH 지연에 걸리면 **뒤 슬롯들이 전부 대기**. 그래서:

- `@Scheduled(5000ms)` 는 **"감지" 전용** — 가볍게 running 전이만 본다
- 실제 수집은 **슬롯마다 독립 `ScheduledFuture`** — 8 스레드 풀에서 `scheduleAtFixedRate(monitorOnce, 0, intervalSec)`
- 한 슬롯이 10 초 걸려도 다른 7 슬롯은 영향 없음

**"감지자 + 작업자 분리"** 는 L2 공통 원칙 #3 ([l2-compare 참고](/learn/l2-compare/#설계-결정의-공통-원칙)) — Head SSE, Agent job 과 같은 패턴.

## 왜 BinMapper 모듈을 재사용하는가

BinMapper 는 원래 **devtool** (사용자가 struct + binary 를 업로드하면 파싱된 트리를 보여줌). Metadata 의 binary `command_type` 은 같은 파이프라인을 **서비스 계층에서 호출** :

- `binMapperService.parseFromBytes(data, predefinedStructId, ...)` → `MappedResult`
- `MappedResultFlattener.flatten(mapped)` → dot-notation `Map<String, Object>`

즉 **BinMapper 의 첫 in-process 소비자**. `predefined_structs.kind='metadata'` 로 사전을 분리해서 사용자가 Admin UI 에서 metadata 전용 struct 만 고를 수 있게 됨.

> 이런 재사용 구조가 L2 비교의 "**엔진 축**" 이 가지는 힘 — 순수 엔진은 devtool 로도, 서비스로도 쓰일 수 있다.

## 왜 1초 HTTP polling 인가 (SSE 아니고)

TC 수집은 **수십 분~수 시간** 지속. 사용자는 대부분 그 시간 동안 다른 화면에서 일하다가 **가끔 열어본다**. SSE 는 연결 비용 + 서버 resource 가 상주로 붙는 구조라 오버엔지니어링. 반면:

- `setInterval(1s)` + `fetch('/api/metadata/slot/.../typeKey')` 는 **시트가 열릴 때만** 트래픽
- 시트 닫히면 `clearInterval` — 자원 해제가 간단
- 네트워크 flake 에 관대 — 한 번 실패해도 다음 tick 이 복구

**"연결 유지 비용 vs 조회 빈도"** 의 전통적 갈림길 — 조회 빈도가 **사용자 시야에 있을 때만** 이라면 polling 이 단순.

## 학습 순서와 예상 시간

| 단계 | 제목 | 성격 | 시간 |
|---|---|---|---|
| 1 | 엔티티 & 7가지 `command_type` | 읽기 | 8 분 |
| 2 | 스케줄러 & 슬롯 상태 감지 (@Scheduled + pool) | 코드 투어 | 12 분 |
| 3 | 6 가지 adb shell 파싱 방식 | 코드 투어 | 15 분 |
| 4 | binary `command_type` + BinMapper 통합 | 코드 투어 | 10 분 |
| 5 | REST API & `{userdata}` placeholder | 코드 투어 | 10 분 |
| 6 | MetadataDialog & f2fs 전용 뷰 3종 | 의사코드 | 12 분 |

총 약 **67 분**. BinMapper L2 를 먼저 읽었으면 4장은 빠르게 넘겨도 무방.

## 새로 등장할 L3 개념

없음 — `@Scheduled` / `ScheduledExecutorService` / `ConcurrentHashMap` / JSch SSH 는 이미 앞 L2 에서 다뤘습니다. 새로 만나는 개념은 **"7가지 파싱 분기"** 라는 **도메인 패턴** 이지 프레임워크 개념이 아님.

기존 L2 에서 이미 보았을 것:

- `@Scheduled` + 슬롯별 Executor (Agent L2, TestDB Reparse)
- JSch SSH + adb shell (Pre-Command L2, T32 L2)
- `MappedResult` + BinMapper (BinMapper L2 — 이번이 **서비스로 재사용** 하는 첫 사례)
- 1초 polling + `$effect` cleanup (TestDB Reparse L2 와 유사)
