## 30초 요약

UFS 컨트롤러는 상태 덤프를 **C struct 바이너리** 로 제공하는 경우가 많습니다. 이걸 JSON 으로 바꾸는 기존 경로:

1. **adb shell** 로 tool 실행 → 디바이스가 `/dev/debug_ssr.bin` 같은 경로에 덤프
2. **adb pull** → tentacle VM 의 `/tmp/metadata-binary-{nanoTime}.bin`
3. **SFTP read** → Portal 의 `byte[] data`
4. **`binMapperService.parseFromBytes(data, predefinedStructId, ...)`** → `MappedResult` (중첩 struct 트리)
5. **`MappedResultFlattener.flatten(mapped)`** → `Map<String, Object>` (dot-notation)
6. **Jackson** → JSON 문자열 반환

5단계가 핵심 — **BinMapper L2 를 devtool 외에 서비스 계층에서 호출하는 첫 사례**.

## 왜 BinMapper 를 재사용하는가

[BinMapper L2](/learn/l2-binmapper/) 는 원래 **devtool**:

- 사용자가 C struct 텍스트 + 바이너리 파일을 업로드 → 파싱된 필드 트리를 Table/Hex/JSON 로 표시
- stateless · pure function · 한 번 실행하고 버리는 구조

Metadata 의 binary `command_type` 은 같은 파이프라인을 **서비스로** 호출:

- `predefined_structs.kind='metadata'` 사전에 struct 등록 (Admin UI)
- 사용자는 Admin 에서 **struct 고르기만** — 실제 매핑은 monitorOnce 가 자동

이 구조의 장점:

- **struct 정의는 DB 에** — binary 포맷이 바뀌면 Admin 에서 struct 수정, 코드 배포 없음
- **sample 데이터를 BinMapper devtool 에서 미리 검증** — 같은 파이프라인이라 devtool 검증 = 서비스 동작
- **엔진 한 개**를 두 진입점(devtool UI · 서비스 호출) 으로 공유

이게 "**엔진 축**" L2 의 가치를 증명하는 지점 — L2 비교에서 BinMapper 를 엔진으로 분류한 이유가 여기서 실현됨.

**변환 예시**:

```java
// MappedResult (트리)
smart_data {
  temperature: 45
  wear_level {
    host: 10
    ffu: 2
  }
  bad_blocks: [120, 122, 150]
}

// flatten 후
{
  "smart_data.temperature": 45,
  "smart_data.wear_level.host": 10,
  "smart_data.wear_level.ffu": 2,
  "smart_data.bad_blocks[0]": 120,
  "smart_data.bad_blocks[1]": 122,
  "smart_data.bad_blocks[2]": 150
}
```

## Admin UI 에서 binary command 등록

Admin → Metadata 탭 → Commands 섹션에서 **"binary"** 선택 시 3 필드가 추가 노출:

1. **`predefinedStruct`** 드롭다운 — `kind='metadata'` 인 사전만 표시 (BinMapper 의 전체 사전이 아니라 **metadata 전용 subset**)
2. **`binaryOutputPath`** — 디바이스 내 덤프 경로 (예: `/dev/debug_ssr.bin`)
3. **`binaryEndianness`** — LITTLE / BIG / AUTO 드롭다운 (기본 LITTLE)

> `predefined_structs.kind` 는 BinMapper 쪽 추가 (2026-04-22~23 세션). **metadata / dlm / general** 3종 — metadata 모니터링 용도로 구분된 struct 만 Admin UI 에서 선택 가능해 사용자 실수 방지.

## 왜 AUTO endianness 를 기본으로 두지 않는가

AUTO 는 매 수집마다 struct 의 magic byte 필드를 검사해 판정. 하지만:

- 매 호출 CPU 비용 (작지만 누적)
- magic 이 없는 struct 는 실패
- UFS 컨트롤러는 **대부분 LITTLE** 로 고정

그래서 **LITTLE 기본** + 필요 시 AUTO 선택. BIG 은 드물지만 일부 legacy 장비용.

## 실패 모드와 복구

| 실패 | 동작 | 복구 |
|---|---|---|
| adb shell 실패 (tool 없음) | RuntimeException | 다음 tick 에 다시 시도 |
| adb pull 실패 (경로 없음) | RuntimeException | 같음 |
| SFTP read 실패 | RuntimeException | 같음 |
| binmapper 파싱 실패 (struct mismatch) | RuntimeException | 다음 tick, struct 재검토 필요 |
| JSON 직렬화 실패 (비정상 타입) | RuntimeException | BinMapper 의 parsedValue 타입 재점검 |

monitorOnce 가 per-command try/catch 라 **한 binary 실패가 다른 command 를 막지 않음** — 같은 슬롯의 다른 타입은 정상 수집 계속.

## 직접 해보기

- [ ] BinMapper devtool(`/devtools/binmapper`) 에서 metadata struct 정의 + sample binary 로 파싱 → `MappedResult` 트리 확인
- [ ] 같은 struct 를 `predefined_structs.kind='metadata'` 로 저장 → Admin → Metadata → Commands 에서 선택 가능해지는지
- [ ] 일부러 잘못된 predefinedStructId 로 binary command 등록 → monitorOnce 가 해당 command 만 skip 하는지 (다른 command 는 정상)
- [ ] binaryEndianness=BIG 로 변경 → 같은 struct 가 완전히 다른 값을 내는지 (bit 순서 역전)
