## 30초 요약

IR(`StructDefinition`)이 있으면 실제 매핑은 세 단계:

1. **calculateStructSize** — 필드마다 `align(offset, alignment)` 누적해 총 크기
2. **mapFields** — 재귀 하강으로 struct/union/bitfield/char[]/array/primitive 6 분기
3. **ByteBuffer.wrap(data, offset, size).order(byteOrder)** — 타입별 `.getInt() / .getFloat() / ...`

**자바 Reflection 없음** — 필드 위치를 모두 수동 계산. "리플렉션은 Java 객체용이지 바이트 맵용이 아니다"는 철학.

## 왜 Java Reflection을 안 쓰는가

Java 객체 필드를 바이트로 매핑하려면 Reflection이 자연스럽지만 BinMapper는 **쓰지 않음**:

| 축 | Java Reflection 접근 | BinMapper 실제 |
|---|---|---|
| 사용자 정의 타입 | Java class 필요 → 런타임 컴파일 복잡 | **문자열로 struct 정의** → DB에 저장 쉬움 |
| 타입 변환 | `Field.setInt(obj, value)` | `ByteBuffer.getInt()` 직접 |
| 성능 | Reflection 호출 비용 | 직접 메서드 호출로 빠름 |
| unsigned 지원 | Java Integer에 표현 불가 | Long 승격, String(UINT64) |
| 사용자 영역 | Java class 업로드 필요 | C struct 텍스트 — **C/FW 팀이 그대로 복붙 가능** |

**핵심 통찰**: "Java 객체를 바이트로" 가 아니라 "바이트를 **설명 가능한 트리로**" 가 목적. Reflection은 전자 문제 해결책이고, 후자에는 맞지 않음.
