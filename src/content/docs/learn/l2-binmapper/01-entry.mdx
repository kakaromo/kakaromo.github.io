## 30초 요약

`POST /api/binmapper/parse`는 **한 엔드포인트에 6개 옵션 파라미터**. 세 묶음으로 나뉨:

- **바이너리 소스**: `binaryFile`(MultipartFile) 또는 `serverPath + serverName`(Log Browser 경로)
- **Struct 소스**: `structText`(inline) / `structFile`(MultipartFile) / `predefinedStructId`(DB FK) / `structName`(다중 struct에서 선택)
- **옵션**: `endianness`(AUTO/BIG/LITTLE) · `repeatAsArray`(bool)

`PredefinedStruct` 엔티티는 **Svelte devtool UX를 지탱하는 사전** — 한 번 만든 struct를 이름으로 불러오기 위함.

## 왜 한 엔드포인트에 6 옵션인가

REST에서는 보통 "입력 형태마다 다른 path" 가 권장되지만 BinMapper는 **반대로 한 API에 몰아넣음**:

```
POST /api/binmapper/parse
  (binaryFile  | serverPath + serverName)  ×  (structText | structFile | predefinedStructId)
```

이유:

- 바이너리-struct **조합이 6×2=12개** — 경로를 분리하면 프론트가 분기 로직 12개
- **공통 응답 타입** (`MappedResult`) — 경로를 나눠도 return shape 동일
- struct 소스는 "텍스트냐 파일이냐"가 **트랜스포트 차이**일 뿐 의미는 같음

대신 컨트롤러 서명이 길어짐 — 그 비용을 받아들임.

## PredefinedStruct의 UX 의미

엔티티가 단순하지만 프론트 devtool에서 **핵심 기능**을 만듦:

```
사용자: "이 UFS Descriptor 파싱을 또 해야 하네"
       ↓
첫 번째: struct 텍스트 입력 → parse → 저장 (name="UFS Desc v3")
       ↓
두 번째: 드롭다운에서 "UFS Desc v3" 선택 → bin 업로드 → 즉시 결과
```

- **팀 단위 공유 가능** — 모든 사용자가 같은 사전 조회
- **category 필터**로 "UFS / ELF / 트레이스 프레임" 같은 그룹화
- **description**이 "v3는 bitfield 2개 추가됨" 같은 메모 — 버전 추적 대체

조직의 **"반복 struct 정의"를 DB에 축적해 지식으로 전환**하는 장치.

## 서버 경로 모드 — Log Browser와의 연결

`serverPath + serverName`이 채워져 오면 BinMapperService가 **내부적으로 Log Browser API를 호출**:

```
POST /api/binmapper/parse
  serverName: "T10"
  serverPath: "/home/octo/ufs/descriptor.bin"
  predefinedStructId: 7
```

→ Service가 `logBrowserService.readFileBytes("T10", serverPath)` 호출 → byte[] 로드 → 나머지는 동일.

**Log Browser L2 재사용**: Head 전용 경로 분기·SSH vs Local 전략을 BinMapper가 다시 구현하지 않음. Log Browser L2에서 본 **"하부 재사용 레이어"** 원칙이 그대로 작동.

## 직접 해보기

- [ ] `POST /api/binmapper/parse`에 `structText`와 `binaryFile` 모두 없이 호출 → 의미 있는 에러가 돌아오는지 확인
- [ ] PredefinedStruct 하나를 DB에 `INSERT` → API로 list 조회 → devtool 드롭다운에 뜨는지
- [ ] `structFile`에 여러 struct가 든 `.h` 업로드 → `structName`이 없으면 **첫 번째** struct가 쓰이는 관례를 코드에서 확인
