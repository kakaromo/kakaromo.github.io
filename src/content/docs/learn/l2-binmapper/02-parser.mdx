## 30초 요약

- **Lexer**: 문자 스트림 → Token 스트림 (`STRUCT`, `IDENT`, `LBRACE`, `NUMBER` 등 15종)
- **Parser**: Token 스트림 → `StructDefinition` 리스트 (재귀 하강 / top-down)
- **상태 1개**: `pragmaPackValue` — `#pragma pack(N)` 만나면 바뀌고 이후 struct에 적용
- **에러 복구 없음** — invalid 입력은 조용히 skipToSemicolon — 엄격 모드 대신 "될 수 있는 부분만"

## "왜 ANTLR / JavaCC 를 안 쓰나"

생성기 기반 파서는 학습·빌드·디버깅 오버헤드. 손으로 쓴 이유:

- **지원 범위가 고정** — "필요하면 추가"가 아니라 "이 패턴만" → 생성기의 grammar 추상화가 오히려 제약
- **에러 허용이 다름** — 생성기는 보통 strict. BinMapper는 **"모르면 skip"** 이 핵심 — skipToSemicolon 같은 복구 로직이 자연스러움
- **의존성 절감** — Samsung 사내 빌드 정책상 새 의존성 추가가 비용. Java 표준만으로 구현
- **개발자 이해 용이** — 새 팀원이 재귀 하강은 읽을 수 있지만 `.g4` 문법 파일은 낯섦

결과: ~1000줄의 Lexer/Parser이지만 **전원이 읽고 고칠 수 있음**.

## 상태 머신 vs 재귀 하강

- **Lexer는 상태 머신** — 다음 문자만 보면 어느 토큰으로 갈지 결정 (피크 1회)
- **Parser는 재귀** — `parseStruct → parseField → parseType → parseStruct`(중첩 struct) 재귀가 자연스럽게 AST 모양을 만듦

이 조합이 **"간단한 Lexer + 풍부한 Parser"** 원칙 — 복잡성을 Parser로 몰아넣어 Lexer는 상수 시간.

## 직접 해보기

- [ ] `#pragma pack(4)` 선언 뒤의 struct와 `#pragma pack(1)` 뒤의 struct를 한 파일에 넣어 → 각각 calculateStructSize가 다른 값을 반환하는지
- [ ] typedef 체인(`typedef int32_t MyInt; typedef MyInt YourInt;`)을 넣으면 Parser가 어떻게 처리하는지 (힌트: CppType.fromString의 기본 alias map 범위 확인)
- [ ] 일부러 잘못된 token 순서 (예: `struct { int x }` — 세미콜론 빠짐)를 넣고 → Parser가 어디서 `skipToSemicolon`으로 복귀하는지
