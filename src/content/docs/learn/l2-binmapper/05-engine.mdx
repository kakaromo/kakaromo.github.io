## 30초 요약

BinMapper는 **작은 컴파일러**입니다. 입력/출력과 단계가 대응됩니다:

| 전통 컴파일러 | BinMapper |
|---|---|
| 소스 코드 문자열 | C struct 정의 텍스트 |
| Lexer → Token | `CppStructLexer` → Token |
| Parser → AST | `CppStructParser` → `StructDefinition` |
| Semantic Analysis → IR | (fromString alias 해석 / pragma pack 반영) |
| Codegen → machine code | `BinaryReaderService` → `MappedResult` 트리 |

엔진이라는 시각이 **왜 다른 L2와 모양이 다른지**를 설명합니다. 이 페이지는 그 관점을 정리하고, 13 L2 전체에서의 위치와 향후 확장을 제시.

## 단계별 대조

### 입력

| | 전통 컴파일러 | BinMapper |
|---|---|---|
| 주체 | 프로그래머가 `.c` 작성 | FW 팀이 struct 정의 |
| 대상 | 전체 언어 | 도메인 부분 집합 |
| 형식 | 파일 시스템 | API body(text/file) |

BinMapper는 **"입력의 shape가 웹 요청"** 이라는 점이 다름. 본질은 같음.

### Lexer

둘 다 **문자 단위 상태 머신**. BinMapper Lexer는 `#pragma pack`과 `__attribute__` 같은 **C 전처리기 일부**를 직접 토큰화. 전처리와 lexing의 경계가 흐릿한 C의 특성을 반영.

### Parser

둘 다 **재귀 하강** 가능. BinMapper는:

- LL(1) — peek 1개 토큰으로 결정
- 에러 허용 — skipToSemicolon으로 복귀
- 중간 결과 저장 — typedefMap으로 앞선 정의를 이후 struct가 참조

### IR

- 전통: Control Flow Graph · SSA form · tree IR 등 복잡
- BinMapper: `StructDefinition` — **이름 + field 리스트 + packed/alignment 플래그**
- 소비 단계가 Codegen 하나뿐이라 **IR을 단순하게 유지**. "IR 풍부도는 소비 단계 수에 비례"

### Codegen

- 전통: x86/ARM 기계어 방출
- BinMapper: `MappedResult` 트리 구성 — byte 배열을 field별로 slice해 값 추출

**"Codegen"이 아니라 "Data Binding"** 이라 부르는 게 더 정확하지만 **파이프라인 마지막 단계** 라는 점은 동일.

## 무엇이 없는가 — 의도된 단순화

전통 컴파일러에 있고 BinMapper에 **없는** 단계:

| 단계 | 전통 | BinMapper 입장 |
|---|---|---|
| 전처리기 매크로 전개 | `#define` 치환 | 생략 (사용자가 손으로 치환) |
| 타입 체크 | Semantic phase | 런타임 in readValue (타입 불명 → "unknown") |
| 최적화 | 수십 가지 pass | 없음 (한 번 돌고 버림) |
| 심볼 테이블 | 여러 스코프 | typedefMap / enumMap 2개만 |
| 에러 리포팅 | 위치 정확 | skip 우선, 자세한 메시지 최소 |

**왜 생략 가능한가**:

- 입력이 수십 줄 단위 — 최적화할 코드량이 없음
- 사용자는 FW 엔지니어 — 자기가 쓴 C를 스스로 검증 가능, 파서의 친절한 에러가 덜 중요
- 한 번 돌고 버리는 일회성 엔진 — stateful 파이프라인이 아닌 **pure function**

## 13 L2 전체 지도에서의 BinMapper

```
크로스-컷 2종:
  🔐 Auth    — 요청 축
  📊 TestDB  — 데이터 축

기능 축 10종:
  실시간: 슬롯 · Metadata
  Job: Agent 벤치마크
  터미널: 원격
  절차: T32 · Pre-Command
  재사용 레이어: Log Browser · MinIO · Bitbucket
  데이터 통합: Excel Export

엔진 축 (신규, 1종):
  🧬 BinMapper — 독립 devtool · 입력→파이프라인→출력
```

**세 번째 축 "엔진"**: 기능도 크로스-컷도 아닌 **"계산 그 자체"** 인 모듈. 향후 비슷한 성격:

- **iotest syscall 엔진** — I/O 명령 DSL → 실행 → 결과 (BinMapper 구조와 유사)
- **Perf Generator** (/devtools/perf-generator) — JSON 입력 → Svelte 코드 생성 (이것도 엔진 축)

## 확장 포인트

### 1. typedef 체인 완전 전개

현재 `typedef int32_t MyInt; typedef MyInt YourInt;` 에서 **두 번째 typedef가 작동할지 불확실**. `CppType.fromString`이 실패하면 Parser가 해당 field를 "unknown"으로. 개선 방향:

- Parser가 typedefMap을 checked lookup으로 깊이 순회
- 순환 참조 방어 (`typedef A B; typedef B A;`)

### 2. Bitfield 정확한 bit 위치 추적

현재 simplified: 각 bitfield가 독립 storage unit을 읽음. 실제 C 의미:

```c
struct S { uint8_t a:3; uint8_t b:5; };  // 같은 1 byte 안에 a(0-2) + b(3-7)
```

개선 방향:

- `StructField`에 `bitOffset` 추가
- Parser가 연속된 bitfield를 같은 storage unit으로 그룹핑
- Reader가 각 bitfield의 `(offset, bitOffset, bitWidth)` 3-tuple로 정확한 추출

### 3. preprocessor 매크로 전개

`#define MAX_NAME 16` 이후 `char name[MAX_NAME];` 를 허용하려면:

- Lexer가 `#define` 값을 Map에 저장
- 이후 IDENT가 Map에 있으면 값으로 치환

단순 case-by-case 전개면 수십 줄 추가로 가능. 복잡한 매크로(`#define FOO(x) ...`)는 여전히 out-of-scope.

### 4. 사용자 정의 타입 참조

현재 user-defined type은 **같은 파싱 텍스트 안**에 있어야. 다른 .h의 struct를 참조하려면:

- `PredefinedStruct`를 외부 라이브러리로 취급
- Parser 초기화 시 typedefMap에 미리 load
- 사용자가 "import structs" 같은 방식으로 의존성 선언

### 5. 엔진 API의 다른 소비자

현재는 devtool UI가 유일한 소비자. 엔진을 **다른 L2에서 프로그래밍 호출** 가능:

- Log Browser에서 `.bin` 파일 미리보기 (Log Browser가 BinMapperService 호출 → 결과 inline 렌더)
- T32 Dump 결과에 자동 struct 매핑
- Agent trace frame 해석

**엔진으로서의 진짜 가치**는 이 확장에서 나옴 — 지금은 devtool 한 소비자지만, 내부 라이브러리로 재사용 가능한 씨앗.

## 배운 것

BinMapper를 통해 13 L2 학습 여정에서 얻은 새 관점:

- **L2 템플릿의 탄력성** — 웹 컨트롤러 + 서비스 + 프론트의 외피로 **컴파일러를 포장** 가능
- **"엔진" 축**의 부상 — 지금까지 "기능"과 "크로스-컷"만 있었는데 세 번째 축 도입
- **파이프라인 사고** — unidirectional + stateless 설계가 디버깅·테스트·확장 모두 유리
- **의도된 단순화** — 전체를 지원하려다 복잡해지는 대신 "우리가 자주 쓰는 것만" 이 적중하면 효과 큼
