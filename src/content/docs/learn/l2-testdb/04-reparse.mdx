## 30초 요약

기존 History의 원본 로그는 보존된 채, **파서만 재실행**해 결과 JSON을 갱신하는 비동기 Job 시스템.

- **ReparseJob** = 상태 머신 (preparing/running/completed/failed) + 진행률
- **PerformanceReparseService** = 4-worker 풀 + Job 등록/조회/SSH 실행
- **history → Job 역 매핑**으로 **같은 history 중복 리파싱 차단**
- 경로 해석은 fetchResultData와 **동일 로직** (코드 중복 허용 — 서로 다른 쓰임)

## 왜 SSE가 아니라 version 폴링인가

리파스 진행률을 UI에 보여줘야 하지만 MOVE는 **SSE 대신 AtomicLong version + 폴링** 선택.

| 축 | SSE (e.g. 벤치마크) | version 폴링 (Reparse) |
|---|---|---|
| 필요 속도 | 초 단위 로그 실시간 | 수 초 허용 |
| 동시 job 수 | 장시간 소수 | 단시간 다수(4 병렬 × 여러 사용자) |
| 구현 비용 | SseEmitter + 브릿지 | Atomic++ + GET |
| 실패 시 재시작 | 재연결 로직 필요 | 다음 폴링에서 자연 복구 |
| 브라우저 연결 한도 | 도메인당 ~6 | 무제한(요청마다 새) |

**수초 허용 + 동시 많음** → 폴링이 더 단순. MOVE의 모든 "작업 진행률"이 SSE를 쓰는 건 아니라는 예시.

## "같은 코드 중복 허용"의 판단

fetchResultData와 Reparse의 경로 해석이 **똑같이 3브랜치**. DRY 유혹이 강하지만 분리한 이유:

- **쓰임이 다름**: Data는 파일 읽기(read), Reparse는 SSH 명령 생성(write) — 시그너처가 다음에 갈라질 여지
- **수명이 다름**: Data는 요청마다, Reparse는 Job 생성 시 한 번
- 현재 3줄 × 3브랜치 = 9줄 정도라 **추출 비용이 결합도 비용보다 큼**

중복이 생기면 주석으로 "another copy in PerformanceResultDataService" 같은 pointer를 두는 편이 안전. 코드베이스 전체에서 이 규칙은 **"중복 2번까지는 OK, 3번째에 추출"** (Rule of Three) 근사치로 운영.
