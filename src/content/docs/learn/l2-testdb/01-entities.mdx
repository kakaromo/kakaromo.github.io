## 30초 요약

- **PerformanceTestRequest(TR)** — FW 메타만. TR 하나가 여러 TC 실행에 공유됨. "어떤 펌웨어를 검증하는가"
- **PerformanceTestCase(TC)** — 시나리오 정의 + `parserId`(측정 결과 파일 이름). "어떻게 측정하는가"
- **PerformanceHistory** — TR × TC 실행 한 번의 결과. TR/TC를 ManyToOne으로 참조.

세 엔티티 모두 `implements Serializable` — **Redis 캐시가 JDK serializer를 쓰기 때문에** 필수.

## 왜 "도메인 × 3-entity" 패턴인가

Performance와 Compatibility 두 도메인이 **동일한 구조**를 복제해 씁니다.

| 의도 | Performance | Compatibility |
|---|---|---|
| FW 정체성 | PerformanceTestRequest | CompatibilityTestRequest |
| 시나리오 정의 | PerformanceTestCase | CompatibilityTestCase |
| 실행 결과 | PerformanceHistory | CompatibilityHistory |

**공통 추상화 안 쓴 이유**:

- 도메인별 필드가 다름 — PerformanceHistory는 setId/fileSystem, CompatibilityHistory는 setSerial/setProductName/failCause
- `@MappedSuperclass` 도입 시 공통 필드 3개 + 도메인 필드 10+개가 섞여 **가독성 악화**
- JPA 상속(`@Inheritance`)은 테이블 조인/식별 컬럼이 추가돼 **대시보드 집계 쿼리가 복잡해짐**

**복제의 비용은 "바꿀 때 두 번"인데, 두 도메인이 서로 영향을 거의 안 줌** → Samsung 테스트 프로세스상 성능/호환성 이벤트가 분리 운영되므로 허용.

## Redis JDK Serializer의 의미

```
ObjectMapper(Jackson) → JSON 직렬화 → fw 키 포함 → deserialize → setFw(String) 호출 ❌
JDK Serializer       → 바이너리 stream → @Transient 무시 → setFw 호출 안 됨 ✅
```

**MOVE는 JDK Serializer를 선택**:

- Hibernate proxy + JSON deserialize의 `activateDefaultTyping` 충돌 회피
- `@Transient`가 저절로 무시됨 — `setFw` 빈 메서드는 "혹시 모를" 안전망
- 단점: 스키마가 바뀌면 `serialVersionUID` 충돌 가능. 배포 전 `FLUSHALL` 운영 정책으로 완화
