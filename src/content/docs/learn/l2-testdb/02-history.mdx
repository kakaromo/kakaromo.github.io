## 30초 요약

- **logPath = 원격 파일로의 포인터 문자열** — DB에는 경로만, 실제 JSON은 tentacle `/home/octo/tentacle/history/...` 또는 HEAD `/home/octo/nas/NAS/...`
- **PerformanceHistory** vs **CompatibilityHistory** — 시간 타입(String vs LocalDateTime), 장치 메타(setId vs setSerial 직접), failCause 유무가 갈림
- **result 상태**: `PASS` · `FAIL` · `RUNNING` · (간혹 `COLLECTING` UI 가상 상태) — RUNNING은 logPath가 null일 수도 있음

## logPath 해석 3 브랜치

`PerformanceResultDataService.fetchResultData`가 결정하는 경로 규칙:

| 조건 | tentacleName | remotePath |
|---|---|---|
| **logPath not null + contains "UFS"** | `HEAD` | `{headLogPath}/NAS/{dirPath}/{parser}.json` |
| **logPath not null + no "UFS"** | `slotLocation → T\\d+` 추출 | `{logPrefix}/history/{dirPath}/{parser}.json` |
| **logPath null** (RUNNING 중) | `slotLocation → T\\d+` 추출 | `{logPrefix}/slot{N}/log/{parser}.json` |

`{headLogPath}` = `tentacle.head.log-path` (기본 `/home/octo/nas`)
`{logPrefix}` = `tentacle.log-prefix` (기본 `/home/octo/tentacle`)

**규칙의 의미**:

- "UFS" 포함 = HEAD에서 실행된 UFS 파일시스템 테스트 → 공용 NAS
- 일반 완료 TC → tentacle 자체 디스크 `history/`
- RUNNING 중 → tentacle의 `slot{N}/log/` 라이브 경로. **아직 history/로 이동 안 됨**

## 성능 vs 호환성 — 한 눈에

| 축 | PerformanceHistory | CompatibilityHistory |
|---|---|---|
| ID 생성 | @GeneratedValue IDENTITY | 외부 제공 (@Id만) |
| 시간 타입 | String (tentacle format) | LocalDateTime |
| 장치 메타 | setId FK (SetInfomation) | 4개 컬럼 평탄화 |
| 실패 원인 | JSON 차트 안 | failCause 전용 컬럼 |
| 식별 위치 | slotLocation (T10-4) | tentacleIp + usbId |
| 결과 상태 | PASS/FAIL/RUNNING | PASS/FAIL (RUNNING 드묾) |

## RUNNING의 의미 — 두 층

**DB 상태**: result = `"RUNNING"` — tentacle이 아직 완료 이벤트를 안 보낸 상태.

**UI 가상 상태**: `"COLLECTING"` — DB에는 없음. fetchResultData가 던지는 `IllegalArgumentException("COLLECTING:{path}")`를 컨트롤러가 **200 OK + body.status="COLLECTING"**으로 전환:

- "결과는 아직 완성이 아니지만 차트 그릴 시도는 해볼 수 있음"
- 프론트가 이 상태를 받으면 1-2초 후 재호출 — 폴링 없이 **lazy refresh**

이 설계로 **완료 알림 채널(SSE/웹훅)을 안 만들어도 됨** — 사용자가 궁금할 때만 재요청.

## 직접 해보기

- [ ] `PerformanceHistory` row의 logPath가 `UFS_...` 문자열을 포함한 경우 `fetchResultData`가 `tentacleName="HEAD"` 로 가는지 디버거로 추적
- [ ] result="RUNNING" + logPath=null인 상태에서 fetchResultData 호출 → `IllegalArgumentException("COLLECTING:...")` 확인
- [ ] `CompatibilityHistory.ID`를 중복 생성 시 JPA가 EntityExistsException을 던지는지 (GeneratedValue 없기 때문)
