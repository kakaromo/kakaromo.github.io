## 30초 요약

TestDB는 **앞 10 기능 L2 중 6개와 직접 연결**. 어떤 축으로 접속하는지 정리해두면 새 L2 설계 시 "TestDB를 쓸지"를 빠르게 판정할 수 있습니다.

## 6 소비자 × 접점 표

| L2 | TestDB 접점 | 방향 | 핵심 메서드/엔티티 |
|---|---|---|---|
| **슬롯** | `SlotInfomation.testcaseIds` CSV → TC 이름 조회 | 읽기 | `PerformanceTestCaseService.findById` |
| **Agent 벤치마크** | 완료 시 PerformanceHistory 쓰기 (외부 tentacle 트리거) | 쓰기 | `PerformanceHistoryService.create` |
| **Excel Export** | `fetchResultData` 공유 | 읽기 | `PerformanceResultDataService.fetchResultData` |
| **Pre-Command** | TC 상태 배열(`testcaseStatus`)에서 첫 미완료 position | 읽기 | `SlotInfomation` |
| **T32 Dump** | TR/TC 이름 조합 → 결과 폴더명 `{date}_{setLocation}_{testToolName}_{testTrName}` | 읽기 | `PerformanceTestCase.name` + `PerformanceTestRequest.getFw()` |
| **Log Browser** | History.logPath가 가리키는 디렉토리를 브라우징 | 읽기 | `LogBrowserController.listFiles(tentacleName, path)` |

## 슬롯 — CSV ID 풀어서 이름 붙이기

`SlotInfomation.testcaseIds`가 `"42,15,7"` 형태의 CSV. 슬롯 카드 렌더 시:

```pseudo
for (Long tcId : parseCsv(slot.testcaseIds)) {
  PerformanceTestCase tc = tcService.findById(tcId);
  renderChip(tc.getName(), statusForIndex(i));
}
```

- **왜 CSV인가**: 중간 테이블(slot_tc) 없이 순서를 단순 문자열로 유지 → 수천 행의 슬롯마다 별도 row 생성 회피
- **Pre-Command와 공유**: 똑같은 CSV를 tcPreCommandIds도 사용 — "position 기반 다중 ID" 패턴이 MOVE에서 재사용됨

## Agent 벤치마크 — 완료 시 History 쓰기

벤치마크 L2는 Portal → Agent(gRPC 스트림)로 실행 → Agent가 완료 시 **tentacle이 직접 PerformanceHistory row 생성**. Portal은 쓰기 주체가 아님.

```pseudo
// tentacle 내부
performanceHistoryClient.create({
  trId, tcId, setId, slotLocation: "T10-4",
  logPath: "20260419_T10-4_SEQREAD",
  result: "RUNNING"  // 완료 후 "PASS"로 update
});
```

- **portal의 역할**: 읽기(조회·대시보드·Excel)만. 쓰기 경로에는 관여 X
- **책임 분리의 이점**: portal 서버 장애에도 tentacle은 독립적으로 결과 기록 → 데이터 유실 방지

## Excel Export — 같은 진입점

Excel L2에서 이미 설명했듯 ExcelExportController가 `fetchResultData`를 호출:

```pseudo
ResultData data = dataService.fetchResultData(historyId);
byte[] xlsx = excelGrpcClient.generate(data.rawJson(), data.parserId());
return ResponseEntity.ok().header("Content-Disposition", ...).body(xlsx);
```

- **JSON API와 Excel이 같은 record 공유** → 복구/스냅샷 시점이 일치
- **parser별 Generator 분기**는 Go excel-service 쪽에서 (Strategy) — TestDB는 parser ID만 넘김

## Pre-Command — TC status 배열 순회

Pre-Command L2가 `testcaseStatus`(CSV of int)를 순회해 "첫 미완료 position"을 찾고, 그 position의 tcPreCommandIds[position]를 실행.

```pseudo
int[] statuses = parseCsv(slot.testcaseStatus);
for (int i = 0; i < statuses.length; i++) {
  if (!isCompleted(statuses[i])) {
    Long cmdId = tcPreCommandIds[i];
    executePreCommand(cmdId, slot.tentacleIp, slot.usbId);
    break;
  }
}
```

- **"position 기반 배열 매칭"** = CSV 배열 두 개를 인덱스로 짝지음 — 중간 테이블 대신
- TestDB가 제공하는 건 **어떤 TC가 있는지의 이름/파서 정보**, 상태는 SlotInfomation이 들고 있음

## T32 Dump — 이름 조합으로 폴더 생성

T32 L2가 리모트 결과 폴더를 만들 때 **TR/TC 메타를 구성요소로 합성**:

```
20260419_143022_T10-4_SEQREAD_AT250220_QLC_1TB_A1B2C3
│───────────┬───│ ───┬── ──┬── ───────────────┬──────
    datetime    slotLoc tcName       TR.getFw()
```

- **getFw() 문자열의 다용도성** — 결과 폴더, Excel 시트명, 대시보드 라벨 3 곳에 쓰임
- TC.name이 파일 시스템에 안전한지(공백/특수문자) 생성 시 validation 필요 — TC 생성 API에 regex 체크

## Log Browser — logPath 디렉토리 탐색

History의 logPath가 가리키는 디렉토리를 **Log Browser UI에서 직접 탐색** 가능:

```
GET /api/log-browser/list?tentacleName=T10&path=/home/octo/tentacle/history/20260419_...
```

- History row의 **logPath를 접두어**로 사용해 Log Browser 탐색 시작점 지정
- SSH/Local 전략은 Log Browser L2가 담당 — TestDB는 경로 조합만

## TestDB가 연결되지 않는 4 L2

| L2 | 이유 |
|---|---|
| 원격 터미널 | 실시간 스크린 세션 — 결과 저장 개념 없음 |
| UFS Metadata | 자체 JSON 저장 경로 운영(SFTP) — History와 독립 |
| Bitbucket | 브랜치 상태 DB는 분리 — 테스트 실행과 무관 |
| MinIO | 파일 오브젝트 저장 — 테스트 결과용 아닌 범용 |
| Auth | 크로스-컷 — TestDB는 데이터 축, Auth는 요청 축 |

## 12 L2 전체 관점 요약

```
크로스-컷 2종:
  🔐 Auth       — 모든 요청을 가로지름 (요청 축)
  📊 TestDB     — 결과 생산·소비 L2를 엮음 (데이터 축)

기능 축 10종:
  실시간: 슬롯 · Metadata
  Job: Agent 벤치마크
  터미널: 원격
  절차: T32 · Pre-Command
  재사용 레이어: Log Browser · MinIO · Bitbucket
  데이터 통합: Excel Export
```

**크로스-컷의 패턴**: 한 축이 생기면 **기능 L2 몇 개가 자동으로 연결됨**. 향후 새 크로스-컷 후보(예: Audit 로깅·Metrics 수집)도 같은 방식으로 추가 가능.
