## 30초 요약

TestDB는 **성능·호환성 두 도메인 × (TestRequest / TestCase / History)** 3-entity 트리플로 실행 이력을 모델링하는 데이터 레이어입니다.

- **TestRequest**: FW 메타(컨트롤러·NAND·CS 버전) — *무엇을 검증하는가*
- **TestCase**: 실행 시나리오 + 파서 — *어떻게 측정하는가*
- **History**: 실행 결과 스냅샷 — *언제·어디서·결과는*

**원본 JSON은 DB에 없음** — History는 `logPath`로 원격 파일을 가리키기만 하고, 실제 측정 데이터는 tentacle/HEAD NAS에 파일로 존재. Excel export·Reparse·결과 조회가 모두 `PerformanceResultDataService.fetchResultData`를 거쳐 JSON을 끌어옵니다.

## 왜 "크로스-컷"인가

Auth가 **요청 축**의 크로스-컷이라면, TestDB는 **데이터 축**의 크로스-컷입니다. 앞 10 L2 중 절반이 TestDB를 참조:

| L2 | TestDB 의존 방식 |
|---|---|
| 슬롯 | `SlotInfomation.testcaseIds` → TC 조회, SlotCard에 현재 TC 이름 표시 |
| Agent 벤치마크 | 완료 시 PerformanceHistory 쓰기(외부 트리거) |
| Excel Export | `fetchResultData` 재사용 — JSON API와 xlsx Generator가 같은 진입점 |
| Pre-Command | TC statuses 읽어 completed position 판정 |
| T32 Dump | 결과 폴더명에 `{setLocation}_{testToolName}_{testTrName}` 조립 |
| Log Browser | History.logPath가 가리키는 원격 파일을 탐색/다운로드 |

나머지(원격·Metadata·Bitbucket·MinIO)는 TestDB와 직접 연결 없음. **"결과를 생산·소비하는 L2라면 모두 TestDB를 지나간다"**.

## 앞 11 L2와의 대조

| 축 | 대부분의 L2 | Auth (11번째) | **TestDB (12번째)** |
|---|---|---|---|
| 역할 | 기능 하나 완결 | 모든 요청을 가로지름 | **여러 L2가 데이터를 공유** |
| 쓰기 시점 | 기능별 상이 | ADFS 콜백 / 권한 변경 | **외부(tentacle) 트리거가 대부분 — portal은 소비** |
| 읽기 진입점 | 각 Controller | SecurityContext | **`fetchResultData` 하나로 JSON·Excel·Reparse 전부** |
| 상태 머신 | 각 L2가 고유 | 세션 만료만 | **RESULT: PASS/FAIL/RUNNING — RUNNING 중 부분 JSON도 허용** |
| 저장 매체 | DB 또는 파일 | DB + HttpSession | **DB(메타) + 원격 파일(측정값) 이원 저장** |

## 이원 저장 원칙

**왜 JSON을 DB에 안 넣는가**:

- 측정 데이터가 수십~수백 MB (한 TC 실행당 차트 수십 개) — DB 부하
- Excel Generator(Go)와 Reparser(SSH로 tentacle에서 파서 재실행) 둘 다 **파일 시스템 기반** — DB에 넣으면 매번 export/re-read 왕복
- 원본 log/trace는 History 삭제 후에도 tentacle에 남아있어 재현 가능

**DB에는 무엇만 남는가**:

- 메타: 누가(TR) · 무엇을(TC) · 어디서(slotLocation/setId) · 언제(startTime) · 결과(PASS/FAIL)
- 포인터: logPath (원격 파일 절대 경로를 암시하는 상대 키)
