## 30초 요약

`PerformanceResultDataService.fetchResultData(historyId)`가 TestDB 읽기 경로의 **유일한 진입점**:

1. History → TC → Parser 조회 (각 null 방어)
2. logPath + slotLocation → `tentacleName` + `remotePath` 조립 (3 브랜치)
3. `LogBrowserService.readFileContent`로 원격 JSON 로드
4. Jackson `readTree` 시도 → 실패 시 `tryRepairJson` → 그래도 실패 + RUNNING이면 `COLLECTING` 예외
5. TR에서 FW, SetInfomation에서 modelName 보충
6. `ResultData` record로 묶어 반환

**JSON API · Excel Export · 미래의 Reparse UI** 모두 이 한 메서드를 거칩니다.

## "왜 복구까지 시도하나"

원칙적으로는 완료된 JSON만 읽으면 되지만, **사용자 경험을 위해 부분 성공을 허용**:

- RUNNING 중 10초마다 UI 갱신 → 이미 쓰인 60% 차트라도 보여주는 게 blank보다 유용
- HEAD의 JSON 쓰기는 flush 단위가 크고, race로 "좀 늦게 닫힌" 상태가 흔함
- partial=true 플래그로 프론트에 "아직 완성 아님" 배지를 띄워 오해 방지

이건 **"실패는 빠르게, 부분 성공은 허용"** 의 균형 설계.
