## 왜 JOIN 대신 Map 조립인가

대시보드 쿼리를 JPA로 한 방에 할 수도 있음:

```sql
SELECT tr.fw, h.result, COUNT(h)
FROM PerformanceHistory h JOIN h.testRequest tr
GROUP BY tr.fw, h.result
```

하지만 TestRequest의 `fw`는 **@Transient** — DB에 없어 JOIN 불가. 두 가지 선택지:

1. **DB에 fw 컬럼 추가** — 모든 UPDATE 시 getFw() 재계산 훅 필요, 스키마 마이그레이션 비용
2. **메모리 Map 조립** — TR 전체(1000 row 미만) 메모리 로드 후 조회

현재는 (2). 확장 시 **materialized view**나 **캐시된 tr_fw Map**이 다음 단계.

## Helper 메서드들 — Object[] 처리

`buildTrSummary`와 `buildTcSummary`가 공통 패턴:

```java
Map<Long, long[]> stats = new LinkedHashMap<>();
for (Object[] row : rows) {
    Long id = ((Number) row[0]).longValue();
    String res = ((String) row[1]).toUpperCase();
    long count = ((Number) row[2]).longValue();
    stats.computeIfAbsent(id, k -> new long[3]);  // [pass, fail, total]
    long[] s = stats.get(id);
    s[2] += count;
    if ("PASS".equals(res)) s[0] += count;
    else if ("FAIL".equals(res)) s[1] += count;
}
```

- **long[3] 배열 in Map** — 별도 record 없이 가벼운 stats 컨테이너
- **total 먼저 += → pass/fail 분기** — 모든 결과가 total에 포함 (PASS/FAIL 외 RUNNING도 카운트)
- **rate 계산은 마지막** — `total > 0` 분기로 0-division 방어

## 두 도메인 "미러 코드"의 현실

getStats는 **95% 동일한 구조**가 두 도메인에 대해 반복됨. 추출 시도 시 문제:

- CompatibilityHistory는 `setProductName`, PerformanceHistory는 `SetInfomation` 조회 필요 → 파라미터로 `Function<H, String>` 주입? 가능하지만 **타입 파라미터가 복잡**
- 도메인 추가 시 ratio가 1/2 → 1/3 → 1/4로 떨어짐 — 추출 가치가 감소
- **현재 상태에서 "수정 시 두 곳" 비용 < 추출 후 "타입 게임" 비용** 판단

복제가 반드시 나쁜 게 아니라는 예시.
