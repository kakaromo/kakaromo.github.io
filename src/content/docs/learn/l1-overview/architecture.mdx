## 30초 요약

MOVE는 **Spring Boot 백엔드**가 중앙 허브입니다. 브라우저(SvelteKit)는 이 허브에 REST·SSE·WebSocket으로 붙고, 허브는 각기 다른 외부 시스템(Head·Tentacle·**Go Agent**·**Go Excel**·**Rust Trace**)을 **프로토콜별로 적절히** 부릅니다.

데이터는 MySQL(3개 데이터소스) · Redis(캐시) · MinIO(파일 + Trace parquet range-GET)에 나누어 저장합니다.

## 레이어 읽는 법

- <span class="learn-layer-badge" data-layer="fe">Frontend</span> — 브라우저에서 돌아가는 UI (SvelteKit, Svelte 5 runes)
- <span class="learn-layer-badge" data-layer="be">Backend</span> — Spring Boot 서버 본체
- <span class="learn-layer-badge" data-layer="hw">Hardware</span> — 실제 장비 또는 장비를 대리하는 서버
- <span class="learn-layer-badge" data-layer="state">State</span> — 데이터를 저장/캐시하는 계층
- <span class="learn-layer-badge" data-layer="tx">Transport</span> — 외부 시스템을 부르는 경로·gRPC 서비스

L2/L3에서도 동일 팔레트를 씁니다. 색만 봐도 "이게 어느 계층 이야기인지"가 감이 잡히도록 의도했습니다.

## 왜 Spring Boot가 중앙 허브인가

허브-스포크 구조는 다음 이유로 택했습니다.

- **인증·권한 단일점**: ADFS SSO, 세션 타임아웃, URL 단위 액션 권한을 한 곳에서 검사.
- **프로토콜 변환**: 각 외부 시스템은 서로 다른 프로토콜(TCP/SSH/gRPC)을 씁니다. 브라우저가 이걸 직접 상대하면 CORS·인증·재연결 로직을 전부 프론트에 복제해야 합니다.
- **캐시·일관성**: Redis 캐시와 DB 트랜잭션을 한 프로세스에서 관리.
- **장애 격리**: 외부 시스템 중 하나가 죽어도 다른 기능은 계속 동작.

단점은 허브가 병목이 될 수 있다는 것입니다. MOVE는 실시간 스트림(슬롯 SSE, Agent 모니터링 SSE)에 별도 스레드·connection 관리를 두어 이를 완화합니다 — L2에서 상세히 봅니다.
