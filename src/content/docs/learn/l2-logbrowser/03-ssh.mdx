## 30초 요약

`SshLogBrowserService`는 JSch 라이브러리로 원격 VM에 붙습니다.

- **파일 탐색**: `ChannelSftp` (SFTP 세션) — ls / get / put
- **검색**: `ChannelExec` — 원격에서 `rg -n --no-heading` 실행
- **세션 캐싱**: `ConcurrentHashMap` + 5분 TTL + 스케줄러로 정리 — 매 요청마다 새 세션 맺는 비용 회피

세션은 **요청 type별로 생명주기가 다릅니다**. ls·download 같은 단발 작업은 매번 새로 만들고, 검색(exec)만 캐시된 세션을 재사용. 이유는 아래.

## 왜 listFiles는 캐시 안 쓰고 searchInFile만 쓰는가

디렉토리 리스팅은 **요청 간 시간차가 큼** (사용자가 폴더를 탐색하는 속도). 캐시가 있어도 대부분 TTL 만료 후 재연결이 됨. 캐시 관리 복잡도만 늘어남.

검색은 반대: **같은 파일을 키워드 바꿔가며 여러 번** 하는 패턴이 흔함. 세션 재사용의 이득이 큼.

**"자주 오는 호출만 캐시"**. 모든 호출에 캐시를 다는 것은 나쁜 설계.

## SftpInputStream 트릭

`downloadFile`은 `InputStream`을 반환합니다. 하지만 스트림이 끝나기 전에 **세션·채널을 닫아버리면 스트림도 끊김**.

```java
return new SftpInputStream(channel, session, channel.get(remotePath));
```

SftpInputStream은 **세션·채널·내부 InputStream 세 가지를 모두 소유**하고, `close()`에서 다 함께 닫습니다. 컨트롤러가 `InputStreamResource`를 응답하면 Spring이 전송 후 close → 자원 정리 자동.

이 래퍼가 없었다면 컨트롤러가 직접 채널 lifecycle을 관리해야 했음.
