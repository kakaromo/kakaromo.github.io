## 30초 요약

`LogBrowserService` 인터페이스는 **8 개의 메서드**와 **3 개의 record**로 이뤄져 있습니다.

- **탐색**: `listFiles` → `List<FileEntry>`
- **읽기**: `downloadFile` (InputStream) / `readFileContent` (전체 문자열) / `readLines` (범위) / `readLastLines` (tail)
- **검색**: `searchInFile` (ripgrep)
- **판정**: `isBinaryFile`
- **쓰기**: `uploadFile` / `deleteFile`
- **추가**: `getActiveSessionCount` default

인터페이스에서 record로 반환 타입까지 함께 정의 — **API를 한 파일에서 완결**.

## 왜 인터페이스 안에 record를 두었는가

세 record는 이 서비스 전용 DTO. 별도 파일로 빼면 파일 수가 늘고 스캔 복잡도만 증가. **중첩으로 두면 "이 타입이 어디 속하는가"가 이름으로 드러남**.

```java
// 호출 쪽
import com.samsung.move.logbrowser.service.LogBrowserService;
import com.samsung.move.logbrowser.service.LogBrowserService.FileEntry;

List<FileEntry> files = service.listFiles("T1", "/var/log");
```

`FileEntry`만 임포트해도 의미가 명확. 인터페이스명 + 타입명 조합이 **문서 역할**을 함.

## 왜 메서드가 8개나 되는가 (합칠 수 있지 않나?)

예를 들어 `readLines(start, limit)`와 `readLastLines(n)`을 하나로 합칠 수도 있지만:

- **의미가 다름** — 앞에서부터 vs 뒤에서부터
- **구현이 다름** — `readLines`는 `sed`/`head`/`tail`, `readLastLines`는 `tail -n`
- **사용처가 다름** — 뷰어 페이지네이션 vs tail 뷰

**도메인의 동사를 그대로 드러내는 API**가 오히려 유지보수가 쉽습니다. 억지로 합치면 파라미터가 튀어나오고 분기가 내부에 숨음.

## 인터페이스가 하는 일 한 문장

**"파일 시스템을 도메인 스타일로 요청하면 된다"**. 호출자(컨트롤러·다른 모듈)는 SSH인지 Local인지 몰라도 됨. 이 분리 덕분에 T32 L2가 SSH 환경에서도 Local 환경에서도 똑같이 `/api/log-browser/download-dir`을 호출할 수 있음.
