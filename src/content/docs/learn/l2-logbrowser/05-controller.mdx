## 30초 요약

`LogBrowserController`는 **6 개의 엔드포인트**를 제공합니다.

| Method | Path | 용도 |
|---|---|---|
| GET | `/prefix` | log-prefix, headPrefix, headHost 반환 (UI 초기화) |
| GET | `/files` | 디렉토리 리스팅 |
| GET | `/download` | 파일 다운로드 (스트리밍) |
| GET | `/download-dir` | **폴더 ZIP 다운로드** |
| GET | `/view/lines` | 라인 범위 읽기 (뷰어) |
| GET | `/view/search` | ripgrep 검색 |

이 페이지는 그중 **가장 특이한 `/download-dir`** 한 가지에 집중합니다. 같은 엔드포인트가 SSH·Local 모드에서 **전혀 다른 전략**으로 ZIP을 만듭니다.

## 왜 Strategy가 깨지는가 — 의도된 누수

인터페이스에 `downloadDir`을 추가하면 양쪽 구현체가 각자의 로직을 가지게 됩니다. 하지만:

- **메서드 증가** — 인터페이스가 9개로. 쓰는 곳 하나를 위해 전체 API 확장.
- **Local은 ZIP 바이너리를 어떻게 반환?** — `byte[]` vs `InputStream`의 일관성 문제.
- **SSH의 side-effect 명령 트릭을 정당화하기 어려움** — `readFileContent`가 zip 생성에 쓰이면 안 되는 오남용.

**"인터페이스가 깨끗한 대신 한 컨트롤러에서 분기"** 를 선택. Strategy 패턴이 절대적 규칙은 아님. 한 모듈 안에서 **드물게 특수한 동작**은 추상화를 억지로 넓히지 말고 그 자리에서 처리.

## SSH 쪽 readFileContent 오남용은 왜 허용되는가

본래 `readFileContent(tentacleName, path)`는 **path의 파일 내용**을 반환해야 합니다. SSH 구현은 path를 SSH 명령 문자열로 해석해 실행하는 것이 **의도되지 않은 방식**입니다. 리팩토링이 필요한 신호:

```java
// 더 올바른 API 설계:
interface LogBrowserService {
    String execRawCommand(String tentacleName, String command);  // 명시적으로
}
```

현재는 **작동하지만 이름이 거짓말**. 이런 부채는 비교 문서 체크리스트의 "왜 X가 아닌가" 문단 후보입니다 — 다음 리팩토링 때 다듬을 가치.

## 다른 L2에서 이 엔드포인트 소비 지점

[T32 Dump L2](/learn/l2-t32/05-ui/)의 Stepper UI에서 완료 후 **"전체 ZIP 다운로드"** 버튼:

```ts
window.open(`/api/log-browser/download-dir?tentacleName=${t}&path=${encodeURIComponent(p)}`);
```

T32는 자기 로직 없이 **이 모듈의 엔드포인트를 그대로 사용**. Log Browser가 "재사용 레이어"인 좋은 예.

## 일곱 L2의 역할 유형 최종 요약

| L2 | 주 사용자 | 역할 |
|---|---|---|
| 슬롯 | 사용자 직접 | 실시간 대시보드 |
| 벤치마크 | 사용자 직접 | job 실행·중계 |
| 원격 | 사용자 직접 | 터미널 세션 |
| Metadata | 사용자 직접 | 시계열 수집·시각화 |
| T32 Dump | 사용자 직접 | 절차 실행 |
| Pre-Command | 자동 (도메인 이벤트) | 사전 명령 실행 |
| **Log Browser** | **다른 L2 + 사용자** | **재사용 파일 레이어** |

**일곱 번째 L2에서 처음 등장한 패턴**: 주된 소비자가 다른 모듈. 이런 "하부 레이어" L2는 인터페이스 설계가 문서화의 중심.
