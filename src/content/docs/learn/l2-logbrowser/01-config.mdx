## 30초 요약

`application.yaml`의 **`tentacle.access-mode`** 값 하나가 전체 동작을 바꿉니다. Spring Boot의 **`@ConditionalOnProperty`** 가 조건에 맞는 Bean만 등록 — 컨트롤러는 `LogBrowserService` 인터페이스에 의존하므로 **구현체 교체가 투명**.

- `ssh` (또는 미지정): `SshLogBrowserService` Bean 생성
- `local`: `LocalLogBrowserService` Bean 생성

yaml 핵심:

```yaml
tentacle:
  access-mode: ssh           # or 'local'
  log-prefix: /home/octo/tentacle
  ssh:
    username: samsung
    password: tentacle
    port: 22
  mount-path: /mnt
  head:
    log-path: /home/octo/nas
    mount-path: /mnt/head/nas
```

## 왜 조건부 Bean인가 (switch문 대신)

컨트롤러 한 개에 `if (mode == ssh) sshService else localService` 형태로 들어가는 방법도 가능합니다. MOVE는 **조건부 Bean**을 택했습니다.

- **컴파일 시점에 선택** — 한 가지 구현만 컨테이너에 존재. 잘못된 구현이 런타임에 끼어들 여지 없음.
- **테스트 단순화** — 단위 테스트에서 `mode=ssh`로 띄우면 Local 구현은 로드 자체가 안 됨.
- **리소스 효율** — 반대편 구현의 의존성(예: SSH 모드일 때 NFS 마운트 경로)이 로드되지 않음.

`@ConditionalOnProperty`는 Spring Boot의 auto-configuration 스타일. **여러 모듈 중 하나를 선택**할 때 표준 패턴.

## 설정 레이어 — Head 전용 경로가 별도인 이유

yaml을 자세히 보면 `tentacle.head.*`가 분리되어 있습니다.

```yaml
tentacle:
  log-prefix: /home/octo/tentacle   # 일반 VM 로그
  head:
    log-path: /home/octo/nas        # HEAD NAS (SSH 모드)
    mount-path: /mnt/head/nas       # HEAD NAS (Local 모드)
```

**HEAD 장비는 일반 Tentacle과 다른 NAS를 가리킴**. 같은 서비스가 두 종류의 장비를 처리하므로 경로 매핑이 둘이어야 함. Local 구현의 `resolveLocalPath`가 `tentacleName == "HEAD"` 특수 분기를 가짐 (단계 4).
