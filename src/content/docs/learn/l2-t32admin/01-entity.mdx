## 30초 요약

T32 Admin 의 데이터 모델은 두 테이블입니다.

- `portal_t32_configs` — **19 컬럼**의 설정 마스터 (그룹별 1행)
- `portal_t32_config_servers` — 위 설정과 슬롯 서버의 **N:M 매핑**

JPA 어노테이션은 평범하지만 **컬럼 19개가 왜 필요한가** 가 핵심.

## 19 컬럼이 정당한 이유

테이블 정규화 관점에서 컬럼 19개는 보통 분리 신호이지만, T32Config 는 **분리 안 하는 게 정답**입니다.

| 분리 후보 | 왜 안 했나 |
|---|---|
| 4 명령 → `t32_commands` 테이블 | 명령 4개가 **순서·세트로 의미가 있음**. 따로 떼면 N+1 쿼리 + 일관성 깨짐 |
| 2 계정 쌍 → `t32_credentials` 테이블 | 마찬가지. 계정 1개만 있으면 안 됨. 단위로 같이 관리 |
| 4 경로 baseline → `t32_path_mapping` | 경로는 환경 의존이지만 **설정과 1:1**. 분리하면 join 1회 추가만 발생 |
| description · enabled · 타임스탬프 → 운영 메타 | 운영 메타는 모든 설정의 공통 필드 — 분리할 이유 없음 |

**핵심**: "같이 쓰이고 같이 변경되는 것"은 한 테이블. 19 컬럼이라도 모두 같은 entity 의 속성이라면 정상.

## 직접 해보기

- `portal_t32_configs` 에 새 행을 INSERT 해보고, 같은 `(serverGroupId, jtagServerId, t32PcId)` 조합으로 두 번째 행을 INSERT 시도 — DB 가 막는지 확인 (현재는 unique 제약 없음 — Add 가능 여부 판단)
- `portal_t32_config_servers` 에 같은 `(t32ConfigId, serverId)` 두 번 INSERT — unique constraint 가 차단해야 함
- `enabled=false` 로 변경 후 T32 Dump 시도 — 실행 막히는지 확인
