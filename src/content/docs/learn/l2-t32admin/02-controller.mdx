## 30초 요약

CRUD 4 엔드포인트는 평범하지만 두 패턴이 다른 L2 와 다릅니다.

1. **비밀번호 마스킹** — 프론트가 빈 문자열을 보내면 기존 값 유지 (수정 시 매번 비밀번호를 다시 입력하지 않게)
2. **N:M 매핑 동기화** — 매번 `deleteByT32ConfigId` → `saveAll` (diff 계산 안 함)

두 패턴 모두 운영 UI 에서 자주 보이는 트레이드오프 — **단순함을 위한 의도적 비효율**.

## 왜 비밀번호 마스킹을 백엔드에서 처리하나

대안 두 가지:

| 방법 | 단점 |
|---|---|
| 프론트가 GET 응답에서 비밀번호 받기 + 수정 시 그대로 다시 보내기 | 비밀번호가 응답으로 노출 — 브라우저 캐시·로그·DevTools 에 남음 |
| 별도 "비밀번호만 수정" 엔드포인트 | UX 가 나빠짐 (비밀번호 외 필드 수정해도 비밀번호 입력해야 한다는 인상) |
| **현재: 빈 문자열은 무시** | 프론트는 비밀번호 모르는 채로 다른 필드 수정 가능 + 새 비밀번호 입력 시 변경 |

3번이 **운영 UI 에서 가장 흔한 패턴**. Auth · Server Admin · 외부 API 키 관리 등에서도 같은 패턴.

## 직접 해보기

- 설정 1개 만든 후 `description` 만 수정 — 비밀번호가 그대로 유지되는지 확인
- 비밀번호 칸에 새 값 입력 후 저장 — 변경 됐는지 SSH 시도로 확인
- 매핑 5개 저장 후 매핑 3개로 줄여서 저장 — 줄어든 2개가 DB 에서 삭제됐는지 확인
