## 30초 요약

499줄짜리 `AdminT32Tab.svelte` 의 핵심은 **상태 14개 + 1개 derived**. 데이터 로드는 3종(configs · servers · groups)을 `Promise.all` 로 병렬, 폼은 create/edit 양쪽 모드를 한 dialog 로 처리, 매핑 후보는 `serverGroupId` 가 바뀔 때마다 `$derived` 로 자동 필터링.

이 패턴은 **Admin 18 탭 모두 거의 동일**. T32Tab 을 읽으면 다른 17 탭도 빠르게 읽힙니다.

## 18 탭 공통 패턴 일반화

T32Tab 의 패턴이 다른 탭에서 어떻게 반복되는지:

| 탭 | 메인 엔티티 | 외래키 후보 | derived 필터링 | dialog 모드 |
|---|---|---|---|---|
| **T32** | t32_configs | servers · groups | groupServers (그룹별) | create/edit 통합 |
| Slots | head_connections | servers | (없음) | create/edit 통합 |
| Sets | sets | products | (없음) | create/edit 통합 |
| Ufs Info | ufs_info | products | (없음) | create/edit 통합 |
| Metadata | metadata_commands | products | productCommands (제품별) | create/edit 통합 |
| Debug | debug_types | (없음) | (없음) | create/edit 통합 |

T32 와 Metadata 만 `$derived` 필터링이 있는 이유: **다대다 또는 다대일 관계가 명시적**이기 때문. Slots/Sets/UfsInfo 는 외래키 후보가 단순 (모든 서버/제품에서 고를 수 있음).

## 왜 dialog 1개로 create/edit 둘 다

대안 (별도 dialog):

```
+ AddT32ConfigDialog.svelte
+ EditT32ConfigDialog.svelte
```

90% 같은 폼이라 **같은 코드를 두 번** 유지해야 함. 폼 필드 추가 시 두 곳 동시 수정 — 정합성 깨질 위험.

현재 (mode 로 구분):

```
let dialogMode = $state<'create' | 'edit'>('create');
// 저장 시:
if (dialogMode === 'create') createConfig(form);
else updateConfig(editingId, form);
```

차이는 **저장 분기 1줄 + 초기값 분기**. 폼 자체는 1번만 작성.

## 직접 해보기

- 그룹 변경 → 매핑 후보가 자동 줄어드는지 확인
- 비밀번호 빈 채로 저장 → 다른 필드만 수정되는지 확인
- 저장 후 dialog 닫고 다시 edit → 비밀번호 칸이 빈 채로 보이는지 확인 (보안)

## 18 탭 = 18 회 같은 패턴

**관리 UI 축의 핵심 통찰**: 18 탭이 거의 같은 패턴 → 추상화 가능? 그러나 [Admin L2](/learn/l2-admin/) 에서 봤듯 **God Controller + 단일 페이지 + 탭 컴포넌트**가 정답. 탭마다 미묘한 차이(필드 수·필터링·dialog 흐름)가 있어 일반화 시 파라미터 폭증.

**현재 정답**: 패턴은 **머리에 두고**, 코드는 **복사 후 수정**. 18 탭 × 90% 동일 코드 → 일반화 비용 > 중복 비용.
