## 30초 요약

**`ActionPermissionInterceptor`가 모든 `/api/**` 요청을 전처리**합니다.

1. `portal.auth.disabled=true` → **바로 통과** (로컬 개발)
2. URL + HTTP 메서드를 DB의 `ActionPermission` 룰과 **AntPathMatcher**로 매칭
3. 매칭 없으면 통과 (권한이 걸린 엔드포인트만 체크)
4. 세션에서 PortalUser 꺼냄 — 없으면 **401 대신 403 "로그인 필요"**
5. ADMIN이면 통과
6. `UserPermissionService.hasPermission(userId, key)` 체크 → 실패 시 **403 JSON 응답**

**컨트롤러 수정 없이** URL 단위로 권한을 걸 수 있음이 핵심. Admin UI에서 DB만 수정하면 즉시 반영.

## ActionPermission 엔티티 (간략)

```java
@Entity
@Table(name = "portal_action_permissions")
public class ActionPermission {
    @Id Long id;
    String urlPattern;       // "/api/agent/benchmark/**"
    String httpMethod;       // "POST" or "*"
    String permissionKey;    // "action:agent:benchmark"
    boolean enabled;
}
```

**3 속성**이 룰 한 개: URL 패턴 + HTTP method + 필요한 권한 키. Admin UI에서 추가·편집·활성화.

예시 룰:

| urlPattern | method | permissionKey |
|---|---|---|
| `/api/performance-results/*/excel` | GET | `action:performance:excel_export` |
| `/api/agent/benchmark/run` | POST | `action:agent:benchmark` |
| `/api/remote/session/**` | * | `action:remote:connect` |
| `/api/minio/**` | * | `menu:storage` |

이 테이블 한 줄 추가가 **그 엔드포인트에 권한 게이트 설치**와 같은 효과.

## 왜 컨트롤러 수정이 필요 없는가

이전에 흔한 패턴은 컨트롤러 메서드에 어노테이션:

```java
@PreAuthorize("hasPermission('action:agent:benchmark')")
@PostMapping("/run")
public ... { ... }
```

Spring Security + SpEL. 장점: 컴파일 시점 권한이 코드에 명시. 단점:

- 권한 추가·제거에 배포 필요
- 같은 권한을 여러 엔드포인트에 중복 어노테이션
- Admin UI에서 동적 변경 불가

MOVE는 **런타임 DB 매핑**을 택함:

- Admin이 UI에서 즉시 권한 부착/해제
- 컨트롤러는 권한을 전혀 모름 — 순수 기능만
- 인터셉터 하나가 일관된 정책

**trade-off**: 코드에 권한이 명시 안 되어 IDE에서 "이 엔드포인트 권한 뭐야?" 추적이 어려움. Admin UI에 집중된 문서화가 필요.
