## 30초 요약

ADFS가 form_post로 id_token을 보내면 `POST /api/auth/adfs/callback`이 받아:

1. 에러 체크 → 에러 있으면 `/?adfs_error=...`로 redirect
2. **JWT payload 파싱** (서명 검증 생략)
3. claims에서 `userid · loginid · username · mail` 추출
4. `PortalUser` 조회 또는 자동 생성
5. 신규 사용자면 **17 권한 전체 부여**
6. `HttpSession`에 PortalUser 저장 + 120분 timeout 적용
7. **HTML + JS replace**로 SPA로 이동

redirect가 아니라 **HTML 응답**인 이유: 세션 쿠키가 확실히 설정되도록.

## HttpSession vs JWT — 왜 세션을 쓰는가

MOVE는 **세션 쿠키** 방식. JWT Access Token을 헤더로 넣는 방식도 가능:

| 축 | 세션 쿠키 | JWT Bearer |
|---|---|---|
| 상태 | 서버에 저장 | 클라이언트가 들고 있음 |
| 취소 | **세션 파기로 즉시 로그아웃** | TTL 만료까지 유효 |
| 확장 | Redis 공유 필요 | stateless, 확장 쉬움 |
| CSRF | 취약 (쿠키 자동 첨부) | 안전 (헤더 조작 불가) |
| 모바일 / 서드파티 | 쿠키 관리 필요 | 헤더 주입 쉬움 |

MOVE는 **단일 포털 + 내부망** 전제라 세션이 맞음:

- 확장성 요구 없음 (현재 in-memory로 충분)
- **즉시 로그아웃**이 중요 (민감 계정 회수)
- 브라우저 단일 사용처

## 120분 타임아웃 + 카운트다운

```java
private void applySessionTimeout(HttpSession session) {
    int minutes = sessionConfigService.getTimeoutMinutes(); // 기본 120
    session.setMaxInactiveInterval(minutes * 60);
}
```

프론트는 `/api/auth/session-info`를 호출해 남은 시간을 받고 **1초 타이머**로 카운트다운. 5분 남으면 연장 버튼 표시. **백엔드의 진실 vs 프론트의 동기화**를 주기적 재조회 없이 유지하는 단순한 패턴.
