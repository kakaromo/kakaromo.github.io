## 왜 상수로 정의하는가 — "권한 카탈로그"

권한 키를 DB 테이블로 관리할 수도 있습니다:

```sql
-- 대안 A
CREATE TABLE permission_definitions (
  key VARCHAR PRIMARY KEY,
  description VARCHAR
);
```

MOVE는 이 대안 대신 **static 상수**를 택함:

- **권한 추가에 배포 필요** — 의도적. 권한 정의는 코드 관심사이지 운영 관심사 아님
- **IDE 자동 완성** — 컨트롤러에서 `"action:agent:benchmark"` 타이핑 시 검색 가능
- **테이블 하나 줄임** — `user_permissions`만 있으면 됨
- **오타 방지** — IDE가 상수 참조 검증

**동적 권한 필요 없음**이 전제. 그룹/역할 기반(RBAC)이 아닌 **flat 17 권한 + 사용자별 on/off**라 단순.

## 17 권한이 앞 10 L2에 어떻게 매핑되는가

| L2 | 메뉴 권한 | 액션 권한 |
|---|---|---|
| 슬롯 | `menu:slots` | `action:slots:initslot`, `action:slots:pre_command` |
| 벤치마크 (Agent) | `menu:agent` | `action:agent:benchmark`, `action:agent:scenario`, `action:agent:trace` |
| 원격 | `menu:remote` | `action:remote:connect` |
| Metadata | (Slots 내부) | — |
| T32 Dump | (Debug, 별도 권한 없음) | — |
| Pre-Command | (Slots 내부) | `action:slots:pre_command` |
| Log Browser | (라우트 없음) | — |
| Excel Export | `menu:performance` | `action:performance:excel_export` |
| Bitbucket | (Admin 영역) | — |
| MinIO | `menu:storage` | `action:storage:run` |

**"각 L2의 관문 최소 1개"** 커버. 모든 동작이 권한으로 막혀 있지는 않음 — **무거운 영향만 권한**이라는 기준.

## 권한 분리의 세 단계

```
1. 메뉴 권한이 없으면 사이드바에 메뉴 자체가 없음 (프론트 필터)
   ↓
2. URL로 직접 접근해도 백엔드 인터셉터가 403
   ↓
3. 페이지 안의 민감 버튼은 액션 권한으로 추가 차단
```

**프론트 필터 + 백엔드 인터셉터 + 액션 세분화**의 3단 방어. 프론트만 믿으면 URL 조작으로 뚫림, 백엔드만 믿으면 UI가 무질서 — 둘 다 필요.
