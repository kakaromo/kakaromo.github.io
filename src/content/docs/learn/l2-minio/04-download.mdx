## 30초 요약

두 엔드포인트가 **대비를 이룹니다**.

- **`/download`** — 단일 파일. `InputStreamResource` 래퍼로 **Spring이 응답 후 close**. UTF-8 filename 헤더.
- **`/download-folder`** — prefix 하위 모든 파일을 **ZipOutputStream으로 한 번에** 브라우저에 스트리밍. 메모리에 ZIP을 올리지 않음.

Log Browser L2의 `/download-dir`과 **같은 ZIP 스트리밍 전략**이지만 **MinIO 쪽은 각 object마다 SDK 호출**이 추가됨.

## Log Browser의 `/download-dir`과의 비교

두 모듈 다 "폴더를 ZIP으로" 라는 같은 UX이지만 구현 방식이 다름:

| 축 | Log Browser `/download-dir` | **MinIO `/download-folder`** |
|---|---|---|
| 파일 목록 조회 | `listFiles` (SSH ls) | `listObjectsRecursive` (S3 API) |
| 각 파일 읽기 | SSH SFTP | S3 GetObject (HTTP) |
| ZIP 생성 | **Local 모드만** ZipOutputStream. SSH는 **원격 zip + SFTP** 편법 | **항상** ZipOutputStream (한 방식) |
| 스트리밍 | Local은 메모리 버퍼, SSH는 편법 | HttpServletResponse.outputStream 직접 |

**MinIO 쪽이 더 깔끔**. 한 전략으로 통일 가능한 건 S3 API 덕분 — 두 전략이 모두 같은 HTTP 기반.

## 폴더 ZIP의 성능 특성

N개 파일이 있는 폴더를 ZIP으로 다운로드:

- **S3 GetObject N번** — 각 파일마다 HTTP round-trip
- **순차 처리** — 병렬 읽기 없음 (의도적으로 단순)
- **응답 스트리밍** — 브라우저에 즉시 전달 시작

N이 매우 크면(수백 파일) 병목이 될 수 있지만 실 사용 케이스(벤치마크 1회 결과)는 보통 수십 파일로 문제 없음.

**개선 가능성**: `CompletableFuture`로 병렬 GetObject 후 순서대로 ZIP에 쓰면 빨라짐. 복잡도 증가 vs 이득을 보고 결정.
