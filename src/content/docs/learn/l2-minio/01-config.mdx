## 30초 요약

- **`MinioProperties`** — yaml `minio.*` 바인딩 (endpoint · port · useSsl · accessKey · secretKey)

## 왜 이렇게 짧은가

**SDK의 고수준 추상화 덕분**. `io.minio:minio 8.5.14`가:

- HTTP 클라이언트
- 자격 증명 서명
- XML 응답 파싱
- 오류 처리

를 모두 내부에 숨겨서 포털은 **"버킷/오브젝트 이름 + 스트림"** 수준에서만 상호작용하면 됩니다.

비교: Bitbucket L2의 `BitbucketApiClient`는 **직접 HttpClient로** HTTP 호출·JSON 파싱·SSL 설정까지 작성해야 했음. MinIO는 "이미 만들어진 SDK"가 있어 Client 설정이 이렇게 짧음.

## prod/dev 환경 분리 전략

현재 MOVE의 구조:

| 환경 | MinIO 위치 | 접속 |
|---|---|---|
| dev | 단일 VM (192.168.x.x:9000) | 공유 dev 자원 |
| prod | 전용 인스턴스 | 고정 IP |

yaml 프로파일 기반(`application-dev.yaml` / `application-prod.yaml`)으로 분기. **설정 키는 동일**, 값만 다름 — 코드 경로는 환경 무관.

## Bean 1개 · 서비스 1개 구조

```
MinioConfig (@Configuration)
    │ @Bean
    ▼
MinioClient (싱글턴)
    │ @Autowired (via @RequiredArgsConstructor)
    ▼
MinioStorageService (@Service)
    │ 호출
    ▼
MinioController / MinioUploadController
```

**단계마다 딱 하나**의 주체. Spring DI가 자동으로 연결하므로 우리가 작성하는 건 필드 선언 + 생성자뿐.
