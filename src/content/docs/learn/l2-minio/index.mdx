## 이 모듈에서 배울 것

"Agent가 뽑은 **벤치마크 결과·I/O 트레이스**를 어디에 저장하고, 관리자가 브라우저에서 어떻게 탐색·다운로드하는가? **MinIO**는 어떻게 포털과 연결되고, Log Browser와 **어떤 축에서 다른가**?"

열 번째 L2. [Log Browser L2](/learn/l2-logbrowser/)와 **짝을 이루는 두 번째 재사용 파일 레이어**입니다.

- **Log Browser** — SSH/Local로 **Tentacle 서버의 로컬 파일 시스템** 탐색
- **MinIO** — S3 호환 **오브젝트 스토리지** (버킷 + prefix + flat key-value)

**왜 두 레이어 모두 필요한가**: 성격이 다른 데이터는 저장 위치도 달라야 합니다. 자세한 대비는 아래.

## Log Browser vs MinIO — 두 레이어의 대조

| 축 | Log Browser (L2 #7) | **MinIO (L2 #10)** |
|---|---|---|
| **저장 모델** | 파일 시스템 (경로 계층) | **버킷 + prefix (flat key-value)** |
| **접근 방식** | SSH/Local (두 전략) | **S3 호환 SDK 단일 전략** |
| **메타데이터** | inode (크기/수정일/권한) | **object 속성 + 커스텀 태그** |
| **스토리지 위치** | Tentacle 서버 각자 | **중앙 집중 (MinIO 서버 9000 포트)** |
| **주 용도** | **실시간 테스트 로그** (testdb 결과 파일) | **결과 아카이브** (벤치마크 · trace · 업로드) |
| **주 소비자** | History 결과 페이지, slots context menu | **Agent gRPC (UploadTrace) + Admin UI + 포털 사용자** |
| **용량 관리** | Tentacle 디스크 | **버킷별 정책 가능** (현재는 단순) |
| **쓰기 주체** | 테스트 실행 (tentacle 측) | **여러 주체** (Agent, 업로드 API, 수동) |

**비유**: Log Browser가 "책상 서랍" 같은 **장소별 분산 스토리지**라면, MinIO는 "도서관 아카이브" 같은 **중앙 색인된 영구 보관소**.

## 왜 MinIO인가 (단순 NFS 대신)

- **S3 API 호환** — 외부 도구(mc, aws cli, Agent Go SDK)가 표준 API로 붙을 수 있음
- **중앙 집중** — Tentacle이 수십 대여도 결과를 한 곳에 모음
- **접근 제어** — 버킷·prefix 단위 정책
- **scale-out 가능** — 분산 모드로 확장하기 쉬움
- **자체 호스팅** — AWS S3 아닌 사내 인프라 (데이터 주권)

단점: **운영 복잡도** (NFS보다 관리 요소 많음). 규모가 커지면서 얻는 이득이 이를 상쇄.

## 학습 순서와 예상 시간

| 단계 | 제목 | 성격 | 시간 |
|---|---|---|---|
| 1 | 설정 + 클라이언트 Bean | 코드 투어 | 6분 |
| 2 | 스토리지 서비스 (CRUD 래퍼) | 코드 투어 | 12분 |
| 3 | 업로드 (Multipart + 2GB 검증) | 코드 투어 | 10분 |
| 4 | 다운로드 + 폴더 ZIP | 코드 투어 | 12분 |
| 5 | 가시성·소비 모듈 (Admin + Agent) | 코드 투어 + 의사코드 | 10분 |

총 약 **50분**. 단계 1이 가장 짧음 (17줄 × 2 파일), 단계 2가 가장 두꺼움 (CRUD 전부).

## 새로 등장할 L3 개념

**없음** (8번째 연속). `@Configuration`·`@ConfigurationProperties`·`@RestController`·`MultipartFile`이 전부 기존 사전.
