## 30초 요약

`MinioUploadController.upload`는 **53줄짜리 단일 엔드포인트**.

1. `@RequestParam MultipartFile` — Spring이 HTTP multipart 파싱
2. **2GB 이상이면 압축 확장자 체크** — 아니면 400
3. prefix + 파일명으로 object key 조립
4. `uploadObject` 호출 → 완료 응답

Spring Multipart 기본 한도(`spring.servlet.multipart.max-file-size`)가 매우 커야 2GB 업로드 가능 — **application.yaml에 1TB로 설정되어 있음** (CLAUDE.md 참고).

## 왜 MultipartFile인가 (raw InputStream 대신)

Spring에는 두 업로드 패턴이 있습니다.

| 축 | `MultipartFile` | `InputStream` body |
|---|---|---|
| 요청 형식 | multipart/form-data | raw (application/octet-stream) |
| 메타 정보 | filename · contentType 자동 | 헤더로 직접 |
| 임시 저장 | Spring이 처리 (메모리/디스크) | 스트림 직접 |
| 브라우저 `<input type="file">` | 자연스러움 | 별도 JS 필요 |

**사용자 업로드 UI**가 주 경로이므로 MultipartFile 선택이 맞음. **Agent gRPC UploadTrace**는 별도 경로(proto 기반, 단계 5)라 여기 통과 안 함.

## 2GB 검증의 숨은 효과

이 한 검증이 하는 일:

1. **비용 관리** — 큰 텍스트 파일(trace, core dump)을 그대로 보관하면 스토리지 폭발
2. **전송 효율** — 압축 상태로 넘겨 네트워크도 절약
3. **관례 강제** — "로그는 압축해서 업로드"라는 팀 규칙을 기술적으로 자동 적용

예외적인 경우(예: 미디어 파일)라면 이 검증이 방해 — **임계값·확장자를 yaml로 외부화**하면 유연성 향상. 현재 하드코딩 상수는 기술 부채로 표시.

## prefix의 "트레일링 슬래시" 주의

```java
String objectName = prefix.isEmpty()
    ? file.getOriginalFilename()
    : prefix + file.getOriginalFilename();
```

호출자가 `prefix=traces/`를 보내야 `traces/fio-001.zip`이 됨. `prefix=traces`로 보내면 `tracesfio-001.zip` — **Foot-gun**.

방어 로직을 추가한다면:

```java
String safePrefix = prefix.isEmpty() ? ""
    : (prefix.endsWith("/") ? prefix : prefix + "/");
```

**현재는 프론트 UI가 항상 /를 붙이도록** 신뢰. API 직접 호출자가 실수하면 디버깅이 귀찮아짐.

## Spring Multipart 한도 vs MinIO 한도

Spring Boot가 업로드를 받으려면 **application 설정**이 허용해야 함:

```yaml title="application.yaml"
spring:
  servlet:
    multipart:
      max-file-size: 1TB
      max-request-size: 1TB
```

(CLAUDE.md에 기록된 MOVE 설정). 이 값이 없으면 Spring 기본 1MB에서 413 에러. MinIO 쪽은 기본적으로 **거의 무제한**이므로 Spring 한도가 실질 bottleneck.
