## 이 모듈에서 배울 것

"사용자가 **T32 Dump** 버튼을 누르면, **JTAG 서버**(SSH)와 **T32 PC**(gRPC)에 차례로 명령을 보내 **5단계 Stepper**가 돌아가고, 같은 장비를 두 사람이 동시에 못 만지도록 **단독 점유 lock**이 걸리며, 성공하면 core별 dump 결과를 ZIP으로 압축해 **MinIO에 보관**하고 다운로드까지 — 어떤 계층을 거치는가?"

다섯 번째 L2입니다. [L2 비교](/learn/l2-compare/)의 체크리스트를 따라 작성되었고, 이 모듈의 특이점은:

- **외부 장비 두 대를 서로 다른 프로토콜로** 오케스트레이션 — JTAG 서버는 **SSH**, T32 PowerView는 **t32remote(gRPC)**
- **단독 점유 lock** — 같은 T32 PC를 공유하는 다른 slot의 동시 dump를 막음 (RDP 단독접속 방식)
- **Step 순서와 실패 판정**이 도메인 특화 — voltage/success regex, `STATE.TARGET()`, ExecuteCommand 실패
- **Linux ↔ Windows 경로 변환** + **core별 cmm 스크립트**(coreScriptsJson)
- **결과 영속화** — ZIP을 MinIO에 업로드하고 `(testType, source, historyId)`로 추적, 60일 후 자동 삭제

:::note[옛 SSH+bat 경로는 legacy]
초기 구현은 모든 Step을 **SSH + `.bat`** 로 돌렸습니다. 지금은 `T32Config`에 `t32RemoteHost`/`t32RemotePort`가 채워져 있으면 **Step 2·3을 gRPC(t32remote)** 로 수행합니다(권장 경로). 이 모듈은 **gRPC 경로**를 기준으로 설명하고, SSH+bat 경로는 fallback으로만 언급합니다. t32remote 서비스 자체는 별도 모듈 [L2 — t32remote (Go gRPC)](/learn/l2-t32remote/)에서 다룹니다.
:::

## 앞 네 L2와의 차이

| 축 | 슬롯 | 벤치마크 | 원격 | Metadata | **T32 Dump** |
|---|---|---|---|---|---|
| 트리거 | Head | 사용자 Run | 세션 | 자동 | **사용자 Run (+lock)** |
| 외부 시스템 | Head TCP | Go Agent gRPC | guacd | adb shell | **SSH(JTAG) + gRPC(T32 PowerView)** |
| 동시성 제어 | 없음 | job | 세션 | 없음 | **configId 단독 점유 lock** |
| 수명 | 페이지 | job | 세션 | 분 주기 | **단발 (수십 초~수 분)** |
| 전송 | SSE | SSE | WebSocket | HTTP 폴링 | **SSE (Stepper)** |
| 실패 판정 | 없음 | exitCode | 연결 실패 | non-JSON skip | **voltage/regex + `STATE.TARGET()` + ExecuteCommand 실패** |
| 결과물 | 없음 | DB + MinIO | 없음 | SFTP JSON | **폴더 + ZIP → MinIO + DB 아티팩트** |

가장 큰 특징은 "**단발 + 5단계 절차 + 단계별 실패 판정 + 단독 점유**". 성공이 아닌 **"어느 단계에서 실패했는가"가 핵심 출력**이 되는 워크플로우.

## 왜 JTAG 서버는 SSH, T32 PC는 gRPC인가

- **JTAG 서버** (Linux·라즈베리파이): JTAG 케이블을 꽂은 물리 장치. 단순 쉘 명령 한 줄(전원/연결)로 끝나 **SSH 가 충분**.
- **T32 PC** (Windows): Lauterbach **TRACE32 PowerView** 가 설치된 PC. PowerView 는 **세션 안에서만** USB 디버그 모듈을 보고, 명령(PRACTICE)·다이얼로그·진행 이벤트가 **비동기**라 SSH 화면 긁기로는 불안정했음.

그래서 T32 쪽은 PowerView 의 네이티브 Remote API(`t32api64.dll`)를 감싼 Go 데몬 **t32remote** 를 두고 **gRPC** 로 호출합니다. Step 2 Attach, Step 3 core별 dump, 스크린샷, OS 명령(robocopy/ZIP) 이 모두 같은 세션에서 안정적으로 돌아갑니다.

```
Portal(T32DumpService) ──SSH──▶ JTAG 서버 (Linux)
                       ──gRPC─▶ t32remote.exe ──t32api64.dll──▶ TRACE32 PowerView (Windows)
                       ──S3───▶ MinIO (결과 ZIP 보관)
```

## 학습 순서와 예상 시간

| 단계 | 제목 | 성격 | 시간 |
|---|---|---|---|
| 1 | 엔티티 & 템플릿 | 읽기 | 8분 |
| 2 | 사용자 트리거 & Bitbucket | 의사코드 + 읽기 | 8분 |
| 3 | Step 실행자 (SSH + gRPC) | 코드 투어 | 15분 |
| 4 | SSE 중계 (step-* 이벤트) | 코드 투어 | 10분 |
| 5 | Stepper UI + 다운로드 | 의사코드 | 10분 |
| 6 | 단독 점유 lock + 중단 | 코드 투어 | 8분 |
| 7 | 결과 보관 (MinIO + cleanup) | 코드 투어 | 8분 |

총 약 **65분**. gRPC 자체가 처음이면 먼저 [t32remote 모듈](/learn/l2-t32remote/)을 보고 와도 됩니다.

## 새로 등장할 개념

- **JSch** — JTAG 서버 SSH 실행에 쓰는 라이브러리 (Step 1, legacy 경로). 도메인 도구라 인라인 소개만.
- **t32remote gRPC** — Step 2·3의 핵심. 별도 [L2 — t32remote](/learn/l2-t32remote/) 모듈에서 깊게.
- **단독 점유 lock** — `ConcurrentHashMap` 한 개로 구현하는 in-memory 락 (Phase 6).
