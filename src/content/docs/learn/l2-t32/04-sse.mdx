## 30초 요약

`executeDump()`가 Step 1~4를 순차 실행하면서 SSE 이벤트를 프론트에 전송합니다.

| event | 언제 | data 예시 |
|---|---|---|
| `locked` | 다른 사람이 이미 점유 중 | `{ lockedBy: "홍길동", since: "..." }` |
| `step-start` | Step 시작 시 | `{ step: 1, name: "JTAG 연결" }` |
| `step-output` | stdout/스트림 라인마다 | `{ step: 3, line: "[H-Core] dump 20%" }` |
| `dump-progress` | Step 3 전용 core 진행 | `{ step: 3, core: "H-Core", phase: "dump", status: "running" }` |
| `step-done` | Step 종료 시 | `{ step: 2, status: "success" \| "failed", output }` |
| `done` | 전체 종료 | `{ success, resultPath, resultWindowsPath, failedStep? }` |
| `cancelled` | 사용자 중단 | `{ message: "사용자에 의해 중단됨" }` |
| `error` | 예외 | `{ message: "..." }` |

핵심 골격은 모든 L2 공통 — **시작·진행·종료 3단 + 완료 통지**. T32 는 여기에 **`locked`(동시성)** 와 **`cancelled`(중단)** 가 더 붙습니다.

:::note[Step은 4개로 보이지만 Step 3가 두 일을 한다]
`step-start` 의 `step` 은 1~4 입니다. 다만 **Step 3(Dump)** 안에서 core별 dump·스크린샷·Canary·**result ZIP 생성**까지 끝나고, **Step 4(결과 업로드)** 는 그 ZIP 을 MinIO 에 올립니다. 즉 "5단계 절차"가 SSE 상으론 4개의 `step-*` 로 나타납니다 — 자세한 업로드 로직은 [7. 결과 보관](/learn/l2-t32/07-results/).
:::

## 왜 step-start + step-done 두 이벤트인가

"step-done만 있어도 되지 않나?" — 굳이 start 가 필요한 이유:

- **Step 이 길 수 있음** — Step 3 dump 는 수십 초~수 분. 시작 신호가 있어야 UI 가 "진행 중"으로 바뀌고 경과 타이머가 시작됨.
- **step-output 과의 시맨틱 매칭** — start 이후부터 output 이 쌓이고 done 에서 멈춘다는 예측 가능한 순서.
- **실패 경로에서도 일관성** — 실패해도 start 는 반드시 발생 → 프론트가 "어느 Step 에서 실패했나"를 표현.

## sendEvent 구현 — 한 곳에서 예외 처리

`sendEvent(emitter, name, data)` 는 `SseEmitter.send()` 의 `IOException` 을 잡고 로그만 남깁니다(데이터는 `application/json;charset=UTF-8` — 한글 깨짐 방지). 이유:

- **프론트가 먼저 끊어졌을 수 있음** — 사용자가 페이지 이탈 (그 경우 `emitter.onError` 가 워커를 interrupt → `cancelled`)
- **예외가 상위 loop 를 막으면 안 됨** — 다음 이벤트도 계속 시도

## 이벤트의 프론트 매핑

```ts
function onEvent(name: string, data: Record<string, unknown>) {
  switch (name) {
    case 'locked':       showLockBanner(data.lockedBy, data.since); break;
    case 'step-start':   startStep(data.step); break;
    case 'step-output':  appendOutput(data.step, data.line); break;
    case 'dump-progress': updateCore(data.core, data.phase, data.status); break;
    case 'step-done':    finishStep(data.step, data.status, data.output); break;
    case 'done':         finishAll(data.success, data.resultPath); break;
    case 'cancelled':    markCancelled(data.message); break;
    case 'error':        showError(data.message); break;
  }
}
```

단순한 switch 하나로 수신 완결. 다음 단계에서 **이 핸들러가 UI 상태를 어떻게 반영**하는지 다룹니다.
