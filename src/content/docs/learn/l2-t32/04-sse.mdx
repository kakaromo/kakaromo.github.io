## 30초 요약

`executeDump()`가 Step 1~4를 순차 실행하면서 **5가지 이벤트**를 SSE로 프론트에 전송합니다.

| event | 언제 | data 예시 |
|---|---|---|
| `step-start` | Step 시작 시 | `{ step: 1, name: "JTAG 연결" }` |
| `step-output` | SSH stdout/stderr 라인마다 | `{ step: 3, line: "H-Core dump 20%" }` |
| `dump-progress` | Step 3 전용 Core 진행 파싱 | `{ core: "H-Core", phase: "dump", status: "running" }` |
| `step-done` | Step 종료 시 | `{ step: 2, status: "success" \| "failed", output }` |
| `done` | 전체 종료 | `{ success, resultPath, resultWindowsPath, failedStep? }` |

이벤트 스키마는 **모든 L2에서 공유되는 원칙** — 시작·진행·종료의 3단 구조 + 완료 통지.

## 왜 step-start + step-done 두 이벤트인가

"step-done만 있어도 되지 않나?"라는 질문이 자연스럽습니다. 굳이 start가 필요한 이유:

- **Step이 길 수 있음** — Step 3 dump는 수십 초~수 분. 시작 신호가 있어야 UI가 "진행 중"으로 바뀌고 경과 타이머가 시작됨.
- **step-output과의 시맨틱 매칭** — start 이후부터 output이 쌓이고 done에서 멈춘다는 예측 가능한 순서.
- **실패 경로에서도 일관성** — 실패해도 start는 반드시 발생 → 프론트가 "어느 Step에서 실패했나"를 UI에 표현.

## sendEvent 구현 — 한 곳에서 예외 처리

`sendEvent(emitter, name, data)`는 내부에서 `SseEmitter.send()` 호출 시 `IOException`을 잡고 로그만 남깁니다. 이유:

- **프론트가 먼저 끊어졌을 수 있음** — 사용자가 페이지 이탈
- **예외가 상위 loop를 막으면 안 됨** — 다음 이벤트도 계속 시도 (이미 끊긴 emitter가 다 뱉고 결국 server 측에서 정리)

## 5가지 이벤트의 프론트 매핑

```ts
function onEvent(name: string, data: Record<string, unknown>) {
  switch (name) {
    case 'step-start':   startStep(data.step); break;
    case 'step-output':  appendOutput(data.step, data.line); break;
    case 'dump-progress': updateCore(data.core, data.phase, data.status); break;
    case 'step-done':    finishStep(data.step, data.status, data.output); break;
    case 'done':         finishAll(data.success, data.resultPath); break;
    case 'error':        showError(data.message); break;
  }
}
```

단순한 switch 하나로 수신 완결. 다음 단계에서 **이 핸들러가 UI 상태를 어떻게 반영**하는지 다룹니다.
