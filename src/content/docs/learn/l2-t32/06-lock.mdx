## 30초 요약

한 T32 장비에서는 **한 번에 한 명만** dump 할 수 있어야 합니다 (RDP 단독 접속과 같은 발상). 물리적으로 같은 JTAG/PowerView 를 두 dump 가 동시에 만지면 둘 다 깨지기 때문입니다.

- **lock 키는 `t32ConfigId`** — serverId 가 아니다. 여러 slot 이 한 T32 PC(=한 Config)를 공유하므로.
- **저장소는 `ConcurrentHashMap` 한 개** — in-memory, 단일 인스턴스 가정. 재시작 시 lock 초기화(진행 중 dump 도 같이 죽으니 정합성 OK).
- **획득은 dump 시작 전** — 거부되면 `locked` 이벤트만 보내고 끝.
- **해제는 워커 finally 단일 지점** — 성공/실패/예외/중단 모두 한 곳에서.
- **중단은 점유자만** — 다른 사람 dump 를 실수로 못 끊게 `userKey` 일치 확인.

`serverId 11`(slot A)과 `serverId 12`(slot B)가 **같은 T32 PC**(`configId 7`)에 매핑돼 있다고 합시다. serverId 로 잠그면 키가 11·12 로 달라 둘 다 통과 → 같은 장비를 동시에 만져 충돌. **configId 로 잠그면 두 번째가 막힙니다.** `serverId → configId` 변환은 `resolveConfigId(serverId)`(= `T32ConfigServer` 조회).

## 중단(cancel)의 전체 경로

"중단" 버튼(또는 브라우저 연결 끊김)이 lock 을 어떻게 푸는지:

1. 프론트가 **`POST /api/t32/dump/cancel?serverId=`** 호출 (SSE fetch 의 abort 만으로는 서버가 늦게/못 감지하므로 명시적 호출).
2. `cancelDump(serverId, userKey)` → `resolveConfigId` → `running` 에서 워커 조회.
3. **점유자 일치 확인** — `userKey` 가 다르면 무시(남의 dump 보호).
4. `future.cancel(true)` 로 **워커 스레드 interrupt**.
5. interrupt 가 `Thread.sleep`/blocking I/O 를 풀고, Step 3 루프의 `isInterrupted()` 체크가 즉시 빠져나옴.
6. try-with-resources 가 **t32remote 세션 close**(→ Detach), 워커 **finally 가 lock 해제** + running 엔트리 제거.
7. 워커는 `cancelled` 이벤트를 보낸다(error 로그 없음 — 정상 취소).

`emitter.onError`/`onTimeout` 도 같은 `cancelDump` 를 호출하므로, **사용자가 페이지를 닫아 SSE 가 끊겨도** 동일 경로로 lock 이 풀립니다.

:::caution[lock 해제는 반드시 한 곳에서]
lock 해제를 cancel 경로와 정상 종료 경로 양쪽에 흩어 두면, 늦게 도착한 cancel 이 <strong>다음 사람의 lock</strong>을 풀어버리는 사고가 납니다. 그래서 해제는 워커 <code>finally</code> 단일 지점 + <code>release</code> 의 <strong>점유자 일치 검사</strong> 두 겹으로 보호합니다.
:::

## 이 패턴의 한계 (의도된 단순화)

- **단일 인스턴스 전제** — 맵이 JVM 메모리라 멀티 인스턴스에선 안 통한다. 확장 시 DB row lock / Redis 로 승격.
- **재시작 시 lock 소실** — 하지만 그 순간 진행 중 dump 도 같이 죽으니 "유령 lock" 은 안 생긴다.
- **타임아웃 없는 lock** — 워커가 죽으면 finally 가 반드시 풀어주므로 stale lock 이 남지 않는다(SseEmitter 10분 timeout 이 안전망).
