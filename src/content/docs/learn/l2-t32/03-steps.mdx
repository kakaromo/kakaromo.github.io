## 30초 요약

`T32DumpService`가 **Step 1 → Step 2 → Step 3 → Step 4**를 순차로 돌립니다. 대상 서버와 **프로토콜**이 Step마다 다릅니다.

| Step | 대상 | 프로토콜 | 판정 기준 | timeout |
|---|---|---|---|---|
| 1. JTAG 연결 | JTAG 서버 (Linux) | **SSH** (JSch) | `jtagSuccessPattern` regex · "Measured Voltage" > 0 · exitCode=0 | 30초 |
| — PowerView 재시작 | T32 PC (Windows) | SSH(taskkill) + **PsExec** | (best-effort, `t32StartCommand` 있을 때만) | — |
| 2. T32 Attach | T32 PC (Windows) | **gRPC** (t32remote) | `attach` 의 `t32_version` ≠ 빈값 **+** `STATE.TARGET()` 정상 | 30초 |
| 3. Dump 실행 | T32 PC (Windows) | **gRPC** (t32remote) | core별 `ExecuteCommand` 전부 성공 | 300초/명령 |
| 4. 결과 업로드 | MinIO | S3 API | (soft-fail, dump 자체는 이미 완료) | — |

앞 단계가 실패하면 **뒤는 실행하지 않고 즉시 `done {success: false, failedStep: N}`** 을 전송.

:::note[gRPC 경로 vs legacy SSH+bat]
`t32RemoteHost`/`t32RemotePort`가 채워져 있으면 Step 2·3은 **gRPC(t32remote)** — `executeStep2AttachGrpc` / `executeStep3DumpGrpc`. 비어 있으면 **legacy** `executeStep2Attach`(SSH port check, "Down" 키워드) / `executeStep3Dump`(SSH `.bat`, "fail" 키워드). 아래는 **gRPC 경로**를 기준으로 설명합니다. Step 1(JTAG)은 두 경로 공통으로 SSH 입니다.
:::

## 왜 Step 1과 Step 2 사이에 PowerView 재시작이 들어가는가

JTAG 연결을 바꾸면(Step 1) **이미 떠 있던 PowerView 가 fatal `#FF` 상태**가 됩니다. 그래서 Step 2 attach 전에 PowerView 를 한 번 재시작해야 합니다. 그런데 그냥 SSH 로 띄우면 **콘솔 세션 밖**이라 USB 디버그 모듈을 못 봅니다(검은 화면). 그래서:

1. `taskkill /f` 로 PowerView 종료 (USB 핸들이 풀리도록 ~6초 대기)
2. `query session` 으로 활성 RDP/콘솔 세션 ID 탐지
3. **PsExec** `-i <세션ID> -d` 로 그 세션에 GUI 가 보이게 다시 실행

`t32StartCommand` 가 비어 있으면 이 과정을 통째로 건너뜁니다(수동 운영). 자세한 의도는 코드 주석 `restartPowerView` 참고.

## Step 1 (JTAG, SSH)은 그대로

Step 1 은 두 경로 공통으로 **JTAG 서버에 SSH 한 줄**입니다(`executeStep1Jtag`). 판정은 우선순위:

1. `jtagSuccessPattern` regex 가 설정돼 있으면 stdout 매치 (CASE_INSENSITIVE)
2. 없으면 stdout 의 `Measured Voltage: X.XXXXXV` 를 파싱해 **0V 초과면 연결**
3. 그래도 안 되면 exitCode==0 fallback

`{tentacle}`/`{slot}` 치환 전에는 **영숫자·언더스코어·하이픈만 허용**(명령어 인젝션 방지). SSH 실행 헬퍼(JSch · 실시간 stdout · 단위 timeout)는 legacy 경로와 공유합니다.

## 왜 한 서비스가 SSH와 gRPC를 모두 다루는가

`T32DumpService` 하나가 JTAG 서버(SSH)와 T32 PowerView(gRPC)를 모두 상대합니다. 더 추상화할 수도 있지만 MOVE 는 단순화를 택:

- **dump 시나리오 하나에만 쓰이는 로직** — 다른 곳에서 재사용 의도 없음
- **Step별 특수 로직이 많음** — Step 3의 core 루프·Canary 후처리를 공통 Executor 에 넣으면 오히려 복잡
- **로그 추적이 쉬움** — 한 서비스 안에 Step들이 순서대로

**"미래를 위한 추상화보다 현재의 읽기 쉬움"** 을 우선했습니다. 단, T32 PowerView 제어 자체는 t32remote 라는 **별도 서비스로 분리**되어 있어, gRPC 경계 너머의 복잡성(PRACTICE 명령·세션·폴링)은 거기서 캡슐화됩니다 → [L2 t32remote](/learn/l2-t32remote/).
