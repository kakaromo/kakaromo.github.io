## 30초 요약

`T32DumpService`가 **Step 1 → Step 2 → Step 3 → Step 4**를 순차로 돌립니다. 각 Step은 **하나의 SSH 명령**이고, 대상 서버는 Step마다 다릅니다.

| Step | 대상 서버 | 판정 기준 | timeout |
|---|---|---|---|
| 1. JTAG | JTAG 서버 (Linux) | `jtagSuccessPattern` regex 매치 or exitCode=0 | 30초 |
| 2. Attach | T32 PC (Windows) | exitCode=0 **+** stdout에 `"Down"` **없음** | 30초 |
| 3. Dump | T32 PC (Windows) | exitCode=0 **+** stdout에 `"fail"` **없음** | 300초 |
| 4. 완료 | T32 PC | Canary 폴더 있으면 PowerShell로 ZIP | 60초 |

앞 단계가 실패하면 **뒤는 실행하지 않고 즉시 `done {success: false, failedStep: N}`** 을 전송.

## 왜 두 서버를 한 서비스가 직접 다루는가

`T32DumpService` 하나가 JTAG 서버와 T32 PC를 모두 상대합니다. 더 추상화된 구조(예: `SshExecutor` 빈 하나가 서버 객체를 받아 실행)도 가능하지만 MOVE는 단순화를 택:

- **dump 시나리오 하나에만 쓰이는 로직** — 다른 곳에서 재사용 의도 없음
- **Step별 특수 로직이 많음** — Step 3의 Core 진행 파싱 같은 것을 공통 Executor에 넣으면 오히려 복잡해짐
- **로그 추적이 쉬움** — 한 서비스 안에 Step들이 순서대로 나열됨

**"미래를 위한 추상화보다 현재의 읽기 쉬움"** 을 우선했습니다.

## 직접 해보기

- [ ] Step 3 실행 중 stdout에 `"fail"`이 포함되면 exitCode 관계없이 실패 처리되는지 — 테스트용 스크립트로 검증
