## 30초 요약

매크로를 **시나리오 step 으로 끼워 넣을 수 있게** 한 게 신규 트랙의 마지막 조각입니다. 기존 7 step type 에 8번째로 `app_macro` 추가:

```
benchmark · shell · cleanup · sleep · trace_start · trace_stop · condition · app_macro  ← 신규
```

이로써 **"매크로 → 결과 OCR → 점수가 임계값 이하면 다시 실행"** 같은 자동 검증 루프를 만들 수 있게 됨.

## proto 한 줄로 통합

```proto
message StepNode {
  string type = 1;     // "...", "app_macro"
  ...
  AppMacroConfig macro = 5;   // type="app_macro" 일 때 사용
}

message AppMacroConfig {
  int64 macro_id = 1;          // DB 에 저장된 매크로 ID
  string macro_name = 2;
  repeated MacroEvent events = 3;  // 인라인 (macro_id 없을 때)
}
```

이 한 메시지가 **macro_id 참조 모드**와 **인라인 모드** 둘 다 지원. iotest 의 preset_id vs config_json 과 같은 패턴.

## 두 모드의 트레이드오프

| 모드 | 장점 | 단점 |
|---|---|---|
| **macro_id 참조** | 한 매크로를 여러 시나리오에서 재사용. 매크로 수정 시 모든 시나리오에 즉시 반영 | 매크로 삭제 시 시나리오가 not found 로 깨짐 |
| **인라인 events** | 시나리오가 self-contained — 매크로 삭제와 무관 | 매크로 수정 사항이 시나리오에 자동 반영 안 됨 |

**기본 정책**: 새 시나리오는 **macro_id 참조** 권장 (재사용성). 시나리오를 다른 환경에 export 할 때만 인라인 변환.

## Spring 의 조립 코드

[AgentController](/learn/l2-scenario/03-serialize/) 의 `canvasToProto` 안에:

```java
if ("app_macro".equals(s.get("type"))) {
    AppMacroConfig.Builder mb = AppMacroConfig.newBuilder();
    if (s.get("macroId") instanceof Number n) {
        AppMacro macro = appMacroService.findById(n.longValue());
        mb.setMacroId(n.longValue())
          .setMacroName(macro.getName());
        // events 는 Go Agent 가 macro_id 로 다시 fetch 하거나
        // Spring 이 미리 events_json 을 디코딩해 넣을 수 있음
    } else if (s.get("events") instanceof List<?> events) {
        // 인라인 모드
        for (Object e : events) {
            mb.addEvents(parseEvent(e));
        }
    }
    stepBuilder.setMacro(mb.build());
}
```

`macro_id` 만 보내면 Go Agent 가 자체 DB 조회 — 하지만 현재 구조는 **Spring 이 사전에 events_json 을 디코딩해서 events 배열을 보냄** (Go Agent 가 portal DB 에 직접 접근하지 않음).

## 활용 시나리오 4 가지

### 1. 자동 벤치마크 + 검증

```
[1] app_macro: AnTuTu 실행 → metrics["score"] 추출
[2] condition: metrics["score"] >= 850000 ?
    [3] shell: echo "PASS"
    [4] app_macro: 다시 실행 (재현성 확인)
```

### 2. 회귀 테스트 (loop)

```
[loop 5회]
  [1] app_macro: 매크로 1
  [2] sleep 30s
  [3] app_macro: 결과 화면 OCR 만 (metrics 누적)
[end loop]
```

5회 실행한 점수의 평균/표준편차 확인 → 펌웨어 안정성 검증.

### 3. 멀티 디바이스 비교

같은 매크로를 여러 디바이스에서 동시 재생 → metrics 비교. App Macro + Scenario Canvas 의 `parallel_devices` 플래그로 가능.

### 4. 트레이스 + 매크로 합성

```
[1] trace_start: ftrace
[2] app_macro: 게임 매크로 1분 실행
[3] trace_stop
[4] (자동) Trace L2 가 trace 분석
```

벤치마크 점수가 아닌 **시스템 동작** 자체를 trace 로 분석. App Macro + Trace 두 신규 트랙의 자연스러운 결합.

## 4 트랙 통합 그림 — Orchestration 축의 확장

[Scenario Canvas L2](/learn/l2-scenario/) 가 **Orchestration 축의 첫 사례**였다면, App Macro 통합으로 축이 **확장**됩니다:

| 시나리오 step | 외부 엔진 | 결과 |
|---|---|---|
| benchmark | fio/iozone/tiotest (Agent) | IOPS/BW |
| shell | adb shell | stdout |
| trace_start/stop | ftrace/blktrace (Agent + Rust) | parquet |
| iotest | iotest engine (Agent) | thread별 IOPS |
| **app_macro** | **InputDispatcher + Tesseract** | **ocr_results + metrics** |

5종 다른 엔진을 **하나의 DAG 에서 조합** → Orchestration 축의 진가.

## 여전히 안 풀린 문제

App Macro 가 추가됐지만 미해결 과제:

1. **OCR 실패 시 fallback** — Tesseract 가 인식 못 한 경우 "임의 값으로 진행"인지 "실패"인지 정책 미정
2. **앱 업데이트 자동 감지** — 매크로가 깨진 걸 사용자가 직접 발견해야 함 (자동 재계측 없음)
3. **매크로 버전 관리** — `(복사)` 로 fork 만 가능, semver 같은 정식 버전 없음
4. **녹화 시 의도 추론** — "버튼 X 누르기" 같은 의미적 정보 없음 (좌표만)

이 문제들은 **MOVE 가 향후 어떤 방향으로 진화할지의 단서**. 4번이 풀리면 App Macro 가 UI Automator 보다 강력해짐.

## 직접 해보기

- 매크로 1개를 시나리오 step 으로 추가 → 인라인 모드 vs macro_id 참조 둘 다 테스트
- Scenario 안에서 condition step 으로 metrics 비교 → 분기 동작 확인
- Loop 안에서 같은 매크로 5회 → 5회분 metrics 가 누적되는지

## 모듈 마무리

App Macro 는 **20번째 L2** 입니다. 신규 L3 개념 3종 (Tesseract OCR · Android InputDispatcher · 해상도 비례 변환) 도입.

축 분류:

- **기능 축 12호** — 새 도메인 (Android UI 자동화) 신규 진입. 데이터 차원이 시간 시퀀스 (1차원 배열) 라는 점에서 Trace (다축 컬럼) 와 대조
- **Orchestration 축 확장** — 5종 다른 엔진을 한 DAG 에서 조합 가능 (Scenario step type 8개로 증가)
