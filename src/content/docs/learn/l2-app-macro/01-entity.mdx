## "단일 컬럼 JSON" 패턴 비교 (3 모듈)

| L2 | 컬럼 | 타입 | 내용 | 재생 시 |
|---|---|---|---|---|
| 🔬 iotest | `config_json` | MEDIUMTEXT | thread + commands 트리 | Go executor 가 파싱 |
| 🕸 Scenario | `steps_json` + `loops_json` | TEXT | DAG 노드·엣지 | Spring 이 파싱 → topo sort |
| **App Macro** | `events_json` | MEDIUMTEXT | **MacroEvent 배열** | **Go Agent 가 순서대로 재생** |

3 모듈 모두 같은 트레이드오프:

- **장점**: 정규화 비용 0, 사용자 편집 폼이 트리/DAG/배열 한 단위로 매핑됨
- **단점**: SQL 로 부분 쿼리 불가 (예: "tap 만 5번 이상 있는 매크로 검색" → 불가능)
- **언제 OK**: 부분 쿼리 요구 자체가 없을 때. App Macro 는 매크로 단위 CRUD 외에 안 함 → OK

## 직접 해보기

- 매크로 1개 생성 → `events_json` 컬럼 직접 SELECT 해서 JSON 구조 확인
- 같은 매크로를 `duplicate` API 로 복제 → "복사" 매크로의 events_json 이 똑같이 복제되는지 확인
- DB 에서 events_json 을 직접 수정 (예: 한 tap 의 좌표 변경) → 재생 시 반영되는지 확인 (raw JSON 편집의 위험성)
