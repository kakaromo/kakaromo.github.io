## 프록시 패턴 — 다른 L2 와 비교

| L2 | Spring 의 역할 | 외부 서비스 |
|---|---|---|
| 🎯 Trace | 얇은 프록시 + 인증 | Rust gRPC (:50053) |
| 🔬 iotest | 얇은 프록시 + preset 조립 | Go Agent (:50051) |
| 🕸 Scenario | 직렬화 + 위상 정렬 + 실행 추적 | Go Agent |
| **App Macro 녹화** | **순수 프록시** | **Go Agent** |
| **App Macro 재생** | **순수 프록시 + OCR 결과 가공** | **Go Agent + Tesseract (Agent 내부)** |

녹화는 가장 얇은 프록시. **Spring 의 책임 = 인증 + DB 저장 + JSON 변환** 3가지. 비즈니스 로직 0.

## Recording 동안의 상태

| 단계 | Spring 상태 | Go Agent 상태 | DB |
|---|---|---|---|
| 시작 전 | 없음 | idle | (관계 없음) |
| StartRecording 호출 | 없음 (응답만 전달) | recording (session_id 보유) | 없음 |
| 사용자 조작 | 없음 | events 누적 | 없음 |
| StopRecording | 없음 (응답 전달) | idle (events 일회성 반환) | 없음 |
| 사용자 검토 | 없음 | 없음 | 없음 |
| 사용자 저장 | POST /app-macros | 없음 | events_json INSERT |

**Spring 이 무상태**라는 점이 핵심. 운영 중 Spring 재시작해도 녹화 세션이 살아 있음 (Go Agent 가 보유).

## WebSocket 화면 미러링

녹화 중 사용자가 디바이스 화면을 어떻게 보나? — `getScreenWebSocketUrl()` 로 별도 WebSocket 으로 H.264 스트림 미러링. JMuxer 로 디코딩.

이 부분은 [원격 터미널 L2](/learn/l2-remote/) 와 비슷한 패턴 (브라우저 ↔ 외부 서비스 직결, Spring 미경유). 다만 화면 영상은 "녹화의 일부가 아님" — 단지 사용자가 화면을 보면서 조작할 수 있게 하는 보조.

## 직접 해보기

- StartRecording 호출 후 화면 1번 tap → StopRecording → events 배열에 정확히 1개 tap 이 들어왔는지 확인
- 녹화 중 swipe → events 배열에 type="swipe" 가 1개 (tap 2개로 안 쪼개졌는지)
- StopRecording 후 저장 안 한 채 페이지 새로고침 → events 가 사라지는지 (정상)
