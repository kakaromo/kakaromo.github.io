## 30초 요약

재생은 events 배열을 Go Agent 에 그대로 보내면 끝납니다. 하지만 두 가지 추가 기능이 핵심:

1. **좌표 변환** — 녹화 해상도 ≠ 재생 해상도일 때 자동 비례 (`source_width/height` 전달)
2. **OCR 결과 추출** — screenshot 이벤트마다 Tesseract 가 텍스트 추출, 정규식 매칭으로 숫자만 골라냄 → `metrics map` 으로 응답

응답 형식이 **일반 RPC 와 다름**: `ocr_results` 와 `metrics` 두 map 이 핵심.

## OCR 추출의 3 단계

screenshot 이벤트가 실행되면 Go Agent 가 다음 3 단계 수행:

1. **스크린샷 캡처** — `screencap` 으로 PNG 바이트
2. **OCR 영역 자르기** — `OcrRegion(x, y, w, h)` 가 지정되어 있으면 해당 영역만 crop. 없으면 전체.
3. **Tesseract 실행** — Go 의 [gosseract](https://github.com/otiai10/gosseract) 로 텍스트 추출
4. **정규식 매칭** — `ocr_pattern` 이 있으면 `regexp.FindString()` → `extracted_value` 에 저장

**`metrics` map 이 만들어지는 시점**: 정규식이 매칭되고 결과가 숫자 변환 가능할 때. 예: `"850123 점"` → 정규식 `"\\d+"` 매칭 → `"850123"` → `parseFloat()` → `metrics["screenshot_name"] = 850123.0`.

## 좌표 변환 공식

녹화 해상도 (W₀, H₀), 재생 해상도 (W, H) 일 때:

```
x_replay = round(x_record * W / W₀)
y_replay = round(y_record * H / H₀)
```

swipe 의 (x2, y2) 도 동일. **duration 은 변환 안 함** (시간은 해상도와 무관).

**한계**:

- 비율이 다르면 (16:9 → 4:3) tap 위치가 빈 공간으로 갈 수 있음 → 매크로 작성 시 주의 필요
- 일부 앱은 디바이스 픽셀 밀도(dpi)에 따라 UI 가 다른 위치에 그려짐 → 같은 해상도라도 동작 안 할 수 있음

## scroll_capture — OCR 의 동영상 모드

긴 결과 페이지(여러 화면)에서 모든 점수 추출 시 사용. 흐름:

```
1. 스크린샷 + OCR
2. 스와이프 (direction 방향)
3. scroll_pause 초 대기
4. 새 스크린샷에서 정규식 매칭되는 새 값 발견 시 추가
5. max_scrolls 까지 또는 새 값 없을 때까지 반복
```

응답에서는 metrics map 에 자동 인덱스 (`name_0`, `name_1`, ...) 로 누적.

## OCR 정규식 가이드 (실전)

| 시나리오 | 정규식 | 결과 |
|---|---|---|
| AnTuTu 총점 | `Total Score\s*:\s*(\d+)` | 그룹 1 = 점수 |
| GeekBench Single | `Single-Core Score\s*\n\s*(\d+)` | 점수 |
| 모든 6자리 이상 숫자 | `\d{6,}` | 첫 매칭만 |
| 소수점 포함 | `(\d+\.\d+)` | 1.23 같은 값 |

**문제**: Tesseract 는 영문 위주. 한글이 섞이면 정확도 떨어짐 → OCR 영역을 숫자만 있는 부분으로 좁히면 해결.

## 직접 해보기

- AnTuTu 결과 화면 한 번 녹화 → screenshot 이벤트의 OCR 정규식 입력 → 재생 후 `metrics` map 에 점수 들어오는지 확인
- 같은 매크로를 다른 해상도 디바이스에서 재생 → tap 위치가 자동 보정되는지
- OCR 영역을 고의로 빈 영역으로 지정 → `metrics` map 이 비어 있는지 (또는 정규식 미매칭으로 omitted)
- ScreenshotOcr 단발 RPC 호출 → 정규식 매칭 결과를 즉시 확인
