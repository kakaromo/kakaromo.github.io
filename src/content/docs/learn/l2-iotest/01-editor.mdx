## "왜 normalize 안 했나"

iotest preset을 **정규화**하려면:

```sql
CREATE TABLE iotest_preset (id, name, category, …);
CREATE TABLE iotest_thread (id, preset_id, name, …);
CREATE TABLE iotest_command (id, thread_id, op, parent_command_id, …);
```

3 테이블 + 재귀 self-join (commands 중첩 때문) → **쿼리 복잡도 급증**. 대신 **configJson 단일 컬럼**:

| 기준 | 정규화 | JSON 단일 컬럼 |
|---|---|---|
| 스키마 변경 | 필드 추가마다 ALTER | 없음 |
| loop/if 중첩 | 재귀 관계 지옥 | 자연스러운 트리 |
| 부분 수정 | 가능 | 불가 (통째 rewrite) |
| 검색 가능성 | SQL 집계 가능 | JSON 함수 필요 |
| 쓰기 속도 | 여러 행 | 1 row |

MOVE 선택: **"preset 수 수백 미만, 검색은 name/category로만, commands 변경은 통째 재저장"** → JSON 단일 컬럼이 최적.

**같은 패턴의 다른 L2**:
- **BinMapper** — `PredefinedStruct.structText` (TEXT)
- **Agent 시나리오 빌더** — 시나리오 트리 JSON
- **iotest** — `configJson` (MEDIUMTEXT)

"**사용자 편집 가능한 복잡 트리 데이터 = 단일 컬럼 JSON**" 이 MOVE 반복 패턴.

## Editor UX — 트리 조작 3 연산

프론트 IOTestEditor는 계층 구조를 조작하는 **3 primitive**:

1. **thread 추가/제거** — 최상위 리스트
2. **command 추가/제거/순서변경** — thread 안
3. **nested command 수정** — loop/if의 `commands`/`then`/`else` 안

각 연산이 IOTestConfig의 **불변 업데이트(immutable spread)** 로 구현 — Svelte 5 `$state`가 깊은 변경을 감지 못 하는 케이스 방지.

```svelte
function addCommand(thread, newCmd) {
  thread.commands = [...thread.commands, newCmd]; // 새 배열 할당
}
```

## Preset 로드 — replace vs append

내장 18 preset 각각이 "완결 시나리오" 또는 "조각":

- **완결**: Offset Write 한 덩어리가 그 자체로 실행 가능 → **replace**로 현재 config 교체
- **조각**: "sysfs_write 3 step 세트"처럼 재사용 조각 → **append**로 현재 editor에 이어붙임

UI에 두 버튼 모두 제공:

```
┌─────────────┬──────────┬──────────┐
│ Offset Write│ [Load]   │ [Append] │
└─────────────┴──────────┴──────────┘
```

사용자가 **"이 프리셋을 시작점으로 쓸지, 현재에 섞을지"** 의도를 분명히 표현.

## 내장 vs DB preset

| 축 | 내장 (presets.ts) | DB (portal_iotest_presets) |
|---|---|---|
| 수정 | 코드 변경 → 배포 | UI에서 즉시 |
| 공유 | 모든 설치판 | 해당 포털 인스턴스만 |
| 버전 관리 | git | DB 백업 필요 |
| 분류 | 코드 상수 | category 컬럼 |
| 목적 | 팀 표준/예제 | 사용자 개인/실험 |

**"팀 공유 = 코드, 개인/실험 = DB"** 의 분리. 내장이 늘어나면 category 개수도 늘어 UI 분류 정교화.
