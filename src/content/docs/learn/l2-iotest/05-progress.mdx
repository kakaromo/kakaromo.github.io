## 30초 요약

Go Agent가 실행하면서 **per-op progress event를 `SubscribeJobProgress` 스트림**으로 반환. 포털이 받아 **SSE로 프론트 중계**, 프론트가 **ThreadProgress 객체**로 집계해 thread별 진행 바 렌더.

이 페이지는 진행률 파이프라인 정리 + **엔진 축 3종(BinMapper · Perf Generator · iotest) 총평**으로 L2 14→15 완성의 의미를 짚음.

## 진행률 파이프라인 4단계

```
Go Agent (per-op 실행)
  │
  │ JobProgress { message: JSON_EVENT }
  ↓
Portal (gRPC stream receiver)
  │
  │ SSE data: event
  ↓
Frontend API (EventSource)
  │
  │ onmessage → parse → dispatch
  ↓
IOTestProgressView (ThreadProgress 누적)
```

각 경계에서 **포맷 전환**:

| 경계 | 형식 | 이유 |
|---|---|---|
| Agent → Portal | gRPC JobProgress.message (string) | generic 인프라라 typed struct 불가 |
| Portal → Browser | SSE `data: {...}` | EventSource 표준 |
| Browser 내부 | IOTestProgressEvent object | TypeScript 타입 |
| Browser 표시 | ThreadProgress | 누적 상태 |

**3번 변환**이 필요하지만 각 단계가 명확해 디버깅 가능. 하나라도 짧으면 "generic RunBenchmark 재사용"의 이익이 사라짐.

## IOTestProgressEvent — 서버→클라 1 단위

```ts
interface IOTestProgressEvent {
  thread: string;        // 어느 thread
  step: number;          // commands 배열의 index
  op: string;            // 무엇을 실행했나
  status: 'ok' | 'error' | 'running' | 'skipped';
  error?: string;
  duration_ns?: number;  // 얼마나 걸렸나
  path?: string;
  offset?: number;
  bs?: number;
  bytes?: number;        // 실제 전송된 바이트
  value?: string;        // sysfs_read 등의 읽은 값
  progress?: number;     // loop 진행
  total?: number;        // loop 총
  iter?: number;         // 현재 iteration
  op_inner?: string;     // loop 내부 실행 중인 op
}
```

**핵심 관찰**:

- **step이 고정된 정수** — 편집 중 commands 순서를 바꾸면 어긋남. 실행 중에는 immutable 가정
- **duration_ns(나노초)** — 초당 수십 op 측정 시 밀리초로 반올림하면 0이 많음
- **loop 진행은 별도 필드** — iter/progress/total. UI가 "5/100"같이 표시
- **op_inner** — `loop`가 실행 이벤트를 낼 때 **안에서 돌고 있는 op**도 같이 (시각화 정보)

## ThreadProgress — 클라 누적

```ts
interface ThreadProgress {
  name: string;
  totalSteps: number;          // thread.commands.length
  completedSteps: number;      // 지금까지 ok 상태로 온 것
  currentOp: string;           // 마지막 running 이벤트의 op
  currentIter?: number;
  currentTotal?: number;
  status: 'running' | 'completed' | 'failed' | 'idle';
  percent: number;             // 진행률
}
```

**누적 규칙**:

- `running` 이벤트 오면 currentOp 업데이트
- `ok` 오면 completedSteps++
- `error` 오면 status='failed'
- 모든 step 완료 + 에러 없음 → status='completed'

**서버는 이벤트만, 클라가 집계** — 서버가 상태 머신을 들고 있지 않음. 재연결 시 "지금까지의 완료 수" 복원 불가 (새 연결은 새 세션 가정).

## UI 렌더링 — IOTestProgressView

thread별 progress bar:

```
┌─────────────────────────────────────────────┐
│ offset_writer     5/12 steps                │
│ ▓▓▓▓▓▓▓▓░░░░░░░  42%   (loop 128/256)      │
│ write → /data/local/tmp/test/file1          │
└─────────────────────────────────────────────┘
┌─────────────────────────────────────────────┐
│ reader            12/12 steps  ✓ completed │
│ ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓  100%                       │
└─────────────────────────────────────────────┘
```

- **compact view**: scenario canvas에 임베드되는 축소판 (thread 이름 + 진행률 바만)
- **full view**: detail sheet에 표시 (현재 op 설명 + loop 정보 + 오류 로그)

**두 view가 같은 ThreadProgress 데이터 공유** — Perf Generator가 PerfPreview/생성 코드 간 일관성을 지키는 패턴의 재사용.

## 엔진 축 3종 총평

지금까지 엔진 축에 합류한 세 L2를 나란히:

### 파이프라인 구조

| 엔진 | 단계 수 | 각 단계 |
|---|---|---|
| BinMapper | 4 | Lex → Parse → IR → Read |
| Perf Generator | 2 | Analyze → Generate |
| iotest | 3 | Parse(config) → Execute(syscall) → Stream(progress) |

**단계 수가 작업 복잡도와 비례** — BinMapper는 C struct 문법이 풍부해 4단계, Perf Generator는 JSON 직접 읽어 2단계, iotest는 파싱 + 실행 + 보고 3단계.

### 네 서브축 위치

Perf Generator 05 페이지에서 제시한 **엔진 축 서브축 네 가지**:

| 서브축 | BinMapper | Perf Generator | iotest |
|---|---|---|---|
| 환경 | JVM | 브라우저 | **JVM + Go Agent + Android** |
| 스케일 | KB~MB | KB~MB JSON | **수 분~수 시간 실행** |
| 출력 | 트리 데이터 | 소스 코드 문자열 | **스트림 이벤트 + 디바이스 상태 변화** |
| 재사용성 | 자주 (같은 struct) | 드묾 (파서당 1-2회) | **매번 새 시나리오 또는 반복** |
| Side effect | 없음 | 없음 | **있음 — 파일시스템 변경** |

**iotest가 네 서브축 모두에서 극단에 위치** — 엔진 축이 얼마나 폭넓게 가변할 수 있는지 범위를 확장.

### 공통 원칙 — 엔진이 되려면

세 엔진이 공유하는 원칙이 **엔진 축의 정의**가 됨:

1. **단방향 파이프라인** — 입력→출력, 중간 branching/loop가 파이프라인 외부에 없음
2. **도메인 특화 DSL** — struct 문법 / 필드 role / I/O command. 범용 아닌 좁은 언어
3. **독립 모듈** — 다른 L2에 의존하지 않고 완결
4. **부분집합만 지원** — C 전체가 아닌 자주 쓰는 것만, JSON 모든 shape이 아닌 두 shape만
5. **사용자 시작** — 시스템 이벤트가 아닌 사용자 명시적 액션

5 원칙 중 어느 하나라도 안 맞으면 엔진 축이 아님. 예: TestDB는 (1), (3) 안 맞아 크로스-컷 축. Auth는 (1), (4), (5) 안 맞아 크로스-컷.

## 15 L2 완성 — 세 축의 비대칭

```
기능 축 (10):    실시간 2 + Job 1 + 터미널 1 + 절차 2 + 재사용 레이어 3 + 데이터 통합 1
크로스-컷 (2):   🔐 Auth (요청) · 📊 TestDB (데이터)
엔진 축 (3):     🧬 BinMapper · ⚙ Perf Generator · 🔬 iotest
───────────
합계 15
```

- **기능 축이 다수** (10/15) — 대부분 L2는 기능 단위
- **크로스-컷 2** — 요청·데이터 두 차원만으로도 시스템 전체를 가로지름 가능
- **엔진 축 3** — 엔진이 축으로서 확립되었고, 순수(2)/부작용(1)의 하위 구분까지 가능

**비대칭의 시사점**: L2 수의 대부분을 **기능** 이 차지하지만 **크로스-컷과 엔진이 없으면 전체 지도가 납작**. 세 축이 함께 있어야 "이 L2는 어디에 속하는가" 질문에 의미 있는 답.

## 향후 확장 — 네 번째 축?

새 L2가 세 축에 모두 안 맞으면 네 번째 축 후보. 현재 가능성:

- **관리 UI 축** — Guacamole Admin, T32 Admin, 권한 Admin. **기능 L2들의 관리 화면**이 공통 패턴이라면 축
- **Analytics 축** — 향후 Time-series metrics, Event logging 이 생긴다면
- **Orchestration 축** — Agent Scenario Canvas처럼 여러 단계를 DAG로 엮는 L2가 늘어난다면

**현 시점에 더 만들지 않음** — 축은 "2 이상의 사례가 같은 패턴을 공유할 때" 확립. Guacamole Admin이 하나만 나오면 "기능 축의 변주"로도 해석 가능.

## L2 템플릿의 탄력성 확인

15 L2를 같은 템플릿(4-6페이지 + ArchitectureDiagram + CodeTour + 직접 해보기)으로 씀. **구조적 한계 없음**:

- 가장 짧은 L2(Excel, 4페이지) ↔ 가장 긴 L2(Auth/TestDB/BinMapper/Perf/iotest, 6페이지 또는 5+)
- 기능/크로스-컷/엔진 세 축 모두 같은 껍질 수용
- Frontend-only(Perf Generator) · Backend-only(없음) · Fullstack(대부분) 모두 OK

새 L2 작성 시 템플릿을 기본으로, 도메인 특성만 고민하면 됨.
