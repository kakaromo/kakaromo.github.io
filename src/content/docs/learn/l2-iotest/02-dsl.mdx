## 30초 요약

DSL은 **3 레벨 계층**:

```
IOTestConfig
├── threads[]           ← 병렬 실행 단위
│   └── commands[]      ← 순차 실행
│       ├── {op:"write", offset:"{{i*4096}}", bs:"4k", …}
│       └── {op:"loop", loop_count:256, commands:[ … ]}  ← 재귀
├── duration_seconds    ← 전역 타임아웃
└── sync_start          ← 모든 thread가 barrier로 동시 시작?
```

- **재귀**: loop/if가 `commands` / `then` / `else` 자식을 가짐 → 트리 깊이 제한 없음
- **템플릿**: offset/bs 값이 `"4k"`, `"{{i*4096}}"`, `"random:0-1m"`, `"seq:0,4k,8k"` 같은 **미니 표현식**

## `{{i}}` 템플릿 — iotest의 mini-language

loop 안에서 **반복 변수**를 값에 반영하려면:

```json
{"op": "loop", "loop_count": 256, "commands": [
  {"op": "write", "offset": "{{i*4096}}", "bs": "4k", "count": 1}
]}
```

- `i` = 현재 iteration (0부터)
- `{{i*4096}}` = iteration마다 offset이 0, 4096, 8192, …
- `{{i % 2 == 0}}` = 짝수/홀수 분기

**지원 연산**: `+ - * /`, `%`, 비교(`==`, `!=`, `<`, `>`), 논리(`&&`, `||`)

**왜 풀 스크립트 엔진을 안 넣었나**:
- iotest DSL은 **I/O 시나리오 표현이 목적** — 복잡 로직은 shell op로 위임
- JavaScript-like expression = 혼란 (이게 JS인가 다른 문법인가)
- `{{…}}` 묶음으로 **"여기부터 여기까지가 계산"** 시각 경계 명확

템플릿이 **"DSL 안의 또 다른 DSL"** 이지만 경계가 명확해 학습 비용 작음.

## offset/bs 표현식 4 종

필드 값 자체가 mini-DSL:

| 형태 | 예 | 의미 |
|---|---|---|
| 리터럴 | `"4k"`, `"1m"`, `"512"` | 크기 단위 해석 |
| 템플릿 | `"{{i*4096}}"` | iteration 계산 |
| random | `"random:0-1m"` | 0~1MB 범위 무작위 |
| seq | `"seq:0,4k,8k,16k"` | 리스트에서 순회 |

**문자열로 통일**한 이유:

- 타입이 `string | number | "4k" | "random:…"` 같이 복잡해지는 걸 회피
- 프론트 폼에서 **input 하나로 여러 의미** 수용
- Go Agent가 파싱할 때 같은 parser 재사용

단점: **컴파일 타임 검증 불가** — `"randm:0-1m"` 오타가 실행 시점에야 발견. 방어: opDefs.ts에 `help` 문구로 올바른 형식 힌트.

## "구조가 코드"의 이점

IOTestConfig의 재귀 구조가 `presets.ts`에 **그대로 JS 객체 리터럴**로:

- **문법 하이라이팅** — `op`/`path`/`loop_count` 오타 시 TypeScript가 잡음 (optional 필드여도 타입 체크)
- **IDE 자동완성** — `op: '`에서 enum 후보 없지만 필드명은 자동완성
- **git diff 친화** — 한 op가 한 줄 → 수정 diff 명확

JSON 문자열 대신 JS 객체로 두는 이유 — **"수정 편의 + 타입 안전성"**. 실행 시에는 JSON.stringify로 변환.
