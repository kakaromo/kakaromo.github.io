## 30초 요약

iotest는 **전용 gRPC RPC를 만들지 않고** fio/iozone/tiotest가 쓰는 `RunBenchmark` unary RPC를 기생:

1. 프론트 → `POST /agent/benchmark/run?serverId=N` (tool=IOTEST + config JSON)
2. 포털 `AgentController` → `AgentGrpcClient.runBenchmark(request)`
3. Go Agent가 RunBenchmarkRequest 수신 → **executor switch**에서 IOTEST case 선택
4. iotest executor가 configJson 파싱 → thread마다 **goroutine spawn** → 순차 syscall
5. 중간 상태/진행률은 **`SubscribeJobProgress` server-stream**으로 역류 (다음 페이지)

전용 RPC 대신 generic 재사용한 이유와 **비용**을 함께 짚음.

## "generic RunBenchmark 기생"의 득실

### 이득

| 축 | 설명 |
|---|---|
| job 관리 재사용 | jobs Map · history · cancel 등 기존 인프라 그대로 |
| UI 컴포넌트 공유 | Agent Job Card가 iotest에도 동작 |
| 재시도 정책 재사용 | retry_count/delay 같은 필드가 enum 바꿔도 동일 |
| 새 도구 추가 용이 | enum 값 + executor case만 |

### 비용

| 축 | 설명 |
|---|---|
| 타입 안전성 손실 | `params: map<string,string>` 안에 구조체 인코딩 — proto가 검증 못 함 |
| 디버깅 어려움 | iotest 에러가 "RunBenchmark 실패"로만 표시 → 원인이 params.config 안의 어디인지 추적 필요 |
| 필드 충돌 | fio의 params 키와 iotest의 params 키가 우연히 겹치면 혼란 |
| 확장 한계 | iotest만의 특수 옵션(예: thread priority)을 params에 섞기 애매 |

현재는 **이득이 비용보다 큼** — iotest 전용 필드가 config 안에 모이니 params 키는 `"config"` 하나만 사용. 미래에 iotest 고유 기능이 더 생기면 전용 RPC 분리 검토.

## 포털과 Go Agent의 계약 — JSON shape 동기화

가장 취약한 지점: **프론트 types.ts + Go Agent parser가 IOTestConfig shape을 공유**. 계약이 깨지면:

- **프론트 신규 필드** → Go Agent가 unknown 필드 무시 (보통 안전)
- **프론트 필드 제거** → Go Agent가 기대하던 필드가 없어 nil deref
- **Go Agent 신규 필드 요구** → 프론트가 안 보냄 → 실행 실패

**방어**:

1. Go Agent 쪽도 모든 필드 optional (default zero 처리)
2. proto 파일 대신 **types.ts를 사실상 계약 문서**로 취급
3. 프리셋 18개가 "테스트 슈트" 역할 — 이 중 하나라도 실패하면 계약 깨짐 감지

**미비 지점**: 현재 자동 호환성 테스트 없음. 계약 변경 시 수동 QA에 의존 — 향후 개선 후보.

## busy_policy의 운영적 의미

iotest는 "시간 걸리는 destructive 작업":

- **reject** (기본): 다른 iotest 실행 중이면 거절 → **실수 방지** 우선
- **wait**: 큐에 대기 → **배치 실행** 시 유용 (밤새 여러 시나리오 순차)
- **force**: 기존 중단 후 실행 → **디버깅** 시 빠른 재시도

프론트 기본은 reject — "잘못 눌러도 기존 실험 안 망친다". 전문가용 옵션으로 force 노출.
