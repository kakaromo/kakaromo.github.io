## 30초 요약

원본 I/O 로그는 세 종류 — UFS (ftrace UFS), Block (ftrace block 또는 blktrace CSV), UFSCUSTOM (자체 포맷) — 인데, **프론트에서는 동일 Arrow decoder 한 벌** 만 쓴다. 비밀은 Rust 가 chart/stats payload 를 만들기 전에 **10개 공용 컬럼으로 정규화**한다는 점.

| 원본 | 정규화 전 | 정규화 후(공용) |
|---|---|---|
| UFS | `opcode`, `lba`, `time`, `action=send_req/complete_rsp` | `cmd`, `lba`, `time`, `action` |
| Block | `io_type`, `sector`, `time`, `action=block_rq_issue/complete` | `cmd`, `lba`, `time`, `action` |
| UFSCUSTOM | `opcode`, `lba`, `start_time`, (action 없음) | `cmd`, `lba`, `time`, `action="complete"` |

공용 10컬럼: `time / action / lba / size / qd / dtoc / ctoc / ctod / cpu / cmd`.

## 왜 "공용 10컬럼" 인가

- **프론트 decoder 한 벌** — Arrow `tableFromIPC` 후 `table.getChild('lba')` 같은 코드가 3 타입 모두 동작
- **ECharts/Deck.gl 양 렌더러 공유** — `Series` 타입이 유일하므로 feature flag 로만 교체 가능
- **필터 DTO 단일화** — `TraceFilterDto` 한 개로 chart/stats 양쪽 재사용 (`startLba` / `endLba` 는 Block 의 sector 에도 자동 매핑)

대신 **정규화 위치를 Rust 쪽** 에 둬야 합니다. 프론트에서 하면 세 개의 분기가 필요하고, Java 에서 하면 Arrow batch 를 다시 만들어야 합니다. Rust 는 이미 parquet → Arrow 변환 단계라 비용이 거의 없습니다.

## Parquet 스키마 3종

### UFS (`ufs-v1`)

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `time` | Float64 | ms 단위 timestamp |
| `process` | Utf8 | 프로세스 이름 |
| `cpu` | UInt32 | CPU 코어 번호 |
| `action` | Utf8 | `send_req` / `complete_rsp` |
| `tag` | UInt32 | UFS command tag |
| `opcode` | Utf8 | SCSI opcode (`0x28`=READ_10, `0x2a`=WRITE_10, …) |
| `lba` | UInt64 | Logical Block Address |
| `size` | UInt32 | I/O bytes |
| `qd` | UInt32 | Queue Depth (send +1 / complete −1) |
| `dtoc` / `ctoc` / `ctod` | Float64 | Latency ms |
| `continuous` / `aligned` | Bool | 연속 LBA / 4KB 정렬 여부 |

### Block (`block-v1`)

- UFS 와 동일한 구조 — 단 `lba` 가 아니라 **`sector`** (raw Block layer 용어)
- `opcode` 대신 **`io_type`** (`RA`, `R`, `W`, `FF`, `D`)
- **Flush 이벤트**의 특수 케이스: `size=0`, `sector=u64::MAX` — parser 단계에서 `normalize_sector` 로 `sector=0` 으로 보정 (`log_common.rs`)

### UFSCUSTOM (`ufscustom-v1`)

- `action` / `cpu` **컬럼 자체가 없음**
- `start_time`/`end_time` 으로 쪼개져 있음 — chart 변환 시 `start_time` → `time` 매핑
- `start_qd`/`end_qd` → **`end_qd`** 를 공용 `qd` 로 선택

## 공용 10컬럼 변환 규칙

Rust `output/chart_rpc.rs::build_chart_batch_*` 에서 수행.

```text
UFS → 공용
  opcode → cmd (rename)
  lba    → lba (그대로)
  time, action, size, qd, dtoc, ctoc, ctod, cpu (그대로)

Block → 공용
  io_type → cmd (rename)
  sector  → lba (rename)
  나머지 그대로

UFSCUSTOM → 공용
  opcode     → cmd (rename)
  start_time → time
  end_qd     → qd
  lba, size, dtoc, ctoc, ctod 그대로
  action ← "complete" 채움 (모든 이벤트 완료 상태)
  cpu    ← 0 채움
```

결과적으로 **세 원본 모두 같은 10 필드, 같은 타입 순서**. `SCHEMA_VERSION_UFS` / `_BLOCK` / `_UFSCUSTOM` 은 프론트 decoder 의 호환성 체크에만 쓰이고, 컬럼 레이아웃은 동일.

## 스키마 버전 관리

`chart_rpc.rs` 상수로 고정:

```rust
pub const SCHEMA_VERSION_UFS: &str = "ufs-v1";
pub const SCHEMA_VERSION_BLOCK: &str = "block-v1";
pub const SCHEMA_VERSION_UFSCUSTOM: &str = "ufscustom-v1";
```

이 문자열이 HTTP 응답 헤더 `X-Trace-Schema-Version` 에 실립니다. 프론트는 decoder 에서 **알고 있는 버전만 진행**, 그 외엔 `friendlyError` 로 경로를 유도합니다. 스키마 변경 시:

1. Rust 의 `build_chart_batch_*` 수정
2. `SCHEMA_VERSION_*` 를 올림 (예: `ufs-v2`)
3. 프론트 decoder 에 새 버전 분기 추가
4. 기존 parquet 은 구버전 헤더 그대로 돌아가게 호환 분기 유지 또는 전량 reparse

## 제약

- **스키마 변경은 비용이 큼** — parquet 이 이미 많이 쌓여 있으면 재파싱 필요 (원본이 남아 있어 가능은 함)
- **`continuous` 컬럼이 없는 구버전 parquet** 이 있을 수 있어 `stats_rpc` 는 Optional 로 읽고 없으면 재계산 — 새 parquet 을 만들면 자연히 채워짐
- **UFS MCQ 환경** 에서 hwqid 별 QD 는 아직 공용 `qd` 한 개로 합산만. 분리하고 싶으면 컬럼을 늘리고 스키마 버전을 올려야 함
