## 30초 요약

기존 sync 경로: `MinIO GET 전체 (5GB) → /tmp 저장 → File::open → ParquetRecordBatchReaderBuilder`.

문제:
- `/tmp` 10 GB 왕복 I/O (쓰기 + 읽기)
- 첫 바이트 도착이 **전체 다운로드 완료** 이후로 지연
- 동시 요청 2개만 돼도 `/tmp` 용량 위험

async 경로 (`TRACE_PARQUET_READER=async`):
- `head_object` → 파일 크기 조회
- `tail 64KB range-GET` → footer length 파싱
- 필요 시 추가 range-GET 으로 footer 보강
- `ParquetRecordBatchStream` 으로 row group 단위 async read
- **`/tmp` 에 0 byte 안 씀**

검증: async 모드 기동 후 `/tmp/chart_data_*.parquet` 생성 0건. 첫 바이트 도착 시간 수 배 단축.

## 왜 `AsyncFileReader` trait 구현인가

parquet crate 는 두 경로를 제공:
- `ParquetRecordBatchReaderBuilder` (sync) — `File`·`&[u8]` 등 blocking 소스
- `ParquetRecordBatchStreamBuilder` (async) — `AsyncFileReader` 를 요구

`AsyncFileReader` 는 두 가지 메서드만 요구:
- `get_bytes(range: Range<u64>) -> BoxFuture<Bytes>` — 임의 byte range 공급
- `get_metadata(opts) -> BoxFuture<ParquetMetaData>` — footer 파싱

즉 **임의 range 를 비동기로 가져올 수 있는 backend 라면 parquet crate 가 알아서 스트림으로 읽어줍니다**. S3-호환 MinIO 는 `get_object_range` 가 곧바로 쓰일 수 있어 궁합이 좋습니다.

## 왜 footer 2-step fetch 인가

parquet 파일의 레이아웃:

```
[data blocks...][metadata bytes][u32_le metadata_len][PAR1 magic (4B)]
                                                     ^ file 끝
```

정확한 footer 크기를 알려면 **맨 마지막 8 바이트** 를 읽어 `metadata_len` 을 봐야 합니다. 그 다음 footer 전체를 한 번 더 읽으면 끝입니다. 단순 구현은:

```
1) range-GET (file_size-8, file_size)  → 8 byte
2) range-GET (file_size-footer_total, file_size) → footer 전체
```

그런데 **보통 footer 는 수십 KB** 수준이라, tail 64KB 를 **먼저** 한 번에 가져오면 대부분 한 번으로 끝납니다. footer 가 64KB 를 넘어갈 때만 추가 range-GET. 실측치는 5GB parquet 도 1 호출로 끝나는 경우가 많습니다.

## 분기 (sync/async) 배치

`grpc/server.rs` 의 `get_chart_data` / `get_trace_stats` 핸들러 맨 앞에서:

```rust
if is_async_reader_enabled() {
    return async_path(...).await;
}
sync_path(...)   // 기존
```

- 기본값 sync — 회귀 위험 0
- async 경로는 `chart_rpc_async.rs` / `stats_rpc_async.rs` 에 별도 구현 (공유 helper 는 projection/filter 쪽)
- `continuous` 컬럼이 없는 구버전 parquet 호환이 async 쪽에서 특히 중요 — 스키마 개선 시점이 겹쳤기 때문

## 검증 체크리스트

- [ ] `/tmp/chart_data_*.parquet` 생성 건수 — `ls /tmp/chart_data_* 2>/dev/null | wc -l` 로 확인
- [ ] Rust 로그에 `[GetChartData] reader=async (range-GET)` / `[GetTraceStats] reader=async (range-GET)` 찍힘
- [ ] 큰 parquet(예: 5GB) 차트 조회 시 **첫 바이트 도착 시간** 이 sync 대비 수 배 단축
- [ ] MinIO 액세스 로그에서 range-GET 요청 빈도 확인

## 제약 / 다음 단계

- **concat_batches 여전히 동기 집약** — stream 에서 batch 를 받아 마지막에 concat 하므로 메모리 피크는 남아 있음. 필요 시 chart/stats 집계를 **batch 별 누적** 으로 바꿔 peak 를 낮출 수 있음
- **footer 캐시** — 같은 parquet 에 대한 반복 요청이 많으면 `ParquetMetaData` 를 LRU 로 캐시할 수 있음 (현재는 매 RPC 마다 재파싱)
- **cancel 전파** — 프론트 `AbortController` 로 요청을 끊어도 Rust tokio task 는 계속 돎. 크리티컬해지면 `CancellationToken` 연동 필요
