## 30초 요약

`POST /api/trace/chart` 의 처리는 **3단계** 로 요약됩니다.

1. **parquet projection + row-filter** — 10개 컬럼만 읽고 `time_range` 안에서 pruning
2. **공용 10컬럼 정규화 + Arrow filter** — `lba` / `qd` / `dtoc` 등 범위 필터를 `BooleanArray` mask 로 적용
3. **stratified row sampling → Arrow IPC 직렬화** — 시간 균등 bucket + 빈 bucket fill + 보호 row 5종 강제 (min/max time + argmax(qd) + min/max dtoc). NaN/Inf time 안전. **정확히 `target_points` row** 보장

응답은 바이너리:
- **Body**: Arrow IPC stream bytes (Content-Type `application/vnd.apache.arrow.stream`)
- **Headers**: `X-Trace-Total-Events`, `X-Trace-Sampled-Events`, `X-Trace-Schema-Version`, `X-Trace-Stats`(Base64 JSON)

## 왜 JSON 이 아니라 Arrow IPC 인가

수십만 포인트 scatter 를 JSON 으로 보내면:
- 페이로드 20~30 MB (gzip 이후에도 10 MB 수준)
- 브라우저 `JSON.parse` 수백 ms
- 모든 숫자가 double — UInt64 LBA 도 크게 차지

Arrow IPC 는:
- **Schema + columnar TypedArrays** (`Float64Array`, `UInt32Array` 등)
- 페이로드 5~10 MB (50~70% 감소)
- `tableFromIPC` 파싱 수십 ms
- 디코드 후 **WebGL binary attribute 에 0-copy** 로 넘어갈 수 있음

즉 "**차트 데이터는 row JSON 이 아니라 컬럼 IPC**" 라는 결정이 이 RPC 의 성격을 좌우합니다.

## 왜 projection + row-filter 두 단계인가

parquet 은 **컬럼 지향**이라 `ProjectionMask` 로 필요 컬럼만 디코딩하면 I/O 가 비례해서 줄어듭니다. 그러나 모든 row 를 읽는다면 시간 범위가 좁아도 여전히 큽니다.

`RowFilter` 를 추가하면 **parquet crate 가 row group / page 수준에서 pruning** 을 시도 — `time` 컬럼만 별도 투영해 predicate 평가 후 나머지 컬럼을 읽습니다. 덕분에 "마지막 30 초만 본다" 같은 요청은 파일 뒷부분만 건드립니다.

두 단계가 협력해:
- Projection: **어떤 컬럼?**
- RowFilter: **어떤 row?**

### DuckDB 경로의 두 단계 다운샘플

운영 기본인 DuckDB stats 엔진 (`TRACE_STATS_ENGINE=duckdb`) 은 다른 패턴:

1. **SQL 1차** — `chart_base` view 위에서 bucket 별 first/last/argmax(qd) 추출 → ~target×3 row 로 1차 축소 (1억 row 를 그대로 fetch 하면 client 메모리 위험)
2. **Rust 2차** — `time_bucket_decimate` 가 ~target×3 row 를 입력으로 받아 정확히 target 으로 trim (μs 단위, 비용 미미)

이렇게 두 단계로 두는 이유:
- SQL 만으로 stratified row sampling (빈 bucket fill / 보호 row drop / set 연산) 을 깔끔히 표현하기 어려움
- 1억 row 를 그대로 Rust 로 fetch 하면 메모리 폭발 — SQL 단계가 거른 후 Rust 가 정확성만 마무리
- sync/async/duckdb 세 reader 모두 같은 invariant (`정확히 target_points`) 보장

### 왜 stratified row sampling 인가

| 옵션 | 정확히 N | 시간 균등 | outlier 회복 | min/max time | QD 스파이크 | min/max dtoc | NaN 안전 |
|---|---|---|---|---|---|---|---|
| 옛 (bucket × first/last/argmax) | ❌ (≤ N×3) | ✅ | ❌ (빈 bucket 손실) | ❌ | ✅ | ❌ | ❌ |
| Row index 균등만 | ✅ | ❌ | ✅ | ❌ | ❌ | ❌ | ❌ |
| **Stratified (현재)** | ✅ | ✅ (bucket 채워진 영역) + row index (빈 영역) | ✅ | ✅ (강제) | ✅ (강제) | ✅ (강제) | ✅ |

핵심 통찰 두 가지:

1. **시간 균등 bucket 의 *빈 슬롯* 을 row index fill 로 보충** 하면 두 방식의 장점을 합침
2. **보호 row 5종** (min/max time, argmax qd, min/max dtoc) 강제 — 차트 X축 범위 + QD 스파이크 + latency 양극값 모두 항상 표시. estrace 의 sampling.rs (`bucket_downsample` 의 min-Y/max-Y 보존) 패턴과 NaN 안전성을 가져옴

운영상 의미: portal 사용자가 차트의 양 끝 시간 + 가장 높은 QD + 가장 빠른/느린 I/O 를 다운샘플 결과에서 항상 볼 수 있어 디버깅 누락 위험이 줄어듦.

## 응답 헤더 정리

| 헤더 | 값 | 용도 |
|---|---|---|
| `Content-Type` | `application/vnd.apache.arrow.stream` | `tableFromIPC` 로 decode |
| `X-Trace-Total-Events` | UInt64 | 필터 후·다운샘플 **전** row 수 |
| `X-Trace-Sampled-Events` | UInt64 | 다운샘플 **후** 실제 포인트 수 |
| `X-Trace-Schema-Version` | `ufs-v1` 등 | decoder 호환성 체크 |
| `X-Trace-Stats` | Base64(JSON) | `{dtoc, ctod, ctoc, qd}` 각 min/max/avg/p50/p95/p99 |

> 이 헤더들은 **portal HTTP 어댑터** 가 trace gRPC 응답 (`GetChartDataResponse` 의 `arrow_ipc` bytes 필드 + `total_events` / `sampled_events` / `dtoc` / `ctod` / `ctoc` / `qd` / `schema_version` 필드) 을 HTTP 로 옮긴 것. trace 서버 자체는 proto 필드로 메타를 전달.

stats 만 보고 싶으면 헤더만 읽고 body 를 버리는 것도 가능 — 다만 헤더에 박아둔 건 **가볍고 즉시** 쓰기 위한 것이고, 본격 통계는 `/api/trace/stats` 를 쓴다.

## 응답 캐시 (L5)

같은 (parquet path, time range, filter, target_points, actions) 조합이면 trace 서버가 **첫 호출 결과를 메모리에 캐시 (4시간 TTI, 합 2GB)** — 두 번째 호출부터는 0ms.

- 키에 parquet 의 ETag(fingerprint) 가 들어가 있어 parquet 가 갱신되면 자동 무효화
- `try_get_with` 기반 single-flight — 같은 키 동시 N 요청이 동시에 들어와도 DuckDB 빌더는 1번만 실행 (N-1 호출자는 같은 `Arc<Response>` 공유)
- 멀티 사용자가 같은 화면을 시간차로 보는 portal 패턴에서 hit 률 가장 높음

자세한 정책·키 직렬화·single-flight 시퀀스는 [11. 캐시 시스템](/learn/l2-trace-rust/11-cache-system/) 에서.

## gRPC payload 상한

- Rust 서버: `.max_decoding_message_size(256MB)` / `.max_encoding_message_size(256MB)`
- Portal 클라이언트: `spring.grpc.client.channels.trace-service.max-inbound-message-size: 256MB`
- 1M targetPoints × 10 컬럼 ≈ 80~120MB — **2M 근처까지 안전**
- `OUT_OF_RANGE` 가 나면 프론트가 `friendlyError` 로 "Samples 를 줄이라" 힌트 제공
- **멀티 parquet 입력** (`parquet_paths` repeated, realtime 모드의 윈도우별 parquet 합산 등) 은 N개 응답이 합쳐 single Arrow IPC 가 돼서 256MB 한도에 더 빨리 부딪힘. duckdb reader 만 지원 (`grpc/dispatch.rs`)

## 제약

- **시각적 부드러움 vs 정확한 N** — 빈 bucket 영역에서 row index fill 로 보충하므로 시간 분포가 극단적으로 한쪽에 쏠리면 그 영역에 점이 빽빽해질 수 있음. 일반적인 ftrace/blktrace 분포에선 문제 없음
- **LTTB 미적용** — 시각적 품질이 더 필요하면 Largest-Triangle-Three-Buckets 등 후보. 현재는 stratified + min/max/argmax 강제로 트렌드와 스파이크 모두 보존
- **필터 조합** — 모든 필터가 BooleanArray `and` 로 처리되므로 O(n). 큰 parquet 에서도 보통 초단위
- **`target_points` 의미** — 현재는 **결과 row 수** 의미 (`n > target` 이면 정확히 N). 이전 코드는 bucket 수였고 결과가 ≤ N×3 였음 — 운영에서 1M → 28K 같은 손실 사례가 보고돼 [stratified 알고리즘](#코드-투어--time_bucket_decimate-stratified-row-sampling)으로 교체
