## 30초 요약

`POST /api/trace/stats` 는 **한 번의 RecordBatch scan 으로 5~6가지 통계를 동시에** 계산합니다.

1. `col_f64("time")`, `col_str("action")`, … 으로 **컬럼을 미리 downcast** — 핫루프에서 `as_any` 반복 비용 회피
2. `action_pair(trace_type, action)` — **실제 리터럴을 sniff** 해 request/complete action 결정 (UFS/Block ftrace/blktrace CSV 분기)
3. 단일 `for i in 0..n` 루프에서 send_count / continuous / aligned / I/O bytes 를 **동시 누적**
4. latency 4종(`dtoc/ctod/ctoc/qd`)은 정렬 기반 percentile 로 요약
5. `cmd_stats` / `latency_histograms` / `cmd_size_counts` 도 동일 scan 결과를 써서 **O(n) 1회** 유지

반환은 JSON — Arrow IPC 와 달리 수가 적고 (몇 KB), UI 가 바로 표로 그리는 값들.

## 왜 한 번만 스캔하는가

통계 종류별로 loop 를 돌리면 읽기 코드는 짧아지지만, parquet 을 메모리에 올린 뒤에도 `O(k · n)` 스캔이 됩니다. 1M row 쯤 되면 loop 수 = 속도 차이가 체감됩니다. MOVE 는:

- 컬럼을 **루프 진입 전에 downcast** 해 포인터만 가진다
- 루프 안에서 **모든 누적 배열/맵을 동시에** 채운다
- percentile 만 정렬 기반 **별도 패스** (불가피)

결과: `O(n) + O(n log n)` 수준. latency histograms/cmd stats 도 같은 scan 의 부산물.

## 왜 `action_pair` 를 sniff 하는가

Block 의 `action` 은 두 포맷이 공존:
- **ftrace** 형식: `block_rq_issue` / `block_rq_complete`
- **blktrace CSV** 형식: `Q` / `C`

업로드 시 타입을 고정하려면 UI 에 분기가 필요하고, 재파싱 때 잘못 고르면 stats 가 모두 0. 대신 Rust 가 앞 1024 row 의 리터럴을 sniff 해 두 포맷 중 어느 쪽인지 판정. UFS/UFSCUSTOM 은 리터럴이 고정돼 있어 스캔 없이 바로 결정.

UFSCUSTOM 의 `request_action = ""` 은 의도적: **send_count 가 0 으로 자연 유도**되고, continuous 계산은 complete 기준으로 폴백 경로를 탑니다.

## percentile 계산

정렬 기반 linear interpolation — `sort_unstable_by` + 인덱스 계산.

```rust
fn percentile(sorted: &[f64], p: f64) -> f64 {
    if sorted.is_empty() { return 0.0; }
    let pos = p * (sorted.len() - 1) as f64 / 100.0;
    let lo = pos.floor() as usize;
    let hi = pos.ceil() as usize;
    let frac = pos - lo as f64;
    sorted[lo] + (sorted[hi] - sorted[lo]) * frac
}
```

p50 / p95 / p99 / p999 / p9999 / p99999 / p999999 — 6자리 9 까지 지원. parquet 이 1M 이면 정렬 비용은 수백 ms 수준. percentile 추정 알고리즘(t-digest 등) 도 후보지만 현재 규모에서는 정밀 정렬이 깔끔.

## 프론트 연동

- Statistics 탭을 클릭하면 `$effect` 에서 자동 `loadStats()` — 별도 버튼 불필요
- Brush/Filter 변경이 일어나면 **chart + stats 양쪽 재조회** (동일 필터 DTO 를 공유)
- `cmdStats` 는 cmd 별 탭 (DtoC/CtoD/CtoC/QD 4탭), `latencyHistograms` 는 cmd 별 차트, `cmdSizeCounts` 는 cmd × size 막대 차트
- 필드가 0 이면 UI 에서 해당 섹션 자동 숨김 (UFSCUSTOM 처럼 action 이 없는 경우)

## 제약

- **continuous fallback** — parquet 에 컬럼이 없으면 재계산. 결과는 같아야 하지만 HashMap 삽입 비용이 추가
- **sort percentile** — 수천만 row 까지는 sort 비용 < 로그 로딩. 그 이상은 t-digest 고려
- **`latency_ranges_ms` 는 요청 파라미터** — 기본 `[0.1, 0.5, 1, 5, 10, 50, 100, 500, 1000]` 이지만 workload 특성에 맞게 조정 가능
