## 30초 요약

`PortalServer`는 포털 관리자가 등록한 **접속 대상 서버 한 대**입니다. 프로토콜별 포트(SSH/RDP/VNC)와 인증 정보 외에도 **guacd 호스트·포트를 서버별로 오버라이드**할 수 있는 필드가 있어, 장비가 여러 네트워크 세그먼트에 걸쳐 있을 때도 적절한 guacd로 연결됩니다.

전역 guacd 기본값은 `GuacamoleProperties`(yaml)에서 옵니다. **서버별 설정이 있으면 그걸 쓰고, 없으면 전역 fallback**.

```yaml title="application-dev.yaml"
  web-url: http://192.168.1.248:9005/guacamole
  username: guacadmin
  password: guacadmin
```

## 연결 시점의 선택 로직 (미리보기)

단계 3에서 상세히 볼 코드지만, 핵심은 이 한 줄입니다:

```java
String guacdHost = vmConfig.getGuacdHost() != null
    ? vmConfig.getGuacdHost()
    : properties.getGuacdHost();
int guacdPort = vmConfig.getGuacdPort() != null
    ? vmConfig.getGuacdPort()
    : properties.getGuacdPort();
```

**서버별 > 전역**. 이 패턴은 MOVE 전반에서 반복됩니다(예: 테스트 실행 시 실행자 옵션 > 슬롯 기본값 > 시스템 기본값).

## 왜 guacd를 여러 대 둘 수 있어야 하나

실운영에서 흔한 시나리오:

- **대규모 테스트 장비 랜드**: 장비들이 물리적으로 여러 랙에 분산. 한 guacd가 모두 보려면 네트워크 routing + 지연
- **보안 구역 분리**: 외부 WAN 접속 필요한 장비 vs 분리망 장비
- **장애 격리**: guacd 하나가 죽어도 다른 영역은 살아 있도록

오버라이드 필드 하나가 이 운영 유연성을 만듭니다. 작은 데이터 모델 결정이 긴 수명을 가집니다.
