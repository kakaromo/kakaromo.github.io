## 30초 요약

터널이 열리고 나면 **두 방향**의 트래픽이 각자 다른 경로로 흐릅니다.

- **guacd → 브라우저**: reader 스레드(`readFromTunnel`)에서 `tunnel.acquireReader().read()` 루프 → `session.sendText()`
- **브라우저 → guacd**: WebSocket `@OnMessage` 콜백 → `tunnel.acquireWriter().write()`

종료는 세 가지 경로 중 하나로 옵니다 — 브라우저가 창 닫기 / guacd가 세션 끊기 / 예외. 어느 경로든 `onClose`가 최종 정리(Lock 해제·tunnel close)를 수행합니다.

## 왜 reader는 스레드, writer는 콜백인가

이 비대칭이 혼란스러울 수 있습니다.

- **guacd → 브라우저**: guacd가 "마음대로 밀어주는" 방향. **읽기는 blocking**이라 이걸 별도 스레드가 아니라 main 루프에서 돌리면 다른 메시지 처리가 멈춥니다.
- **브라우저 → guacd**: 브라우저가 WebSocket 메시지를 보낼 때마다 **컨테이너가 콜백으로 `onMessage`를 부름**. 이미 이벤트 기반이라 별도 스레드가 필요 없음.

즉, **blocking API → 스레드**, **콜백 API → 그대로** 라는 일관된 원칙입니다.

## 종료 경로 정리

| 종료 원인 | 감지 지점 | 정리 수행 |
|---|---|---|
| 브라우저가 WebSocket 종료 | `@OnClose` | Lock 해제 + tunnel close |
| guacd가 TCP 종료 | reader의 `read() == null` | finally → `closeSshTunnel` → 옆 효과로 WS close |
| reader 예외 | reader의 catch | 같음 |
| send 예외 | `safeSendText` 내부 log + 다음 loop에서 session.isOpen() false |
| writer 예외 | `onMessage`의 catch | `closeSshTunnel` |
| 외부 종료 (서버 셧다운) | WebSocket 컨테이너가 @OnClose 호출 | 같음 |

**어느 경로든 `@OnClose`에 수렴**하도록 설계되어 있어 청소 코드가 한 곳에 모입니다.
