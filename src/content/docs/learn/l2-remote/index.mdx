## 이 모듈에서 배울 것

"사용자가 포털에서 서버 카드를 고르고 **RDP** 또는 **SSH** 버튼을 누르면, 브라우저 안에서 **실제 디바이스 화면·터미널**이 열리기까지 어떤 계층을 거치는가?"

세 번째 L2입니다. [슬롯](/learn/l2-slots/)·[벤치마크](/learn/l2-benchmark/)와 달리 이 모듈의 핵심은 **WebSocket 양방향 릴레이**와 **guacd라는 외부 데몬**입니다. Spring은 "시작 단계에서만" 관여하고 이후 데이터 릴레이에 충실합니다.

## 앞 두 L2와의 차이

| 축 | 슬롯 / 벤치마크 | 원격 터미널 |
|---|---|---|
| 프로토콜 | SSE (서버→클라) | WebSocket (양방향) |
| Spring 역할 | 지속 중계 또는 이벤트 발신 | **연결 시점에만** 중계, 이후 무한 릴레이 |
| 외부 시스템 | Head TCP / Go Agent gRPC | guacd (별도 데몬, TCP 4822) |
| 데이터 | JSON 이벤트 | Guacamole 프로토콜 프레임 (텍스트 + 바이너리) |
| 세션 수명 | 초~시간 | 세션 단위 (사용자 창이 열려 있는 동안) |
| 다중 접속 | 언제나 | **RDP/VNC는 Lock**으로 1명 제한 (공유 모드 옵션) |

이 차이 때문에 코드에서 새로 만나는 것:
- **`@ServerEndpoint`**: Spring MVC가 아니라 Jakarta WebSocket API 직접 사용
- **`InetGuacamoleSocket`**: guacd와 대화하기 위한 apache-guacamole 라이브러리
- **reader/writer 스레드**: SSE처럼 한 방향이 아니라 두 방향 각자 스레드 필요

## 왜 Spring 없이 브라우저 ↔ guacd 직접이 아닌가

guacd는 **내부망 전용** TCP 데몬이고, 브라우저는 **외부망** 또는 WebSocket만 말할 수 있습니다. Spring이 **인증 + 세션 Lock + WebSocket → TCP 게이트웨이** 역할을 해야만 합니다.

Spring이 "없으면 안 되는 이유"를 정리하면:

1. **인증**: 브라우저가 인증된 사용자인지 `HttpSession`으로 검사
2. **권한 있는 서버만 노출**: `PortalServer.visible=true` + Admin 플래그
3. **Lock 조정**: RDP는 한 명만 들어가야 함 → `SessionLockManager`
4. **네트워크 격리**: 브라우저가 guacd 4822에 직접 붙을 수 없음 (보안·방화벽)

## 학습 순서와 예상 시간

| 단계 | 제목 | 성격 | 시간 |
|---|---|---|---|
| 1 | 서버 DB 모델 (PortalServer) | 읽기 | 5분 |
| 2 | WebSocket 엔드포인트 열기 (onOpen) | 코드 투어 | 12분 |
| 3 | guacd 터널 만들기 (buildConfig + connect) | 코드 투어 | 15분 |
| 4 | 양방향 릴레이 (reader/writer + onMessage) | 코드 투어 + 시퀀스 | 15분 |
| 5 | 프론트 client (Guacamole.Client + WebSocket) | 읽기 + 의사코드 | 8분 |

총 약 **55분**. 슬롯·벤치마크 L2를 먼저 읽었다면 반복 개념(예: Spring 생성자 주입, `@Component`)은 빠르게 넘기세요.
