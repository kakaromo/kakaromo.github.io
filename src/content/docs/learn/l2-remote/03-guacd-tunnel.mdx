## 30초 요약

`connectDirect`가 핵심입니다. **세 줄**의 라이브러리 호출로:

1. `InetGuacamoleSocket(guacdHost, guacdPort)` — guacd에 **TCP 연결**
2. `ConfiguredGuacamoleSocket(socket, guacConfig)` — **프로토콜 핸드셰이크** 수행 (SSH/RDP/VNC 선택 + 파라미터 전달)
3. `SimpleGuacamoleTunnel(socket)` — 양방향 **tunnel 추상화** 감싸기

이후 별도 스레드에서 guacd → 브라우저 방향을 읽는 `readFromTunnel`이 돌고, 브라우저 → guacd 방향은 `onMessage`에서 처리합니다(다음 단계).

## 왜 소켓이 세 겹인가

Apache Guacamole의 **데코레이터 패턴**:

- **`InetGuacamoleSocket`** — 순수 TCP (raw bytes)
- **`ConfiguredGuacamoleSocket`** — 위에 "Guacamole 프로토콜 핸드셰이크" 한 층 올림
- **`SimpleGuacamoleTunnel`** — 위에 "reader/writer acquire/release" 추상화

각 층은 책임이 분리되어 있어 테스트하기 쉽습니다. MOVE가 이 패턴을 바꾸는 일은 거의 없고, 라이브러리가 제공하는 조립만 따라가면 됩니다.

## 파라미터 맵 vs DTO

`GuacamoleConfiguration`은 `Map<String, String>` 기반입니다. 타입 안전성은 떨어지지만:

- **라이브러리 API라 바꿀 수 없음**
- **프로토콜별 파라미터 이름이 너무 많음** (SSH 40개+, RDP 80개+) → 각각 DTO 만들면 유지보수 비용
- **문자열 값도 guacd가 파싱** — 숫자도 string으로

타입 안전성이 필요한 케이스는 없어 지금 구조로 충분합니다.
