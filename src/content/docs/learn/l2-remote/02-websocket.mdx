## 30초 요약

`GuacamoleTunnelEndpoint`는 **Jakarta WebSocket API**(`@ServerEndpoint`)로 선언된 엔드포인트입니다. Spring MVC의 `@RestController`와는 다른 세계 — 메서드가 요청-응답이 아니라 **라이프사이클 훅**(`@OnOpen`/`@OnMessage`/`@OnClose`)으로 불립니다.

브라우저가 `/api/guacamole/tunnel?vm=X&protocol=rdp&user=Y`로 붙으면 `onOpen`이 호출되고, 이 함수가 **쿼리 파라미터 → Lock → VM 조회 → guacd 연결**을 순서대로 수행합니다.

## <ConceptPopover client:visible term="@ServerEndpoint" concept="spring/server-endpoint"><p>Jakarta WebSocket API의 어노테이션. 경로와 프로토콜을 선언하고 @OnOpen/@OnMessage/@OnClose/@OnError 메서드를 통해 WebSocket 라이프사이클을 처리합니다.</p></ConceptPopover> — Spring MVC와 무엇이 다른가

- **라이프사이클 훅** vs **요청-응답**: 한 번 열리면 여러 메시지가 오가므로 메서드 하나가 요청 하나를 받는 모델이 맞지 않음.
- **`@Controller` 스코프 → `@ServerEndpoint` 클래스는 기본으로 Spring DI 대상이 아님**: `SpringWebSocketConfigurator`가 이 간극을 메움. 만약 설정 빠뜨리면 `new` 생성자가 호출되어 DI가 실패합니다.
- **서브프로토콜**: WebSocket 핸드셰이크에서 `Sec-WebSocket-Protocol: guacamole` 헤더로 프로토콜을 선언해 guacamole-common-js가 올바른 포맷으로 대화하도록.

## 왜 JSON이 아니라 Guacamole 프로토콜 문자열인가

슬롯·벤치마크 SSE는 `JSON.stringify` 후 `event: data`로 보냈습니다. 원격 터미널은 그 층을 뺍니다.

- **Guacamole 프로토콜**: `4.mouse,4.1920,4.1080,1.0,1.0;` 같은 opcode + 길이접두 + 콤마구분 포맷. 파싱·생성 비용이 매우 작음.
- **프레임 수가 매우 많음**: 1초에 수백 프레임이 오가므로 JSON 오버헤드가 부담.
- **라이브러리 계약**: guacd가 정한 포맷이라 MOVE가 바꿀 수 없음. 프레임 텍스트를 **그대로** WebSocket으로 뒤 클라이언트 lib(Guacamole.Client)이 해석.

"중간 레이어를 얼마나 두껍게 만들 것인가"의 판단에서 원격은 **가장 얇게** 선택했습니다.
