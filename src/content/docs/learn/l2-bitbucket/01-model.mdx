## 30초 요약

- **`BitbucketMonitorProperties`** — yaml `bitbucket.monitor.*` 바인딩 (enabled + defaultTargetPath)
- **`BitbucketRepo`** — 감시할 저장소 한 개 (serverUrl, projectKey, repoSlug, **PAT**, autoDownload, controller, targetPath)
- **`BitbucketBranch`** — 감지·다운로드된 브랜치 한 개 (status: `DETECTED`/`DOWNLOADING`/`DOWNLOADED`/`FAILED`)

**상태 전이**는 Branch의 `status` 컬럼이 담고, Repo는 "무엇을 어떻게 볼 것인가"라는 설정만.

## 왜 `status` 기본값이 `DOWNLOADING`인가

`@Builder.Default`가 `"DOWNLOADING"`으로 설정되어 있습니다. **이상적으로는 `DETECTED`가 더 적절**:

- `autoDownload=true` 흐름에서도 save 직후 실제 다운로드 시작 전까지 잠깐 잘못된 상태를 띔
- `autoDownload=false` 흐름에서는 `.status("DETECTED")`로 명시적으로 오버라이드하지만 실수 가능성

현재 작동하는 데는 문제 없으나 **기본값 리팩토링 후보**로 표시.

## 세 엔티티의 책임 분리

```
Properties (yaml)            BitbucketRepo (DB)            BitbucketBranch (DB)
───────────────────          ────────────────              ────────────────────
- 전역 enabled                - 이 Repo 감시할지           - 이 브랜치의 상태
- 기본 경로                   - 어디에 저장할지            - 어디에 저장됐는지
                              - autoDownload 여부          - 언제 받았는지
                              - PAT / Controller
```

"**어떻게 볼 것인가**"(Properties + Repo) vs "**무엇이 보였는가**"(Branch)로 **시간 축에서 분리**. 설정은 거의 안 바뀌고, 브랜치 상태는 매 폴링마다 늘어남.

## 직접 해보기

- [ ] `SELECT * FROM portal_bitbucket_repos` → 현재 감시 중 저장소 확인
- [ ] `SELECT status, COUNT(*) FROM portal_bitbucket_branches GROUP BY status` → 상태 분포
- [ ] Repo에 `autoDownload=false`로 새 저장소 등록 → 이후 DETECTED 상태만 생기는지 로그 관찰
