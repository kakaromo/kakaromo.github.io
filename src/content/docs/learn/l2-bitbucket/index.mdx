## 이 모듈에서 배울 것

"외부 Bitbucket Server에서 **FW 코드 브랜치**가 생기면, 어떻게 자동으로 감지하고 ZIP으로 받아 **로컬에 풀어두었다가** T32 L2가 드롭다운으로 쓸 수 있게 하는가?"

아홉 번째 L2. 외부 REST API를 **폴링으로 관찰**하는 첫 L2 (Metadata는 내부 SSH였음).

- Bitbucket Server가 **Webhook 미지원** → `@Scheduled` 5분 폴링으로 대체
- **autoDownload 토글**로 자동/수동 분기 (용량 관리)
- **DETECTED → DOWNLOADING → DOWNLOADED/FAILED** 상태 머신
- **SSE 진행률 중계** — 수동 다운로드 시 ZIP 받는 속도를 프론트가 MB 단위로 표시

## 앞 L2들과의 대조

| 축 | Metadata | Bitbucket |
|---|---|---|
| 폴링 대상 | 내부 Android adb (SSH) | **외부 REST API (HTTPS + Bearer PAT)** |
| 주기 | 5초 (슬롯 상태) + 분 (수집) | **5분 (고정)** |
| 스케줄 제어 | 슬롯별 on/off | **저장소별 enabled** |
| 결과 저장 | 인메모리 + SFTP JSON | **로컬 FS ZIP 해제 폴더** |
| 실패 처리 | 슬롯 상태 + 재시도 | **로그만 + 다음 폴링 대기** |
| SSE | 슬롯 상태 스트림 | **다운로드 진행률 (MB)** |

Bitbucket은 **단순한 폴링 + FS 부작용**. 상태 머신은 작지만 (4 상태) 외부 시스템 통합의 전형적 패턴을 담고 있음.

| 축 | Log Browser | Bitbucket |
|---|---|---|
| 소비자 | 다른 L2 + 사용자 | **주로 T32 L2** + Admin UI |
| 인터페이스 | 8 메서드 공용 인터페이스 | **CRUD REST + SSE** |
| 외부 의존 | SSH · Local 선택 | **Bitbucket Server 고정** |
| 상태 | SSH 세션 캐시 | **DB에 브랜치 이력** |

**재사용 레이어** 성격이 있지만 Log Browser가 추상화 대상이라면 Bitbucket은 **구체적 외부 통합 결과 제공**.

## 왜 Webhook이 아니라 폴링인가

Bitbucket Server(Data Center)는 Webhook을 지원하지만 MOVE 환경에서는 **단방향 방화벽 정책**으로 외부 → 포털 서버 역방향 호출이 막혀 있습니다. 따라서 **MOVE가 주기적으로 밖으로 나가 확인**하는 방식.

trade-off:

- 단점: 5분 지연, 불필요한 API 호출
- 장점: 방화벽 해제 불필요, **인증·오류 처리가 한쪽에서 관리**
- 대안: SSE/gRPC stream처럼 Bitbucket이 열어두는 경우 → 현재 버전은 지원 안 함

## 학습 순서와 예상 시간

| 단계 | 제목 | 성격 | 시간 |
|---|---|---|---|
| 1 | 설정·엔티티 (Repo + Branch) | 코드 투어 | 10분 |
| 2 | 폴링 스케줄 + autoDownload 분기 | 코드 투어 | 12분 |
| 3 | REST API (PAT · 페이지네이션) | 코드 투어 | 12분 |
| 4 | 다운로드 + 압축 해제 | 코드 투어 | 15분 |
| 5 | T32 L2 소비 + 프론트 드롭다운 | 의사코드 | 8분 |

총 약 **57분**. 단계 4가 가장 무거움 — 6 step + Zip Slip 방어 + SSE 스트리밍.

## 새로 등장할 L3 개념

**없음**. `@Scheduled`·`@ConfigurationProperties`·`@Entity` 모두 기존 사전. 새로 만나는 Java API(`HttpClient`, `SSLContext`, `ZipInputStream`)는 도메인 개념이 아니라 본 튜토리얼 안에서 짧게 소개.
