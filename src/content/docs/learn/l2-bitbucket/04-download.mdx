## 30초 요약

`downloadBranch`가 **6 단계**로 진행됩니다.

1. controller 결정 + 경로 조립
2. Branch record 생성 (status=DOWNLOADING)
3. 디렉토리 생성 + ZIP 파일 경로 확정
4. Bitbucket API → ZIP 스트리밍 (1MB마다 SSE)
5. **extractZip** — Zip Slip 방어 + 압축 해제
6. ZIP 삭제 + DB status=DOWNLOADED

에러 시 status=FAILED + errorMessage. 모두 try 안에서 처리되어 **폴링 루프는 안 죽음**.

## Zip Slip 방어

`extractZip`은 다음을 강제합니다.

```java
Path entryPath = targetDir.resolve(entry.getName()).normalize();
if (!entryPath.startsWith(targetDir)) {
    throw new SecurityException("Zip Slip attempt: " + entry.getName());
}
```

**악의적 ZIP이 `../../etc/passwd` 같은 엔트리를 포함하면** 해제 시 타겟 밖으로 탈출 가능. `normalize() + startsWith` 두 단계로 방어.

- `normalize()` — `../` 경로를 실제로 축소
- `startsWith(targetDir)` — 축소 후 여전히 타겟 아래인지 확인

Log Browser L2의 Local 구현 `resolveLocalPath`와 **같은 방어 패턴**. 외부 데이터를 파일 시스템에 반영할 때는 필수.

## 자동/수동 경로의 공통 함수

```java
// 자동 폴링 경로
downloadBranch(repo, branchName, branchId, commitId);   // emitter=null

// 수동 사용자 경로 (단계 5 Controller)
downloadBranch(repo, branchName, branchId, commitId, emitter);  // SSE
```

**같은 함수**에서 `emitter != null` 체크로 분기. 두 경로의 로직 일관성 보장, 수정 시 한 곳만.
