## 30초 요약

`BitbucketApiClient`는 세 가지 API를 호출합니다.

- `listBranches` — `/rest/api/latest/projects/{p}/repos/{r}/branches?start=N&limit=100` (페이지네이션)
- `downloadArchive` — `/archive?format=zip&at={encodedRef}` (InputStream 반환)
- `getCommitTimestamp` — `/commits?until={branchId}&limit=1` (DETECTED fallback)

모두 **Bearer PAT** 인증. Java 표준 `HttpClient` 사용, 자체 서명 인증서를 위해 **SSL 검증 비활성** TrustManager.

## PAT 관리 — 평문 저장의 trade-off

현재 PAT는 DB에 **평문**으로 저장됩니다:

- **장점**: 설정 UI · API 모두 단순. Admin이 PAT 변경 시 바로 반영.
- **단점**: DB 덤프가 유출되면 Bitbucket 접근 토큰도 함께 유출.

개선안:

```java
@Column(nullable = false, length = 500)
@Convert(converter = EncryptedStringConverter.class)
private String pat;
```

JPA `@Convert`로 AES 암호화 Converter를 붙여 투명 암복호화. **기술 부채**로 기록.

## 왜 HttpClient로 충분한가

Spring에는 `RestTemplate`·`WebClient` 같은 상위 추상화가 있지만 이 모듈은 Java 내장 `HttpClient`를 씁니다.

- **호출 수가 적음** — 5분에 수 회
- **bean 주입 없이 단순** — 설정·SSL 커스터마이징을 한 곳에서 제어
- **의존성 감소** — RestTemplate은 곧 deprecated, WebClient는 Reactor 전체

**가장 가벼운 도구로 충분**한 사례. 호출 수가 폭증하면 커넥션 풀링 / 재시도 같은 기능이 필요하고 그때 상위 API로 이행.
