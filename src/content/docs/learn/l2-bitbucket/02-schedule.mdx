## 30초 요약

`BitbucketMonitorService`가 5분마다 `pollAllRepos`를 호출하고, 각 활성 Repo에 대해 `pollRepo`가 돕니다.

1. Bitbucket API로 브랜치 목록 조회
2. DB에 없는 새 브랜치 발견
3. `autoDownload=true` → 즉시 `downloadBranch` (단계 4)
4. `autoDownload=false` → DETECTED만 저장 (사용자 수동 트리거 대기)
5. `repo.lastPolledAt` 업데이트

## 왜 fixedDelay 5분으로 고정인가

- **Bitbucket API는 rate limit이 있음** — 너무 자주 호출하면 throttle
- **브랜치 생성은 빈도가 낮음** — FW 개발 타임라인 기준 수 시간~수 일 간격이 현실
- **지연 허용** — T32 dump 실행 시 브랜치 드롭다운 당장 최신이 아니어도 무방

yaml로 주기를 외부화하지 않은 이유: **변경이 거의 없다**. 필요해지면 `fixedDelayString = "${bitbucket.monitor.poll-interval-ms:300000}"`로 간단히 바꾸면 됨.

## 같은 브랜치의 새 커밋은 왜 감지 못 하는가

현재 판정은 `existsByRepoIdAndBranchName` — **branchName만 확인**. `latestCommitId` 변화는 감지 안 함.

이유:

- **브랜치 단위가 dump의 단위** — 커밋 단위가 아님
- FW가 같은 브랜치에서 여러 번 업데이트되는 경우가 드물고, 있어도 수동 "Redownload" 버튼으로 해결
- **latestCommitId 비교를 추가하면** 매 폴링마다 모든 브랜치 update 체크 필요 → API 비용 증가

현재 설계는 **트래픽과 단순성 trade-off**에서 단순성 선택. 운영 중 요구가 커지면 개선.

## 직접 해보기

- [ ] Bitbucket에 새 브랜치를 push한 뒤 최대 5분 이내에 DB에 행이 생기는지 확인
- [ ] `autoDownload=false`인 Repo에서 DETECTED 상태만 생기는지 Admin UI에서 관찰
- [ ] `bitbucket.monitor.enabled=false`로 재시작 → 폴링 로그가 완전히 멈추는지 검증
