## 왜 sortOrder가 필요한가

대안:

- **name 알파벳 정렬** — 사용자 의도 반영 못 함 ("T10, T2, T3" 순서)
- **생성 순서** — 오래된 그룹이 위로, 새 그룹 아래 — 역시 인위적
- **프론트에서만 정렬** — 사용자마다 순서 다르면 혼란
- **sortOrder int (현재)** — 사용자가 명시적 배치 가능

**Samsung 운영상 "메인 디바이스 그룹은 맨 위"** 같은 관습이 있어 수동 정렬 필요. 숫자 할당이 단순하고 유연.

## sortOrder 재정렬 방식

드래그-드롭으로 UI에서 그룹 순서 바꿀 때:

```
기존: A(0), B(1), C(2), D(3)
드래그: B를 맨 아래로
새 배치: A(0), C(1), D(2), B(3)
```

**클라이언트가 새 배치 계산 후 모두 PUT**:

```
PUT /api/admin/server-groups/1 { sortOrder: 0 }  // A
PUT /api/admin/server-groups/3 { sortOrder: 1 }  // C
PUT /api/admin/server-groups/4 { sortOrder: 2 }  // D
PUT /api/admin/server-groups/2 { sortOrder: 3 }  // B
```

**4개 PUT 일괄** — 트랜잭션으로 묶지 않아도 UI 최종 상태가 동일. "부분 실패 시 시각적 혼란" 은 있지만 데이터 일관성 유지.

대안: `sortOrder` 간격 띄우기 (0, 10, 20, 30). 삽입 시 기존 값 두 번 사이에 (예: 15). **간격 관리 복잡** → MOVE는 단순하게 전부 재할당 선택.

## is_visible — 소프트 숨김

PortalServer에만 있는 플래그. 서버 삭제 vs 숨김 차이:

| 작업 | DB | 다른 L2 참조 |
|---|---|---|
| 삭제 (DELETE) | row 제거 | **FK 깨짐** (HeadConnection이 server_id 보유한 경우) |
| 숨김 (visible=false) | row 유지 | 참조 유효 · UI에서만 안 보임 |

**왜 "숨기기"가 기본인가**:

- 실험 중 잠시 비활성화
- 실수 복구 쉬움 (다시 visible=true)
- History · Log Browser 탐색 시 "과거 데이터 주인" 로컬 식별

완전 삭제는 별도 "영구 삭제" 버튼 — 확인 다이얼로그 + FK 정리 안내.

## UI 카드 뷰 — 그룹별 묶음

프론트 Admin 서버 탭 구조:

```
┌─ T10 Group (sortOrder 0) ─────────────────────┐
│  ┌─ T10-1 ┐  ┌─ T10-2 ┐  ┌─ T10-3 ┐          │
│  │ SSH+   │  │ SSH    │  │ SSH+VNC│          │
│  │ guacd:X│  │ global │  │ guacd:Y│          │
│  └────────┘  └────────┘  └────────┘          │
└────────────────────────────────────────────────┘

┌─ HEAD Cluster (sortOrder 1) ──────────────────┐
│  ┌─ HEAD-01 ┐  ┌─ HEAD-02 ┐                  │
│  ...                                          │
└────────────────────────────────────────────────┘

┌─ 그룹 없음 (기본) ─────────────────────────────┐
│  ┌─ dev-server ┐                              │
│  ...                                          │
└────────────────────────────────────────────────┘
```

**렌더 로직**:

```pseudo
const groups = await fetch('/api/admin/server-groups')  // sorted by sortOrder
const servers = await fetch('/api/admin/servers')        // with is_visible=true filter
for (group of groups + [{id: null, name: '그룹 없음'}]) {
  const groupServers = servers.filter(s => s.serverGroupId === group.id)
  renderCard(group, groupServers)
}
```

**두 쿼리, 프론트 join**. 수십~수백 row 수준에서 성능 문제 없음.
