## 30초 요약

REST API 4종 + 1동기:

- `GET /slots` / `POST /slots/assign` / `POST /slots/unassign`
- `GET /tc` / `POST /tc/assign` / `POST /tc/unassign`
- `POST /tc/sync` — testcaseIds 변경 시 CSV를 통째 갱신

핵심 **양립성 규칙**:

- 슬롯 assign → 기존 `tcPreCommandIds` **자동 null**
- TC assign → 기존 `preCommand` **자동 null**
- unassign 시 나머지 필드가 모두 비면 **행 자체 삭제**

## 양립성이 서버·DB 양쪽에서 강제되는 이유

직관적으로는 **프론트에서 조건 분기**만 하면 될 것 같습니다. MOVE는 **서버·DB 양쪽**에서 강제합니다.

- **멀티 탭**: 같은 슬롯을 두 탭에서 동시에 편집할 수 있음. 프론트 만 체크하면 race.
- **다른 클라이언트**: API는 Admin 외에도 스크립트·CI에서 호출 가능. 프론트만 신뢰 못 함.
- **데이터 정합성**: 슬롯과 TC가 모두 세팅된 행이 남으면 단계 4의 우선순위 판정에서 TC가 언제나 이기는 무의미한 상태.

**Transaction + 자동 해제**를 API 기본 정책으로 삼아 "UI가 이 규칙을 의식하지 않아도 되게" 만든 설계.

## 삭제 조건 — 빈 행 금지

`unassign`은 두 가지 결과가 있습니다:

- **행 유지**: 반대쪽이 여전히 세팅되어 있음
- **행 삭제**: 슬롯·TC 모두 비어있음

이 정리 덕에 `SELECT * FROM portal_slot_pre_commands`에는 **의미 있는 행만 남음**. 디버깅·통계에서 가치.
