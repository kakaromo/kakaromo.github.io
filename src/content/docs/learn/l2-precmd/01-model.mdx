## 30초 요약

두 엔티티로 나눕니다.

- **`PreCommand`**: 재사용 가능한 **명령 템플릿** — 이름 + 명령어 배열(JSON). 예: `["adb push tiotest /dev", "adb shell chmod +x /dev/tiotest"]`.
- **`SlotPreCommand`**: `setLocation`(UK)을 키로 **슬롯별 등록** — 슬롯 Pre-Command 한 개 + TC별 Pre-Command ID 배열(CSV).

**핵심 결정**: 슬롯과 TC 할당을 **한 테이블에 합쳤습니다**. 별도 테이블로 분리하지 않은 이유는 "한 슬롯 = 한 행" 구조로 양립성 체크를 DB가 보장하기 위함.

## 왜 tcPreCommandIds가 CSV인가

직관적으로는 **테이블을 분리**하는 것이 "정석"입니다.

```
portal_slot_pre_command_tcs (
  slot_pre_command_id  FK,
  tc_position          INT,
  pre_command_id       FK,
  PRIMARY KEY (slot_pre_command_id, tc_position)
)
```

MOVE는 하지 않았습니다:

- **통째 교체가 기본** — TC 목록이 바뀌면 array를 새로 만들어 저장. 부분 UPDATE가 흔하지 않음.
- **조회가 단순** — 한 setLocation에 대해 `SELECT tcPreCommandIds`면 끝.
- **테이블 수 절약** — Spring Data JPA의 EAGER join이 더 늘어나는 부담.
- **position이 index** — 관계형 PK가 `(parent, position)` 합성 키가 되는데, 이걸 ORM으로 다루면 오히려 복잡.

단점은 **DB 레벨에서 integrity 보장이 약해짐**(잘못된 ID를 넣어도 FK 체크 없음). 서비스 계층에서 `preCommandRepo.existsById`로 커버.

## 테이블 간 관계 요약

```
portal_pre_commands (템플릿)
   ↑
   │ ManyToOne (preCommand)
   │
portal_slot_pre_commands (setLocation UK)
   ├── preCommand_id   → portal_pre_commands.id   (슬롯 Pre-Command)
   └── tcPreCommandIds = "0,3,0,5,0"              (TC Pre-Command CSV)
```

슬롯 한 개에 최대 두 종류(슬롯 + TC 여러 개)가 등록 가능하지만 **우선순위가 TC > Slot**이므로 현실적으로는 **한 쪽만 세팅**하는 것이 일반적. API도 "한 쪽 등록 시 반대쪽을 자동 해제"하도록 만듭니다 — 단계 2에서 상세.
