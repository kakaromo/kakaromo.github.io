## 30초 요약

`tryExecuteTcPreCommand`는 이 모듈에서 가장 복잡한 함수입니다. **TC 여러 개의 진행 상태**를 보고 **지금 실행하려는 TC가 어느 position인지** 결정합니다.

1. setLocation 파싱 → `SlotInfomation` DB 조회 (testcaseIds + testcaseStatus 확보)
2. 선행 슬래시 제거 → 3개 배열 정렬 (`tcIds` / `statuses` / `preCommandIdArr`)
3. **첫 번째 미완료** status 찾기 → `targetPosition`
4. 해당 position의 `preCommandId` 유효성 검증
5. **testToolName 매칭** — 실제 돌고 있는 TC가 맞는지 교차 검증
6. 중복 방지 Set guard + submit

## 복잡한 이유: TC 여러 개가 순차 실행됨

한 슬롯이 한 테스트 실행 동안 **여러 TC를 순차로 돎**니다. DB는 그 진행을 이렇게 저장:

```
testcaseIds     = "/42/43/44/45"    (5개 TC)
testcaseStatus  = "/27/27/0/0/0"    (처음 둘은 PASS, 나머지 NOTSTART)
tcPreCommandIds = "0,3,0,5,0"       (position 2, 4에만 TC Pre-Command)
```

첫 미완료 status는 `statuses[2] = 0` (NOTSTART). 그러니 **"지금 init 진입은 TC position=2 직전"** 으로 해석되고, `tcPreCommandIds[2] = 0` → **등록 없음, 무시**. 다음 init이 `position=3`에서 올 때 `tcPreCommandIds[3] = 5`면 → PreCommand id 5 실행.

## 27가지 완료 상태 — 왜 이렇게 많은가

TC status는 0~60 정도의 정수 코드. 그중 **완료**로 분류되는 것만 27가지입니다.

- **PASS 계열 8개**: 정상 완료·경고 통과 등
- **FAIL 계열 11개**: 일반 fail·timeout·boot fail·critical…
- **종결 계열 8개**: STOP·DISCONNECT·EMERGENCY_STOP…

이 분류 기준은 **프론트 UI의 `tcStatusLabel`과 동일**. 두 곳이 서로 달라지면 "완료됐는데 Pre-Command가 또 실행됨" 같은 버그가 생기므로 **코드 코멘트로 동기화 규칙을 못 박음**. Set으로 코드를 상수화해 변경이 쉽고 검증 테스트도 가능.

## 왜 testToolName 교차 검증을 추가했는가

**position만으로 판정하면 race가 일어납니다.**

실제 발견된 시나리오:
1. TR이 바뀌어 testcaseIds가 `[42, 43]`에서 `[99, 42, 43]`으로 바뀜.
2. 하지만 Head가 보낸 슬롯 데이터는 아직 이전 `testToolName=fio`(position=0, tcId=42).
3. DB와 Head 데이터가 **잠깐 어긋난 시점**에 init이 들어오면 position=0이 tcId=99로 잘못 매칭.

`testToolName == TC.name` 체크로 **"이 슬롯이 진짜 그 TC를 돌고 있는가"** 를 한 번 더 확인. false면 skip — 다음 상태 변화에서 다시 평가.

## 직접 해보기

- [ ] DB에서 `testcaseIds = "/42/43"`, `testcaseStatus = "/0/0"`인 슬롯을 만들고 tc position 0에만 PreCommand 등록
- [ ] Head를 시뮬레이션해 testToolName이 TC 42의 이름과 일치하는 init 상태 보내기 → position=0이 실행되는지 로그 확인
- [ ] testToolName을 일부러 다른 값으로 바꿔 → 실행되지 않는지 확인 (race 방어)
