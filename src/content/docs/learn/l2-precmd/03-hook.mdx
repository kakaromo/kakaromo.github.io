## 30초 요약

**`PreCommandAutoExecutor`는 Spring 빈 하나**입니다. `HeadSlotStateStore.updateSlots`가 슬롯마다 `onSlotStateChanged`를 부르고, 이 메서드가 **4가지 분기**를 결정합니다.

1. setLocation 없으면 **무시**
2. testState가 `init` 아니면 **실행 추적 초기화** + return
3. SlotPreCommand DB 조회
4. **TC 우선** → 없으면 슬롯 Pre-Command → 중복 방지 Set로 guard

실행 자체는 **4 스레드 풀**에서 비동기로 SSH 쏨 (단계 5).

## "이벤트 훅" 패턴이란

MOVE가 이 모듈에서 처음으로 쓰는 설계 패턴입니다. 요약하면:

1. **변화의 출처**는 다른 모듈(여기서는 HeadSlotStateStore).
2. **훅을 주입받은 부수 로직**이 그 변화를 감지해 움직임.
3. **출처는 훅의 존재를 거의 모른 채** 한 줄 호출만 함.

구현 포인트는 StateStore 쪽 한 줄:

```java
// HeadSlotStateStore.updateSlots (슬롯 L2에서 본 그것)
preCommandAutoExecutor.onSlotStateChanged(source, oldData, data);
```

이 한 줄이 **두 모듈의 유일한 접점**. Pre-Command 모듈이 빠지거나 에러를 던져도 StateStore는 try/catch로 감싸 보호(슬롯 L2 05-state-store.mdx 참고).

## 왜 HeadSlotStateStore에서 직접 실행하지 않는가

"StateStore.updateSlots 안에 바로 SSH 코드를 넣으면 안 되나?" 안 됩니다:

- **StateStore는 상태 저장 전용** — 추가 책임이 늘어나면 "슬롯 상태 관리"와 "부수 동작"이 섞여 읽기 어려워짐
- **단일 책임 원칙** — 한 클래스의 역할은 하나
- **테스트 분리** — StateStore 단위 테스트가 SSH까지 모킹하지 않아도 됨
- **확장성** — 나중에 또 다른 훅(예: 이벤트 로깅, Slack 알림)이 필요하면 같은 방식으로 붙일 수 있음

**"호출 한 줄 + 훅 주입"** 은 확장 포인트를 명시적으로 드러내는 패턴입니다.
