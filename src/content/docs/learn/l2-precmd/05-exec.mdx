## 30초 요약

`PreCommandService.executeSync`가 실제 실행을 담당합니다.

1. PreCommand DB 조회 → JSON 배열 파싱
2. StateStore에서 현재 슬롯 데이터 조회 (usbId + setLocation 확보)
3. setLocation에서 VM name 추출 (`T3-0` → `T3`)
4. 각 명령에 **`adb -s {usbId}`** 자동 삽입 → VM으로 SSH 전송
5. 중간에 exitCode != 0이면 **즉시 중단** (명령 순서가 의미 있음)

명령 한 개의 실패가 뒤 명령을 막는 이유: `adb push` 실패 시 이어지는 `chmod +x`는 의미 없음.

## 왜 executeSync와 execute를 분리했는가

`PreCommandService`에는 두 개의 실행 API가 있습니다.

- **`execute(..., SseEmitter emitter)`** — 프론트 수동 실행용. 사용자가 다이얼로그에서 "즉시 실행" 버튼 누르면 호출. 각 명령 결과를 SSE로 라인 단위 중계.
- **`executeSync(..., List<Integer> slotNumbers)`** — 자동 실행용. AutoExecutor가 호출. **SSE 없음**, 조용히 실행·실패.

두 개를 한 메서드로 합치면 `emitter == null` 분기가 계속 나와 읽기 어려워짐. **용도별로 분리하되 `resolveCommand` 같은 공통 로직은 private 헬퍼로 재사용**.

## adb 명령만 변환하는 이유

`resolveCommand`는 `adb`로 시작하는 명령에만 `-s {usbId}`를 넣습니다. 다른 명령(예: `echo`, `mkdir`)은 그대로. 이유:

- **adb는 디바이스 식별이 필수** — 한 VM에 여러 디바이스가 연결되면 `-s` 없이는 어느 디바이스를 대상으로 할지 불명확.
- **다른 셸 명령은 VM 자체에서 실행** — 디바이스 식별과 무관.
- **템플릿 작성자의 실수 허용** — `-s`를 안 써도 자동으로 주입되므로 템플릿이 간단해짐.

## Pre-Command의 전체 여정 되돌아보기

5단계를 관통하는 데이터 경로:

```
[사용자 편집 API]         ───► portal_slot_pre_commands (DB)
                                         │
                                         │ findBySetLocation
                                         ▼
[Head TCP 수신]     ───► StateStore.updateSlots
                                         │
                                         │ onSlotStateChanged (훅)
                                         ▼
                          PreCommandAutoExecutor
                                         │
                                         │ init 감지 + TC/Slot 우선순위
                                         ▼
                          PreCommandService.executeSync
                                         │
                                         │ resolveCommand + SSH
                                         ▼
                                  VM (adb -s usbId ...)
```

**실시간 스트림 없이도 한 모듈이 완성**되는 모범. 비교 문서의 체크리스트가 실시간 L2 외에도 유효함을 검증.

## 여섯 L2의 실시간 vs 이벤트 훅 대조

| L2 | 프론트 실시간 | 백엔드 자동 실행 |
|---|---|---|
| 슬롯 | SSE 상시 | — |
| 벤치마크 | SSE (job) | — |
| 원격 | WebSocket | — |
| Metadata | HTTP 폴링 | @Scheduled 자동 |
| T32 Dump | POST+SSE (단발) | — |
| **Pre-Command** | **없음 (CRUD)** | **이벤트 훅 (StateStore → onChange)** |

Pre-Command는 **"프론트 실시간이 없지만 백엔드 자동 실행이 있음"** 이라는 새로운 조합. Metadata의 `@Scheduled`와 다른 점은 트리거가 **도메인 이벤트(init 진입)** 라는 것.

## 직접 해보기

- [ ] 템플릿에 `["echo hello", "adb shell pwd"]`를 넣고 → echo는 그대로, adb는 `-s` 주입되는지 로그 확인
- [ ] `adb push missing-file` 같이 실패하는 명령을 첫 줄에 넣어 → 두 번째 명령이 실행 안 되는지
- [ ] DB의 commands 필드를 일부러 망가뜨린 뒤 훅 실행 → 조용히 실패(예외 삼켜짐)하는지 확인
