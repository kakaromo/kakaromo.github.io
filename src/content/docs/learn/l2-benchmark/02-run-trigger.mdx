## 30초 요약

사용자가 `AgentBenchmarkForm`에서 디바이스·도구·파라미터를 고르고 **Run**을 누르면, 프론트는 `runBenchmark()` API를 호출합니다. 이 호출은 **잠깐만** 서버와 통신하고 **jobId 문자열**만 받아옵니다. 이후 실제 작업은 비동기로 계속되고, 프론트는 jobId로 진행률 SSE에 따로 구독합니다.

## 트리거 흐름 (의사코드)

```svelte
<script lang="ts">
  import { runBenchmark } from '$lib/api/agent';

  let selectedDevices = $state<string[]>([]);
  let tool = $state<'FIO' | 'IOZONE' | 'TIOTEST'>('FIO');
  let params = $state<Record<string, string>>({
    bs: '4k',
    rw: 'randread',
    size: '1G',
  });

  async function handleRun() {
    const res = await runBenchmark(serverId, {
      deviceIds: selectedDevices,
      tool,
      params,
      jobName: '',
      busyPolicy: 'reject',
    });
    onJobStarted(res.jobId); // 플로팅 Job Card에 전달 → SSE 구독 시작
  }
</script>
```

핵심은 "**Run 버튼 핸들러가 작업이 끝날 때까지 기다리지 않는다**"는 것. 벤치마크는 수초~수시간까지 걸릴 수 있어 브라우저 요청을 붙들어두는 모델은 불가능합니다. 대신 **트리거와 관찰을 분리**합니다.

## 왜 POST + SSE 분리인가

한 번의 HTTP 요청으로 벤치마크 진행률을 스트림하는 구조도 가능합니다. 하지만 MOVE는 **두 단계로 쪼갭니다**.

- **POST /run**: "시작했다. 이 jobId로 추적하라" — 즉시 응답
- **GET /progress (SSE)**: "이 jobId의 진행률을 구독한다" — 별도 연결

이 분리의 이점:

1. **네트워크 단절에 강함** — SSE가 끊겨도 jobId는 살아있으므로 재연결만 하면 관찰 재개.
2. **UI 상태 관리가 단순** — "실행 중인 job"은 jobId의 집합, SSE는 그 집합을 구독하는 도구.
3. **여러 화면에서 동시 관찰** — 관리자 대시보드와 당사자 화면이 같은 jobId를 각자 SSE로 구독 가능.

슬롯 L2에서는 Head가 스스로 스트림을 시작하는 반대 방향이라 이 분리가 필요 없었습니다. **트리거 주체가 누구인지**가 설계를 가른 예시입니다.

## busyPolicy — reject vs queue

`busyPolicy` 필드는 Agent 서버에 보낼 힌트입니다.

- `"reject"` — 이미 다른 job이 돌고 있으면 즉시 거부 (기본)
- `"queue"` — 뒤에 큐잉 (지원하는 tool만)

UI에서 보통 `reject`를 씁니다 — 사용자가 명시적으로 취소·재시작하도록.
