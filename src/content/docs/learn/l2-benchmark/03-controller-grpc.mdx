## 30초 요약

`AgentController.runBenchmark()`는 세 일을 순서대로 합니다.

1. **gRPC 클라이언트 확보** — `getClient(serverId)` 으로 서버 id에 맞는 `AgentGrpcClient`를 캐시에서 꺼냄
2. **RunBenchmark RPC 호출** — Proto 빌더로 요청 만들고 blocking stub 호출
3. **JobExecution 저장** — 이력 영속화 (실패해도 무시, best effort)

그리고 `{ jobId }`를 반환합니다. 실제 디바이스 실행은 Go Agent 안에서 별도 스레드로 돌아갑니다.

## 왜 엔드포인트마다 getClient 하는가

각 엔드포인트 첫 줄이 `getClient(serverId)`로 시작하는 게 번거로울 수 있습니다. 왜 공통 인터셉터로 빼지 않나?

- **명시적 > 암묵적**: serverId가 어느 필드에서 오는지 엔드포인트마다 다를 수 있음 (query, body, path). 인터셉터로 숨기면 오히려 읽기 어려움.
- **테스트 용이성**: 단위 테스트에서 `AgentConnectionManager`를 mock해 특정 클라이언트를 주입하기 쉬움.
- **오류 경로 명확**: 서버가 연결 불가일 때 에러가 어디서 발생하는지 스택 추적에 그대로 보임.

"한 줄 반복"은 의도된 신호입니다.
