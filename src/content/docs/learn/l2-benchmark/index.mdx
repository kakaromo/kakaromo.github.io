## 이 모듈에서 배울 것

"사용자가 벤치마크 Run 버튼을 누르면, **실제 Android 디바이스**에서 fio/iozone이 돌고, 그 진행률이 브라우저의 Job Card에 실시간으로 뜨기까지 어떤 계층을 거치는가?"

이 모듈은 [슬롯 모니터링 L2](/learn/l2-slots/)와 같은 **7단계 여정 템플릿**을 재사용하되, 스케일이 다른 프로토콜(gRPC 서버 스트림)과 비동기 job 모델을 다룹니다.

## 슬롯과 무엇이 다른가

슬롯 모니터링은 "**서버가 일방적으로 밀어주는** 상태"를 보여줬습니다. 벤치마크는 반대로 "**사용자가 시작한 작업**이 진행되는 과정"을 보여줍니다. 이 차이가 설계에 반영됩니다.

| 축 | 슬롯 모니터링 | 벤치마크 |
|---|---|---|
| 트리거 | Head가 알아서 | 사용자 Run 버튼 |
| 프로토콜 | TCP (커스텀) | gRPC 서버 스트림 |
| 지속 시간 | 항상 연결 | job 하나 단위 (수초~수시간) |
| 식별자 | source 이름 | jobId (UUID) |
| 저장소 | 인메모리 StateStore | DB JobExecution (영속) |
| 완료 통지 | 없음 (지속적) | `complete`/`error` 이벤트로 종료 |

두 모듈이 동일한 SSE 프론트 패턴을 쓰면서도 내부 동작은 꽤 다릅니다. 템플릿이 "어디까지 재사용되고 어디서 갈라지는가"를 관찰하면 다른 L2를 만들 때 판단 기준이 됩니다.

## 학습 순서와 예상 시간

| 단계 | 제목 | 성격 | 시간 |
|---|---|---|---|
| 1 | 데이터 모델 (AgentServer + JobExecution) | 읽기 | 8분 |
| 2 | Run 트리거 (프론트 Form → API) | 코드 투어 | 10분 |
| 3 | Spring 컨트롤러 + gRPC 클라이언트 | 코드 투어 | 15분 |
| 4 | 진행률 중계 (gRPC stream → SSE) | 코드 투어 + 시퀀스 | 15분 |
| 5 | 프론트 Job Card 수신 | 코드 투어 | 10분 |

총 약 **1시간**. 슬롯 L2를 읽은 뒤라면 반복 개념(@RestController, SseEmitter, EventSource)은 빠르게 훑고 넘어가도 됩니다.
