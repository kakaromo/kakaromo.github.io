## 30초 요약

Agent 모듈은 **두 종류의 DB 행**으로 관리됩니다.

- **`AgentServer`** — 어떤 Go Agent 서버에 붙을지 (`host`/`port`). 관리자가 등록.
- **`JobExecution`** — 벤치마크·시나리오·trace 각각의 **실행 이력 한 건**. `jobId` UUID로 식별, 상태(running/completed/failed) 추적.

이 외에 `BenchmarkPreset`(파라미터 프리셋), `ScenarioTemplate`(복합 워크플로) 등이 있지만 이 튜토리얼 범위 밖입니다.

## 왜 두 엔티티를 나누는가

- **AgentServer**는 **정적 설정** — 관리자가 서버를 등록하면 거의 바뀌지 않음.
- **JobExecution**은 **동적 이력** — 실행할 때마다 새 행이 쌓이고 끝날 때 상태가 업데이트됨.

이 분리는 [슬롯 L2 단계 2](/learn/l2-slots/02-db-row/)에서 `HeadConnection`(설정) vs `HeadSlotStateStore`(런타임) 분리와 **같은 원칙**입니다. 다만 Agent는 job 이력을 재시작 후에도 조회할 수 있어야 해서 **DB에 영속**합니다.

## 운영 데이터의 형태

실제 DB를 들여다보면 이런 모양입니다.

```sql
-- Agent 서버 2개 등록
SELECT id, name, host, port, enabled FROM portal_agent_servers;
-- 1 | dev-agent-1  | 192.168.0.10  | 50051 | 1
-- 2 | staging-agent| 192.168.0.20  | 50051 | 1

-- 최근 job 이력 3개
SELECT job_id, type, tool, state, started_at FROM portal_job_executions
 ORDER BY started_at DESC LIMIT 3;
-- abc-1234 | benchmark | FIO     | completed | 2026-04-19 10:00:00
-- def-5678 | trace     | null    | running   | 2026-04-19 10:15:00
-- ghi-9012 | scenario  | null    | failed    | 2026-04-19 10:20:00
```

## 직접 해보기

- [ ] Admin 페이지 또는 DB에서 `portal_agent_servers`에 로컬 Go Agent(`localhost:50051`) 한 행 추가
- [ ] 벤치마크 한 번 실행 후 `portal_job_executions`에 새 행이 생기는지 확인
- [ ] 실행 중인 job의 `state` 필드가 `running` → `completed`로 변화하는 순간을 쿼리로 관찰
