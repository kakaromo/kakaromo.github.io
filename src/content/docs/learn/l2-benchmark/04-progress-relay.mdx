## 30초 요약

이 단계가 Agent 모듈의 **하이라이트**입니다. Spring은 gRPC 서버 스트림을 받는 **StreamObserver**의 콜백 안에서 **SseEmitter**를 써서 프론트에 이벤트를 보냅니다. 두 비동기 프로토콜이 한 함수 안에서 만나는 구조.

프론트 관점에서는 슬롯 L2의 SSE 패턴과 **동일**합니다. 차이는 Spring 뒤편에 gRPC 상대가 추가로 붙어 있다는 것.

## <ConceptPopover client:visible term="StreamObserver" concept="spring/stream-observer"><p>gRPC 비동기 콜백 인터페이스. onNext(각 메시지) / onError(에러) / onCompleted(정상 종료) 3개 메서드. 서버 스트림·클라이언트 스트림·양방향 스트림 모두에서 사용됩니다.</p></ConceptPopover> — 세 콜백의 계약

어떤 스트리밍 패턴이든 StreamObserver는 동일한 규칙을 따릅니다.

- **onNext**는 0번 이상 호출. 메시지 하나마다 한 번.
- **정확히 한 번**의 종료 콜백 — onError **또는** onCompleted. 둘 다 불리지 않음.
- onError / onCompleted가 불린 뒤 **onNext는 더 이상 호출되지 않음**.

이 계약 덕에 MOVE는 단일 `completed` 플래그만으로 상태를 관리할 수 있습니다. 계약을 위반한 서버와 통신한다면 훨씬 복잡한 상태 머신이 필요할 것.

## JobExecution 상태 머신

이 함수가 DB의 `state` 필드를 3번 바꿀 수 있는 주체입니다.

```
(단계 3에서 저장) running
    ├── onCompleted → completed
    └── onError     → failed
```

`running` → `running`은 없고, 한번 completed/failed가 되면 **더 바뀌지 않습니다**. 이 단방향 전이가 동시성 안전성을 크게 단순화합니다.

## 왜 SseEmitter timeout이 0L인가

슬롯 SSE와 같은 이유 + 하나 더:

- **작업이 언제 끝날지 모름** — 벤치마크는 1분일 수도, 8시간일 수도 있음
- **유한 timeout 설정 시 AsyncRequestTimeoutException WARN 쌓임** (슬롯과 동일)

프론트의 EventSource가 네트워크 단절 시 자동 재연결하므로, timeout은 서버 측에서 걱정하지 않아도 됩니다.
