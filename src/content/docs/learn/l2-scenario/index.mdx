## 30초 요약

Agent Scenario Canvas는 **`@xyflow/svelte`** 기반 비주얼 DAG 편집기:

1. 사용자가 8 step 타입(benchmark/iotest/shell/cleanup/sleep/trace_start·stop/app_macro) + 2 제어(Loop/Condition)을 **드래그**
2. 노드 사이를 **edge로 연결** → DAG 구성
3. **Save** → `portal_scenario_templates` 테이블에 stepsJson/loopsJson 저장
4. **Run** → `serializer.canvasToProto`가 DAG를 Proto 메시지로 변환 (Y정렬 또는 위상 정렬) → gRPC `RunScenario`
5. Go Agent가 steps를 실행, **JobProgress** 스트림으로 per-step 상태 되돌림

**Orchestration 축 개척** — 단일 작업이 아닌 **여러 step + 제어 흐름**을 엮어 실행. 이전 15 L2 중 가장 가까운 것은 T32 Stepper(순차 4단계 고정)지만 **시나리오는 사용자 구성**. 다음 Orchestration L2 후보가 등장하면 축으로 확정.

## 왜 "Orchestration 축"인가

- 단일 작업이 아님 — 여러 step을 사용자가 **구성 편집**
- **제어 흐름**이 본질 — Loop/Condition 노드가 실행 경로 변경
- **DAG가 IR** — 위치가 아닌 연결이 의미
- 실행은 다른 엔진(Go Agent)에 위임 — 자체 실행 없음

**다른 축과의 구분**:

| 축 | Scenario Canvas와의 차이 |
|---|---|
| 기능 축 | 단일 도메인에 집중 — Scenario는 여러 step 혼합 |
| 크로스-컷 | 모든 요청/데이터 가로지름 — Scenario는 사용자가 선택한 몇 step만 |
| 엔진 축 | 단방향 파이프라인 + 입력→출력 — Scenario는 편집→저장→실행 세 단계 |

**"사용자가 여러 기능을 엮어 흐름을 만드는"** 이 Orchestration 축의 정의.
