## "왜 stepsJson + loopsJson 이원인가"

한 컬럼에 전부 담을 수 있지만 분리:

| 축 | 단일 JSON | 이원 JSON |
|---|---|---|
| 스키마 | 1 컬럼 | 2 컬럼 |
| 쿼리 | 전체 로드 | loops만 읽기 가능 |
| 직렬화 | 한 번 | 두 번 |
| 복원 | `JSON.parse` 한 번 | 두 번 파싱 후 merge |

**트레이드오프**: 쓰기/읽기 효율 vs 전체 일관성. MOVE 선택: **loopsJson 미리보기 use case가 있어 분리**. 기본 SELECT도 2 컬럼만 읽으면 되니 비용 미미.

**같은 패턴**: Bitbucket Branch 정보가 entity 필드 여러 개 / T32Config가 단일 JSON / ScenarioTemplate은 이원. **데이터의 구조와 접근 패턴**에 따라 결정.

## 프론트 타입 ↔ DB 매핑

```
┌─ Frontend ──────────┐      ┌─ DB ──────────────────┐
│ ScenarioNode[]      │──┐   │ portal_scenario_...    │
│   (step/cond/loop)  │  ├─→ │   stepsJson  (nodes +  │
│ Edge[]              │  │   │              edges)    │
│ execOrder, type…    │──┘   │                        │
│                     │      │   loopsJson  (loop     │
│ LoopGroup nodes     │────→ │              groups)   │
│                     │      │   repeatCount          │
└─────────────────────┘      └───────────────────────┘
```

**편집 vs 실행 관점 분리**:
- 편집: 노드/에지/레이아웃(위치 x,y) 모두 보존 → stepsJson
- 실행: 노드 순서/타입/파라미터만 — canvasToProto가 새 구조체 생성

**DB는 편집 관점만** 저장, 실행용 Proto는 매번 변환. DB에 실행 결과를 영속할 계획 없음 — JobProgress는 Agent 쪽에 저장.

## 직접 해보기

- [ ] 한 시나리오 save 후 DB에서 `stepsJson` 컬럼 raw JSON 확인 → nodes 배열 + edges 배열 구조
- [ ] Loop 노드 1개 포함해 save → `loopsJson`에 그 loop만 있는지, `stepsJson`에도 type=loopGroup 노드가 있는지 중복 확인
- [ ] `repeatCount=3`로 저장 → 실행 시 시나리오가 3회 실행되고 각 사이클이 별도 execOrder를 받는지
