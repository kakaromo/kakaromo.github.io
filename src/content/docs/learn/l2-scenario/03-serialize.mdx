## 30초 요약

DAG → 선형 step sequence 변환은 **조건 분기 유무로 두 갈래**:

- **조건 없음** → `Y좌표 정렬` — 사용자가 위에서 아래로 배치한 직관 유지
- **조건 있음** → `위상 정렬(Kahn 알고리즘)` — 의존성 그래프로 정확한 순서

**왜 Y정렬이 기본인가**: 대부분 시나리오는 순차 — Y좌표만 봐도 의도가 맞음. 복잡한 DAG에서만 위상 정렬 필요.

## "왜 Y좌표 정렬을 선호하나"

기본값은 Y정렬 — 이유:

1. **사용자 의도 반영** — 위에서 아래로 배치 = 순차 실행 기대
2. **편집 중 예측 가능** — 노드를 어디 놓아야 순서가 어떻게 될지 눈으로 판단
3. **단순함** — O(n log n) vs O(n+e)
4. **실수 허용** — 엣지를 잘못 연결해도 Y 좌표는 유지

위상 정렬은 **분기가 있을 때만 필요** — edges가 "이 노드는 이것 다음" 을 명시하지 않으면 실행 순서 불명. 분기 없는 시나리오는 edges가 사실상 시각적 장식.

## 이중 전략의 UX

사용자 경험 관점:

| 상황 | 동작 |
|---|---|
| 5 benchmark step 일렬 | Y정렬 → 위치 조정으로 순서 변경 |
| 같은 위에 condition 하나 추가 | **갑자기 위상 정렬로 전환** → edges 주의 필요 |
| 사용자가 edges 잘못 연결 | Y정렬이면 자동 복원, 위상 정렬이면 실행 오류 |

**전환이 암묵적**이라 디버깅 포인트 — condition 추가 시 사용자가 edges를 점검해야 함. UI에서 힌트로 완화 가능(현재 없음).

## 특수: hasBranching 플래그

canvasToProto 반환에 `hasBranching: boolean`. Agent에 전달돼:

- `false` → Agent가 `steps[0], steps[1], …` 순차 실행
- `true` → Agent가 `edges` 따라 실행 (multi-entry/multi-exit 지원)

**Agent 코드 분기**: 단순 시나리오에 DAG 오버헤드 없음. 복잡 시나리오에만 엣지 처리. **"일반 경로는 가볍게, 특수 경로는 명시"**.

## Condition 분기 직렬화 세부

ConditionalBranch proto 메시지:

```
ConditionalBranch {
  source: "metric" | "shell"
  metric_key: string
  operator: string
  threshold: double
  threshold_string: string  // 문자열 비교용
  shell_command: string
  extract_pattern: string
  true_branch_step: int32   // 다음 실행할 step index
  false_branch_step: int32  // 또는 -1 (분기 없음)
  rules: [...]
  logic: string
}
```

- **true_branch_step이 -1**이면 분기 끝 (시나리오 종료)
- **단일 규칙 + rules 둘 다** — 하위 호환
- 구 버전 Agent가 rules 무시하고 평면 필드만 읽어도 동작 — **점진적 마이그레이션 지원**
