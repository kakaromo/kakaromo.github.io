## "왜 @xyflow을 선택했나"

비주얼 DAG 편집기 후보:

| 라이브러리 | Svelte 지원 | 복잡도 | MOVE 선택 |
|---|---|---|---|
| React Flow | Svelte 미지원 | 높음 | ✗ (React 필요) |
| **@xyflow/svelte** | 공식 | 중간 | ✅ |
| Drawflow | 공식 | 낮음 | ✗ (기능 부족) |
| 자체 구현 | — | 높음 | ✗ (시간 대비 가치 없음) |

@xyflow은 **"기능 풍부 + Svelte 네이티브"** 의 유일 옵션. 커스텀 노드 Svelte 컴포넌트로 직접 작성 가능 — `nodeTypes={ step: StepNode, condition: ConditionNode, loopGroup: LoopGroup }` 한 줄 매핑.

## 특수 타입 prefix `__`의 이유

팔레트에서 Loop/Condition을 드래그하면 데이터 값이 `__loop__`/`__condition__`:

- step 타입(`benchmark`, `shell` …)과 **네임스페이스 분리** — 추후 step에 `loop`라는 이름이 추가돼도 충돌 없음
- 드롭 핸들러가 `__` prefix 확인 후 **loopGroup 또는 condition 노드 생성** — 일반 step과 다른 경로
- 팔레트와 실제 노드 타입 **1:1 아님** — UI 표현(step + control) vs 데이터 구조(step/condition/loopGroup) 분리

**UI 분류와 데이터 분류를 섞지 않는** 작은 디자인 결정.

## 커스텀 노드 3종 — 차이

| 노드 | 시각 형태 | 데이터 | 특이점 |
|---|---|---|---|
| **StepNode** | 둥근 사각 + 색상 배지 | StepNodeData | stepForm 전체를 가짐 (AgentStepEditDialog 재사용) |
| **ConditionNode** | 마름모 | ConditionNodeData | **소스 핸들 2개** (true=녹색/오른쪽, false=빨강/왼쪽) |
| **LoopGroup** | 컨테이너 (부모-자식) | LoopGroupData | 내부에 step들이 들어감, loopCount |

SvelteFlow의 `parentNode` 속성으로 LoopGroup이 자식 노드들을 "감싼" 레이아웃을 자동 처리.

## 실시간 상태 반영 — execStatus

각 노드 데이터가 `execStatus`, `execOrder`, `execLoopCurrent` 등 **실행 중 상태** 필드 보유:

```typescript
interface StepNodeData {
  stepForm: StepForm;
  label: string;
  stepType: string;
  execOrder?: number;       // 시나리오 내 실행 순번
  execStatus?: 'idle'|'running'|'completed'|'failed'|'skipped';
  execLoopCurrent?: number;
  execLoopTotal?: number;
  execProgress?: number;
}
```

실행 중 JobProgress 스트림에서 받은 상태를 해당 노드에 write → SvelteFlow가 **rerender** → 노드 배경색이 상태 색상으로 바뀜. 정적 편집 UI가 동적 대시보드로 자연스럽게 전환.
