## 이 페이지는 무엇인가

지금까지의 7단계는 "**어떤 구조로 되어 있는가**"였습니다. 이 페이지는 "**그 구조에 도달하기까지 어떤 판단을 했는가**"에 대한 심화 자료입니다.

프론트엔드 `/testdb/slots/+page.svelte`는 한때 **3,494줄, `$state` 98개**의 거대한 단일 파일이었습니다. Phase 1~4 리팩토링을 거치며 2,572줄로 줄였고, 5개의 재사용 모듈이 분리됐습니다. 이 과정의 원칙과 판단이 별도 문서로 정리되어 있습니다.

**원본 문서**: [개발자 가이드 / slots 페이지 리팩토링 여정](/developer/slots-refactor/)

## 세 가지 원칙 (요약)

원본에서 가장 중요한 부분입니다. 작업할 때마다 스스로 세 번 물어보세요.

<div class="learn-callout learn-callout--concept">

**1. 이 덩어리가 자체 상태·로직을 가지고 있는가?**

자체 상태가 없고 부모 상태의 "뷰"라면 분리해도 prop 폭탄만 늘어나고 결합도는 그대로입니다.

**2. 분리 후 prop/callback이 10개 이하로 수렴하는가?**

10개를 넘으면 파일만 나뉘고 결합도는 그대로. 그럴 거면 차라리 안 나누는 게 낫습니다.

**3. 기능이 100% 유지되는가 (UI·동작·성능)?**

리팩토링 커밋은 행동을 바꾸지 않습니다. 바꿀 거면 **별도 커밋**으로.

</div>

## 이 L2와 어떻게 연결되는가

본 L2(단계 1~7)는 **백엔드 흐름**과 **프론트 store**까지만 다룹니다. 페이지 `+page.svelte` 내부의 UI 조립 방식 — 어떤 다이얼로그가 어디서 열리고, 드래그 선택이 어떻게 구현되어 있고, TC 배포가 어떤 순서로 진행되는지 — 는 이 범위 밖입니다.

리팩토링 여정 원본을 읽으면 다음을 얻습니다.

- **실무에서 "나누지 않는" 판단이 어떻게 생기는가** — "prop 10개 넘으면 안 나눈다" 같은 룰의 근거
- **변경 영향을 어떻게 관측했는가** — `svelte-check` baseline 유지, 수동 테스트 체크리스트
- **롤백 단위로 커밋하는 훈련** — 각 Phase가 독립된 커밋

## 언제 이 문서로 와야 하는가

- 본 L2 7페이지를 끝내고 슬롯 페이지 UI 코드를 실제로 만질 때
- 다른 L2 모듈(벤치마크·원격)에서 비슷한 규모의 `+page.svelte`를 리팩토링할 때
- **본인 코드에서 "언제 컴포넌트를 나눠야 하나" 기준이 모호할 때**
