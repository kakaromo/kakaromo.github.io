## 30초 요약

Head와의 TCP는 일반적인 클라이언트-서버가 아닙니다. **두 개의 소켓이 반대 방향으로** 열립니다.

- **outSocket (MOVE → Head)**: 세션 유지용. 한 번 열고 계속 유지. 제어 명령을 보낼 때도 사용.
- **serverSocket → inSocket (Head → MOVE)**: MOVE가 `listenPort`로 **bind+listen**, Head는 데이터를 보낼 때마다 여기에 **새 TCP 연결을 맺고** 데이터만 보내고 닫음.

이 설계는 MOVE가 정한 것이 아니라 Head 서버의 프로토콜입니다. MOVE는 그 규약에 맞춰 `ServerSocket.accept()` 루프를 돈다.

## 단계 1: 연결 수립 (connect)

첫 연결에서 MOVE는 **3단계**를 합니다.

1. MOVE 측 `serverSocket`을 `listenPort`에 bind (앞으로 Head가 이 포트로 올 것)
2. Head 측 `headPort`로 `outSocket` 연결 (세션 유지)
3. 약속된 **"connect!!" 명령** 을 outSocket으로 전송 — "나는 이 IP:listenPort로 받을 준비가 되었다"고 알리는 핸드셰이크

## 단계 2: 데이터 수신 (acceptLoop + readFromSocket)

Head는 **슬롯 데이터를 보낼 때마다 새 TCP 연결**을 엽니다. MOVE는 `accept()`로 이를 받고, 한 연결 안에서 여러 청크를 읽은 뒤 Head가 close하면 닫고 다음 connection을 기다립니다.

하나의 connection은 긴 스트림일 수 있고, 그 안에 **`^end^`** 구분자로 여러 청크가 포함됩니다.

## 왜 듀얼 소켓인가

Head가 한 커넥션 안에서 양방향으로 주고받는 설계를 쓸 수도 있었을 텐데, 실제로는 **방향별 소켓 분리**로 갑니다. 이 선택의 장점:

- **방향이 명확해서 디버깅이 쉽다** — 명령은 outSocket, 데이터는 inSocket. tcpdump 시 필터가 깔끔합니다.
- **많은 데이터가 오는 쪽만 짧은 연결로** — Head가 한 번 보내고 끊어도 outSocket은 영향 없음. 역방향 제어 명령은 언제든 가능.
- **MOVE는 accept만 하면 되니 간단** — 복잡한 데이터 프레임 복원 없이 "^end^까지 읽기"만 처리.

단점은 포트가 두 개 필요하다는 것. 그래서 단계 1의 suffix 규칙으로 포트 관리를 단순화합니다.

## 명령 송신 (옵션) — 왜 매번 새 Socket인가

앞서 말한 outSocket이 "제어 명령도 보낸다"고 했지만, 실제로는 `settr`, `settc`, `makeset` 같은 **제어 명령은 매번 새 Socket을 열어 보냅니다**. 이유는 Head가 제어 명령 응답을 보낸 뒤 연결을 close하기 때문입니다. 세션 유지용 outSocket을 그대로 쓰면 세션이 끊어지는 것. 이 부분은 이 문서 범위를 벗어나지만, `sendCommand(new Socket(...), cmd)` 패턴이 반복되면 이 이유 때문입니다.
