## 30초 요약

`headSlotStore.svelte.ts`는 **SSE를 구독하고 결과를 Svelte 5 runes로 노출하는 팩토리 함수**입니다. 페이지에서 `createHeadSlotStore("foo")`를 부르면 자체 상태를 가진 store 객체가 만들어지고, 페이지의 반응성 시스템이 이 store를 그대로 구독합니다.

Head가 여러 개면 **각 Head마다 별도 store**가 만들어집니다. 서로 다른 EventSource 연결, 서로 다른 $state.

## 왜 `.svelte.ts` 확장자인가

일반 `.ts` 파일에서는 rune(`$state`, `$derived`, `$effect`)을 쓸 수 없습니다. **`.svelte.ts` 확장자**로 저장해야 Svelte 컴파일러가 이 파일을 runes 파일로 인식해 변환합니다. 이 규약 덕분에 "어떤 파일에 runes가 있는지"가 파일 이름만으로 드러납니다.

## <ConceptPopover client:visible term="$state" concept="svelte/state"><p>Svelte 5의 반응형 상태 rune. 변수에 할당된 객체·배열을 deep proxy로 감싸 반응성을 자동 추적합니다. 단순 참조 보관이 필요하면 <code>$state.raw</code>를 쓰세요.</p></ConceptPopover>, <ConceptPopover client:visible term="$derived" concept="svelte/derived"><p>다른 $state에서 파생된 읽기 전용 값. 의존성을 자동 추적해 필요할 때만 재계산됩니다. <code>$derived.by(() =&gt; ...)</code> 형태로 복잡한 식도 가능.</p></ConceptPopover>, <ConceptPopover client:visible term="$effect" concept="svelte/effect"><p>반응형 의존성이 바뀔 때 실행되는 부작용 훅. cleanup 함수를 return해 이전 실행을 정리할 수 있습니다. DOM 수동 조작이나 이벤트 리스너 등록에 사용.</p></ConceptPopover>

`headSlotStore`는 `$state`만 사용합니다 — 단순한 값 보관이 전부이기 때문. 페이지(`+page.svelte`) 쪽에서 슬롯 필터링·정렬·요약 통계를 만들 때는 `$derived`를 사용하고, SSE 연결 라이프사이클은 `$effect`에서 다룹니다.

## 페이지에서 어떻게 쓰는가 (요약)

```svelte
<script lang="ts">
  import { createHeadSlotStore } from '$lib/api/headSlotStore.svelte';

  const store = createHeadSlotStore('foo');

  $effect(() => {
    store.connect();
    return () => store.disconnect();
  });

  const connectedSlots = $derived(
    store.slots.filter((s) => s.connection >= 1)
  );
</script>

{#each store.slots as slot (slot.slotIndex)}
  <SlotCard {slot} />
{/each}
```

`$effect` 안에서 `connect()`를 부르고 return cleanup으로 `disconnect()`. 페이지 이탈 시 EventSource가 확실히 닫힙니다. 이 패턴이 `routes/testdb/slots/+page.svelte`에서 반복됩니다.

## 왜 별도 팩토리 함수인가

Svelte store 패턴(writable/readable) 대신 **팩토리 + runes**를 고른 이유:

- Svelte 5에서 runes가 공식 길이라 store 래퍼는 점점 줄이는 추세
- Head가 여러 개일 때 **store가 여러 인스턴스 필요** — 글로벌 싱글톤이 아니라 페이지별 생성
- 순수 함수처럼 read-only 반환 API를 만들 수 있음 (위 getter 패턴)
