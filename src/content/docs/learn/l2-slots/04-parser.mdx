## 30초 요약

Head는 슬롯 상태를 **텍스트 프로토콜**로 보냅니다. 한 청크는 다음 둘 중 하나입니다.

- **`initslots[N]field1`field2`...`field30^`** — 초기 상태 전체
- **`update[N]field1`field2`...^`** — 특정 슬롯 갱신

`HeadMessageParser`는 이 문자열을 **정규식으로 추출**하고, 필드를 **backtick으로 분리**해 `HeadSlotData`의 32개 setter에 매핑합니다. static 유틸 클래스로 상태를 갖지 않습니다.

## 필드 포맷

<div class="learn-callout learn-callout--concept">

하나의 슬롯 메시지 예시 (간략화):

```text
update[0]2h15m`SM-A536B`85%`8.2GB`fio`write`1`Running`usage`...^
      ^  ^       ^    ^   ^    ^     ^   ^      ^
      │  │       │    │   │    │     │   │      └── setLocation
      │  │       │    │   │    │     │   └── testState (Running/Waiting 등)
      │  │       │    │   │    │     └── state (숫자 1~9)
      │  │       │    │   │    └── testTrName (테스트 Tr 이름)
      │  │       │    │   └── testToolName (fio/iozone/tiotest)
      │  │       │    └── freeArea
      │  │       └── remainBattery
      │  └── setModelName
      └── runningTime
```

필드 15번째(0-indexed)는 추가로 **`;` 서브 구분자**를 갖습니다 — `preIdGlobalCxt;bootCxt;smartReport;osv`. 이 서브필드는 slot의 boot 상태를 나타냅니다.

</div>

전체 필드 정의는 아래 코드 투어에서 순서대로 보입니다. 외울 필요는 없고, **"어느 위치가 어떤 의미인지 여기 한 곳에서만 결정된다"** 는 사실이 중요합니다.

## 왜 static 유틸 클래스인가

`HeadMessageParser`는 필드도 상태도 없는 **pure function** 덩어리입니다. Spring 빈으로 등록할 가치가 없고, 테스트 시 입력 문자열만 바꾸면 완전한 단위 테스트가 가능합니다. 순수 함수로 만들 수 있는 부분은 순수 함수로 두는 것이 MOVE의 기본 방침입니다.

부작용이 있어야 할 곳은 **단계 5의 `HeadSlotStateStore`** 이며, 단계 3의 `processChunk`가 이 둘을 이어붙입니다.

## 공백·누락 필드 대처

Head가 간헐적으로 **빈 필드**를 보낼 수 있습니다(운영 중 발견된 실제 케이스). 파서는 세 가지 방어를 둡니다.

1. `split(..., -1)`로 **끝이 빈 필드도 배열에 포함**
2. `fields.length >= N` 가드로 **실제 필드 개수보다 적게 와도 NPE 없음**
3. state/testState가 비면 `"NONE"` 또는 기본값을 할당

이 방어는 "Head 측이 프로토콜 버전을 조금 바꿔도 MOVE가 멈추지 않는다"는 안전성 마진입니다. 새 필드가 추가될 때마다 이 패턴을 따라가면 됩니다.
