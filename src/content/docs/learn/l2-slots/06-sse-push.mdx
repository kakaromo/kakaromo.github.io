## 30초 요약

`HeadSseController`는 브라우저의 **EventSource가 붙는 엔드포인트** 입니다. 붙으면 현재 스냅샷을 `init` 이벤트로 즉시 보내고, 이후 **1초마다 version 변화를 확인**해서 바뀐 경우에만 `update` 이벤트를 보냅니다.

각 연결은 `EmitterWrapper`로 감싸져서 **`lastVersion`을 개별 보유**합니다. 덕분에 여러 클라이언트가 각자 다른 시점에 붙어도 누락이 없습니다.

## 흐름 요약

1. **GET /api/head/slots/stream** → `SseEmitter` 생성, emitters 리스트에 wrapper 추가, `init` 이벤트 전송
2. `@Scheduled(fixedDelay = head.sse.push-interval-ms)` 주기마다:
   - `stateStore.getVersion()` 조회
   - 각 wrapper의 `lastVersion`보다 크면 `update` 이벤트 전송 + `lastVersion` 갱신
3. 에러가 나면 해당 wrapper만 제거 (다른 클라이언트는 영향 없음)

## <ConceptPopover client:visible term="@Scheduled" concept="spring/at-scheduled"><p>주기 실행 어노테이션. <code>fixedRate</code>는 시작 시각 기준, <code>fixedDelay</code>는 종료 시각 기준, <code>cron</code>은 cron 표현식. <code>@EnableScheduling</code>이 필요하며 MOVE에서는 <code>PortalApplication</code>에 이미 설정되어 있습니다.</p></ConceptPopover> 와 <ConceptPopover client:visible term="SseEmitter" concept="spring/sse-emitter"><p>Spring MVC의 서버 전송 이벤트 추상체. <code>send()</code>로 이벤트를 비동기 전송하고, <code>onCompletion/onTimeout/onError</code>로 종료 경로를 등록합니다. 컨트롤러가 반환하는 순간 요청 스레드는 풀려나고, emitter는 비동기 context에서 살아남습니다.</p></ConceptPopover> 조합

SSE가 Spring에서 자연스럽게 동작하는 이유는 이 두 요소 덕분입니다. `@Scheduled`로 push 주기를 선언적으로 두고, `SseEmitter`는 비동기 응답을 **컨트롤러 메서드가 반환한 뒤에도** 계속 유지합니다.

## 왜 fixedRate가 아니라 fixedDelay인가

`fixedRate`는 **시작 시각 기준**이라 이전 실행이 1초를 넘기면 다음 실행이 바로 겹쳐 실행될 수 있습니다. `fixedDelay`는 **종료 시각 기준**이므로 항상 "직전 완료 + 1초" — 느려진 경우 자연스럽게 뒤로 밀립니다. 100개 emitter에 JSON 직렬화가 쌓이는 상황에서 겹침을 원하지 않아 `fixedDelay`로 선택.

## EmitterWrapper를 없애고 전역 lastVersion 하나만 쓰면 안 되나

안 됩니다. 이유:

- 클라이언트가 **서로 다른 시각에** 붙습니다. A는 10초 전에 붙었고 version 50까지 받았다면, 지금 들어오는 B는 version 55 스냅샷을 받아야 하지만 A는 51~55만 추가로 받으면 됩니다.
- `source` 필터가 다를 수 있습니다. A는 `?source=foo`, B는 `?source=bar`면 페이로드 내용도 서로 다릅니다.

연결당 상태를 분리해야 이런 차이가 자연스럽게 처리됩니다.

## 직접 해보기

- [ ] `curl -N http://localhost:8080/api/head/slots/stream` 로 SSE 연결. 첫 줄에 `event: init` + JSON 페이로드가 보여야 합니다.
- [ ] `?source=foo`를 붙여 필터가 적용되는지 확인.
- [ ] DevTools Network → EventStream 탭에서 1초 주기의 `update` 이벤트 확인. Head에서 데이터가 오지 않을 때는 `update` 이벤트가 오지 **않아야** 합니다(version 무변화).
