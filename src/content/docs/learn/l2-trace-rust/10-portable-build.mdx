## 30초 요약

trace 빌드 스크립트 `scripts/build.sh` 는 **외부 망 의존 없이** 호스트의 system rustc + linker 만으로 빌드하도록 만들어져 있습니다. 회사 사내망에서도 그대로 동작.

```bash
./scripts/build.sh stats-duckdb
```

결과: `target/release/trace`. 빌드 머신과 같은 OS/배포판/glibc 머신에 그대로 배포.

추가 적용된 호환성 카드:
- **rustls 전환** — `rust-s3` 가 OpenSSL(libssl) 동적 링크 안 함 → 사내 머신마다 다른 libssl 버전과 무관
- **DuckDB bundled** — C++ 정적 링크라 `libduckdb.so` 같은 외부 의존 0

## 왜 cross-compile 안 쓰나

Ubuntu 24.04 빌드 머신 → 18.04 target 같은 cross-compile 은 `cargo-zigbuild` + `zig` 가 표준이지만:

- `cargo install cargo-zigbuild` → crates.io 접근 필요
- `pip install ziglang` 또는 zig 바이너리 다운로드 → 외부 네트워크 필요

폐쇄망 빌드 머신에선 둘 다 **불가능**. 그래서 표준 `cargo build` 로 단순화하고, 호환성은 "**같은 OS 버전 머신에서 빌드해서 같은 OS 머신에 배포**" 로 풀기로 결정.

배포 OS 가 여러 종 이라면 빌드 머신을 가장 옛 버전 (예: 18.04) 으로 두고 그 한 번 빌드 결과를 공통 사용. forward-compatible (옛 → 새) 만 지원되니까 18.04 빌드 산출물은 22.04/24.04 에서도 그대로 실행 가능.

## rust-s3 → rustls 전환

`Cargo.toml`:

```toml
# Before — native-tls (libssl 동적 링크)
rust-s3 = "0.35"

# After — rustls (Pure Rust TLS, glibc 외 의존 없음)
rust-s3 = { version = "0.35", default-features = false, features = ["tokio-rustls-tls"] }
```

`Cargo.lock` 변화:
- 빠짐: `hyper-tls`, `native-tls`, `openssl-sys`, `openssl-probe`
- 추가: `rustls`, `tokio-rustls`, `hyper-rustls`, `rustls-webpki`

이 변경의 의미: 사내 머신에 OpenSSL 이 없거나 (`libssl.so` 부재), 버전이 빌드 시점과 다르거나 해도 trace 가 정상 실행. 어떤 신원의 Linux 머신이든 glibc 만 호환되면 돌아감.

## 빌드 스크립트

`scripts/build.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")/.."

FEATURES="${1:-}"
PROFILE="${PROFILE:-release}"
PROFILE_FLAG=$([ "$PROFILE" = "release" ] && echo "--release" || echo "")
OUT_DIR=$([ "$PROFILE" = "release" ] && echo "target/release" || echo "target/debug")

if [ -n "$FEATURES" ]; then
    cargo build $PROFILE_FLAG --features "$FEATURES"
else
    cargo build $PROFILE_FLAG
fi

BIN="$OUT_DIR/trace"
[ "${STRIP:-0}" = "1" ] && strip "$BIN"
ls -lh "$BIN"
file "$BIN"
```

핵심:
- system `cargo build` 외 의존 없음 — 폐쇄망에서도 동작
- `STRIP=1` 옵션으로 빌드 후 디버그 심볼 제거 (배포 크기 절감)
- `PROFILE=debug` 로 dev 빠른 빌드

## 사용

```bash
# 운영 빌드 (DuckDB stats 엔진 포함)
./scripts/build.sh stats-duckdb

# DuckDB 없이 가벼운 빌드 (parser/processor 검증용)
./scripts/build.sh

# 빌드 + strip (배포 크기 ↓)
STRIP=1 ./scripts/build.sh stats-duckdb

# debug 빌드
PROFILE=debug ./scripts/build.sh
```

## 검증 — target 머신에서

```bash
# 동적 링크 라이브러리 — libssl/libcrypto 안 보여야 함
ldd ./trace

# 실제 실행
./trace --help

# 요구하는 glibc 심볼 (호환성 디버깅용)
objdump -T ./trace | grep GLIBC_ | sort -u | tail -5
```

예상 출력:

```
$ ldd ./trace
        linux-vdso.so.1 (0x...)
        libpthread.so.0 => /lib/x86_64-linux-gnu/libpthread.so.0
        libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6
        /lib64/ld-linux-x86-64.so.2
```

`libssl.so` / `libcrypto.so` 가 보이면 → rustls 전환이 안 박힌 것. `Cargo.toml` 의 `rust-s3` features 확인.

## 트러블슈팅

### `GLIBC_2.XX not found`

빌드 머신의 glibc 버전이 target 머신보다 높음. 해결:

1. 더 낮은 버전 OS 에서 빌드 (예: target 이 18.04 이면 빌드도 18.04)
2. Docker 로 옛 OS 환경 재현:
   ```bash
   docker run --rm -v "$(pwd)":/work -w /work \
       -v "$HOME/.cargo":/root/.cargo \
       ubuntu:18.04 bash -c "
       apt update && apt install -y curl build-essential pkg-config protobuf-compiler
       curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh -s -- -y
       source \$HOME/.cargo/env
       ./scripts/build.sh stats-duckdb
       "
   ```
   사내망에선 base image 와 apt 미러를 사내 mirror 로 (예: `apt-mirror.company.local`).

### 빌드 시간이 느림 (DuckDB)

`stats-duckdb` feature 빌드 시 DuckDB C++ 소스 전체 컴파일 = +30~60초. CI 에선 cache 활용:

```bash
# Cargo registry / target dir 캐시
- target/
- ~/.cargo/registry/
- ~/.cargo/git/
```

## DuckDB 의존 (stats-duckdb feature)

- `duckdb = { features = ["bundled"] }` → C++ 코드 정적 링크. 외부 `libduckdb.so` 의존 0
- `parquet` extension 도 bundled (libduckdb-sys feature) — 인터넷 LOAD 불필요
- `httpfs` extension 만 별도 — 사내 미리 깐 `~/.duckdb/extensions/<ver>/<arch>/httpfs.duckdb_extension` 을 LOAD

httpfs 는 한 번 사외 머신에서 받아 사내에 복사하는 패턴. 해당 절차는 [duckdb-pool](/learn/l2-trace-rust/08-stats-pipeline/) 참고.
