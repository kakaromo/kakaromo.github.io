## 30초 요약

Ubuntu 24.04 빌드 머신에서 만든 바이너리를 회사의 18.04/20.04/22.04 머신에서도 그대로 실행하려면 두 가지를 잡아야:

1. **glibc 버전 호환** — 24.04 의 glibc 2.39 동적 링킹은 18.04 의 2.27 에서 실행 불가 (`GLIBC_2.32 not found`)
2. **OpenSSL 의존 제거** — `rust-s3` 의 default `native-tls` 는 시스템 libssl 동적 링크. 버전/존재 여부 환경마다 다름

해결:
- **glibc**: `cargo-zigbuild` + `--target x86_64-unknown-linux-gnu.2.27` 로 cross-link
- **OpenSSL**: `rust-s3` 를 `tokio-rustls-tls` feature 로 전환 → libssl 의존 0

빌드 스크립트 `scripts/build-linux-portable.sh` 한 번 호출:

```bash
./scripts/build-linux-portable.sh stats-duckdb
```

결과: `target/x86_64-unknown-linux-gnu/release/trace` — 18.04 ~ 24.04 모두 실행 가능.

## 왜 이게 문제인가

리눅스 바이너리의 호환성은 **빌드 시점 glibc 버전 ≤ 실행 시점 glibc 버전** 일 때만 성립. forward-compatible (옛 → 새) 만 지원되고 backward 는 안 됨.

| Ubuntu | glibc |
|---|---|
| 18.04 | 2.27 |
| 20.04 | 2.31 |
| 22.04 | 2.35 |
| 24.04 | 2.39 |

24.04 에서 `cargo build --release` → glibc 2.39 동적 링크 → 18.04 에서 실행 시:

```
./trace: /lib/x86_64-linux-gnu/libc.so.6: version `GLIBC_2.32' not found
```

OpenSSL 도 마찬가지 패턴. `libssl.so.3` (24.04) 는 `libssl.so.1.1` (18.04) 환경에서 안 보임.

## 후보 3가지 비교

| 방법 | 장점 | 단점 |
|---|---|---|
| musl static | glibc 의존 0, 어디서든 동작 | DuckDB bundled C++ 빌드 실패 가능 (libstdc++ static link 이슈) |
| Docker 18.04 빌드 | 가장 단순, 실증된 호환 | DuckDB 1.4 가 C++17 필요 — 18.04 gcc 7.5 부족, gcc 9 PPA 추가 필요 |
| **cargo-zigbuild** | 24.04 환경 그대로 + glibc 버전 명시 | zig 설치 필요 (한 번만) |

zigbuild 를 선택한 이유:
- 한 명령으로 cross-link
- DuckDB bundled C++ 도 zig 의 clang 으로 잘 빌드 (musl 보다 안정)
- glibc 버전을 target triple 에 명시 (`x86_64-unknown-linux-gnu.2.27`)

## rust-s3 → rustls 전환

`Cargo.toml`:

```toml
# Before — native-tls (libssl 동적 링크)
rust-s3 = "0.35"

# After — rustls (Pure Rust TLS, glibc 외 의존 없음)
rust-s3 = { version = "0.35", default-features = false, features = ["tokio-rustls-tls"] }
```

`Cargo.lock` 변화:
- 빠짐: `hyper-tls`, `native-tls`, `openssl-sys`, `openssl-probe`
- 추가: `rustls`, `tokio-rustls`, `hyper-rustls`, `rustls-webpki`

빌드 시간 거의 동일. 런타임 perf 도 비슷 (rustls 가 일부 cipher 에서 살짝 느릴 수 있지만 MinIO 호출 frequency 가 낮아 무관).

## 빌드 스크립트

`scripts/build-linux-portable.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$(dirname "$0")/.."

FEATURES="${1:-}"
GLIBC="${GLIBC:-2.27}"
TARGET="x86_64-unknown-linux-gnu.${GLIBC}"

# 도구 확인
command -v cargo-zigbuild >/dev/null || { echo "cargo install cargo-zigbuild"; exit 1; }
command -v zig >/dev/null || python3 -c "import ziglang" 2>/dev/null \
    || { echo "pip install ziglang  또는  brew install zig"; exit 1; }

# target 등록 (한 번만)
rustup target list --installed | grep -q "^x86_64-unknown-linux-gnu$" \
    || rustup target add x86_64-unknown-linux-gnu

# 빌드
if [ -n "$FEATURES" ]; then
    cargo zigbuild --release --target "$TARGET" --features "$FEATURES"
else
    cargo zigbuild --release --target "$TARGET"
fi

ls -lh target/x86_64-unknown-linux-gnu/release/trace
file target/x86_64-unknown-linux-gnu/release/trace
```

## 사용법

처음 한 번:

```bash
cargo install cargo-zigbuild
pip install ziglang   # 또는: brew install zig
```

매 빌드:

```bash
# default features
./scripts/build-linux-portable.sh

# duckdb feature 포함 (운영 빌드)
./scripts/build-linux-portable.sh stats-duckdb

# 더 옛날 OS (CentOS 7 등) 지원하려면 GLIBC 낮추기
GLIBC=2.17 ./scripts/build-linux-portable.sh stats-duckdb
```

## 검증 — target 머신에서

```bash
# 1. 동적 링크된 라이브러리 — libssl/libcrypto 안 보여야 함
ldd ./trace

# 2. 요구하는 glibc 버전 — 최대값이 빌드 시 지정한 GLIBC 와 같거나 낮아야 함
objdump -T ./trace | grep GLIBC_ | sort -u | tail -10

# 3. 실제 실행
./trace --help
```

예상 출력:

```
$ ldd ./trace
        linux-vdso.so.1 (0x...)
        libpthread.so.0 => /lib/x86_64-linux-gnu/libpthread.so.0
        libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6
        /lib64/ld-linux-x86-64.so.2

$ objdump -T ./trace | grep GLIBC_ | sort -u | tail
GLIBC_2.18
GLIBC_2.25
GLIBC_2.27       # ← 이게 최대값. 빌드 시 지정한 2.27 이하면 OK
```

`libssl.so.3` 이 보이면 → rust-s3 features 설정 확인 (rustls 로 안 바뀐 상태).
`GLIBC_2.30` 같이 더 높은 게 보이면 → 의존 crate 가 더 새 syscall 호출. cargo tree 로 누가 쓰는지 추적 후 alternative 검토.

## 트러블슈팅

### `cargo zigbuild --target x86_64-unknown-linux-gnu.2.27` 가 long compile 후 link error

DuckDB 1.4 bundled 가 C++17 표준 라이브러리 의존. zig 의 clang 이 stdlib link 못 찾는 경우:

```bash
# zig 버전 업그레이드 (0.13.0 이상 권장)
pip install -U ziglang
```

### symbol versioning 충돌

특정 crate (예: `ring`, `aws-lc`) 가 너무 최신 syscall 사용:

```bash
# 어느 crate 가 GLIBC_2.30 요구하는지
objdump -T ./trace | grep GLIBC_2.30 | awk '{print $7}' | sort -u
# 그 함수 이름으로 cargo tree -i 검색
```

대안 crate 로 교체 또는 GLIBC target 을 그 함수 도입 버전 (예: 2.30) 으로 올림.

### 빌드 머신에서만 테스트하고 싶을 때

```bash
# Linux container 안에서 검증
docker run --rm -v "$(pwd)":/work -w /work ubuntu:18.04 ./target/x86_64-unknown-linux-gnu/release/trace --help
```

## 빌드 머신 OS 별 정리

| 빌드 머신 | 기본 cargo build | zigbuild GLIBC=2.27 |
|---|---|---|
| macOS (M1/M2) | Mac 만 동작 | Linux 18.04+ 동작 |
| Ubuntu 24.04 | 24.04 만 동작 | 18.04+ 동작 |
| Ubuntu 20.04 | 20.04+ 동작 | 18.04+ 동작 |
| Ubuntu 18.04 | 18.04+ 동작 | 변화 없음 (이미 호환) |

회사 환경이 18.04 ~ 24.04 혼재면 어떤 빌드 머신에서든 zigbuild 로 통일.
