## 30초 요약

`trace --grpc-server`는 `proto/log_processor.proto`에 정의된 **11개 RPC**를 tonic으로 노출합니다. 절반은 streaming(progress/이벤트), 절반은 unary(통계/리스트).

| # | RPC | 모드 | 입력 | 출력 |
|---|---|---|---|---|
| 1 | `ProcessLogs` | server-streaming | MinIO 로그 path | `ProcessLogsProgress` × N (DOWNLOAD→PARSING→…→COMPLETED) |
| 2 | `ConvertToCsv` | server-streaming | MinIO parquet path | `ConvertToCsvProgress` × N |
| 3 | `GetJobStatus` | unary | `job_id` | 마지막 progress snapshot |
| 4 | `ListFiles` | unary | bucket + prefix | 파일 path list |
| 5 | `ReadParquet` | server-streaming | path(s) + offset/cursor + filter + time_range | row JSON × N (+ `columns` / `schema_version` / `next_cursor_*` 첫 메시지) |
| 6 | `StartRealtimeParsing` | server-streaming | log path + window | `ParquetFileReady` × N (윈도우마다 1개) |
| 7 | `StopParsing` | unary | `job_id` | 마지막 파일 + 누적 통계 |
| 8 | `GetChartData` | unary | `parquet_path` 또는 `parquet_paths`(repeated) + 시간 범위 + actions | **Arrow IPC bytes** + latency 요약 |
| 9 | `GetTraceStats` | unary | `parquet_path` 또는 `parquet_paths`(repeated) + 시간 범위 + bucket edges | **JSON 통계** (cmd_stats, histograms, ...) |
| 10 | `GetRawLogLines` | unary | upload bucket/path + `line_numbers[]` + ± context | 라인 번호별 `RawLogBlock` (텍스트 ± 컨텍스트) |
| 11 | `ExportXlsx` | server-streaming | `parquet_paths[]` + filter/time_range + target zip path | `ExportXlsxProgress` × N (DOWNLOADING→FILTERING→WRITING→ZIPPING→UPLOADING) + 최종 ZIP/xlsx URL |

> **멀티 parquet 입력** (8/9/11): `parquet_paths` 는 realtime 모드의 윈도우별 parquet 을 한 번에 분석할 때 사용. `union_by_name=True` 로 스키마가 다른 옛/새 parquet 도 같이 처리. 8/9 는 **DuckDB reader 만 지원** — async/sync reader 는 멀티 입력 시 `InvalidArgument` 반환 (`grpc/dispatch.rs`). 11(ExportXlsx) 도 멀티 path 입력.

> **신규 (5/10/11)**: ReadParquet 은 raw data 페이지 + 무한 스크롤(keyset cursor)·DuckDB 위임 경로 추가 / GetRawLogLines 는 ReadParquet row 의 `line_number` 와 결합해 원본 .log 라인을 lazy 조회 / ExportXlsx 는 ConvertToCsv 의 분할 정책을 그대로 따라 1M row/file 분할 → ZIP → MinIO 업로드.

이 장은 위 11개를 카테고리로 묶어 설명합니다.

## 카테고리 — 데이터 변환 vs 조회 vs 실시간 vs 내보내기

| 카테고리 | RPC | 특징 |
|---|---|---|
| **변환 파이프라인** | `ProcessLogs`, `ConvertToCsv` | 큰 입력 → 큰 출력. 진행률 streaming. job_id로 추적 |
| **메타·내용 조회** | `GetJobStatus`, `ListFiles`, `ReadParquet`, `GetRawLogLines` | MinIO/in-memory 상태·row·원본 .log 조회 |
| **실시간** | `StartRealtimeParsing`, `StopParsing` | tail -f + 윈도우 streaming. CancellationToken |
| **분석 응답** | `GetChartData`, `GetTraceStats` | UI가 직접 호출. parquet → Arrow IPC / JSON |
| **내보내기** | `ExportXlsx` | parquet → 분할 xlsx → ZIP → MinIO. ConvertToCsv 패턴 + xlsx |

분석 카테고리(8/9)가 portal이 가장 자주 호출하고, 내보내기(11)는 사용자 다운로드 트리거 시 단발성 호출입니다.

## tonic 서버 셋업

`grpc/server.rs` 진입점:

```rust
use tonic::transport::Server;

#[tokio::main]
pub async fn run_grpc_server(port: u16, minio_config: MinioConfig)
    -> Result<(), Box<dyn Error>>
{
    let addr = format!("0.0.0.0:{}", port).parse()?;
    let service = LogProcessorService::new(minio_config);

    println!("🚀 gRPC server listening on 0.0.0.0:{}", port);

    Server::builder()
        .max_decoding_message_size(256 * 1024 * 1024)  // 256MB
        .max_encoding_message_size(256 * 1024 * 1024)
        .add_service(LogProcessorServer::new(service))
        .serve(addr)
        .await?;
    Ok(())
}
```

핵심 디테일:

- **256MB max message size** — Arrow IPC payload가 큰 차트 응답을 한 번에 보내기 위해. 기본 4MB로는 부족
- **단일 서비스** — 11개 RPC가 모두 `LogProcessorService` 한 impl 안. 각각 트레이트 메서드
- **MinioConfig 공유** — 모든 RPC가 같은 client 인스턴스를 빌려 씀

## ProcessLogs — 변환 파이프라인의 본보기

```rust
async fn process_logs(
    &self,
    request: Request<ProcessLogsRequest>,
) -> Result<Response<Self::ProcessLogsStream>, Status> {
    let req = request.into_inner();
    let job_id = Uuid::new_v4().to_string();
    let (tx, rx) = mpsc::channel(32);

    // job 상태 등록 (GetJobStatus가 읽음)
    self.jobs.lock().await.insert(job_id.clone(), JobState::default());

    // 무거운 작업은 별도 task에서 — 이 함수는 즉시 stream 반환
    let svc = self.clone();
    tokio::spawn(async move {
        svc.process_logs_internal(req, job_id.clone(), tx).await;
    });

    Ok(Response::new(Box::pin(ReceiverStream::new(rx))))
}
```

패턴:

1. **즉시 stream 반환** — 클라이언트는 끊지 않고 진행률을 받음
2. **job_id 발급** — `GetJobStatus`로 폴링 가능
3. **mpsc channel** — 내부 task가 progress를 push, gRPC가 pull

내부 단계는 4개의 step 함수 + 2개의 finish 헬퍼로 분리돼 있음 (Phase A 리팩토링). 임시 자원은 **외부 스코프에서 추적**해 중간 step 실패 시에도 cleanup 보장:

```rust
async fn process_logs_internal(...) {
    // 임시 자원 경로를 외부 스코프 Option<String> 으로 추적 — 중간 step `?` 트립 시에도
    // async 블록 종료 후 catch-all cleanup 으로 누수 차단 (1억 row 압축본 수십 GB 가
    // step1 성공 + step2 파싱 실패 시 디스크에 남는 운영 사례 방지).
    let mut temp_log_path: Option<String> = None;
    let mut temp_parquet_path: Option<String> = None;
    let extract_dir = format!("{}/trace_extract_{}", home_dir, job_id);

    let result: Result<(usize, Vec<String>), String> = async {
        // 1단계: 다운로드 + (필요 시) 압축 해제
        let (log_path, actual_log_path) =
            self.step_download_and_extract(&job_id, &request, &tx, &home_dir).await?;
        temp_log_path = Some(log_path);

        // 2단계: 파싱 + 필터 + latency
        let (ufs_traces, block_traces, ufscustom_traces, total_records) =
            self.step_parse_filter_latency(&job_id, &request, &tx, &actual_log_path).await?;

        // 3단계: Parquet 저장
        let parquet_path = Self::step_save_parquet(...).await?;
        temp_parquet_path = Some(parquet_path);

        // 4단계: MinIO 업로드
        let uploaded_files = self.step_upload(
            &job_id, &request, &tx,
            temp_parquet_path.as_deref().unwrap(),
            total_records,
        ).await?;

        Ok((total_records, uploaded_files))
    }.await;

    // 정리: 성공/실패와 무관하게 임시 자원 cleanup (best-effort).
    if let Some(p) = temp_log_path.as_ref()    { let _ = std::fs::remove_file(p); }
    if let Some(p) = temp_parquet_path.as_ref(){ let _ = std::fs::remove_file(p); }
    if Path::new(&extract_dir).exists() {
        let _ = std::fs::remove_dir_all(&extract_dir);
    }

    match result {
        Ok((total_records, uploaded_files)) =>
            self.finish_success(&job_id, &tx, total_records, uploaded_files).await,
        Err(e) => self.finish_failure(&job_id, &tx, e).await,
    }
}
```

각 step 함수가 자기 단계의 진행률 송신 (`send_progress` 헬퍼) + 실패 시 stack trace 가 어디서 끊겼는지 직접 드러남. 옛날엔 한 함수 396줄 인라인이라 어디서 실패했는지 추적 어려웠음.

> **stateless helper 분리** — Phase A 리팩토링 산출물로 `grpc/service_helpers.rs` 별도 impl 블록에 무상태 헬퍼 4개 분리: `chart_schema_version` / `stats_schema_version` (스키마 버전 lookup), `convert_filter_options` (proto FilterOptions → 내부 타입), `send_progress` / `send_csv_progress` (진행률 mpsc send + 에러 매핑). 핸들러 본체에서 self 의존성 없이 호출 가능 → 본체 가독성 ↑.

**임시 자원 누수 차단 (N2)**:
- `temp_log_path` (다운로드 압축본, 수십 GB 가능), `temp_parquet_path` (파싱 결과), `extract_dir` (압축 해제 디렉토리) 세 가지가 step 중간 실패 시 누수 가능했음
- 외부 스코프 `Option<String>` 추적 + 블록 종료 후 일괄 cleanup 으로 해결
- 두 cleanup 모두 `let _ = remove_*` best-effort — panic 안전

`ConvertToCsv` 도 비슷한 구조. parquet → CSV 부분만 다른 step 함수.

## 공통 진입부 — `resolve_request_meta` 헬퍼

`get_chart_data` / `get_trace_stats` 두 핸들러는 진입부가 거의 byte-equal — 단/복수 path 분기, trace_type 검증, MinioConfig 생성, 정렬·fingerprint 멀티 join 까지 ~50 줄이 같음. 이걸 `resolve_request_meta` 헬퍼 한 곳으로 모았음:

## GetChartData — Arrow IPC unary + 3-way dispatch + single-flight

핸들러는 공통 헬퍼 호출 + 캐시 lookup + dispatch 호출 + proto 변환만. moka `try_get_with` 로 같은 키 동시 N 요청을 빌더 1회로 collapse (자세한 건 [11. 캐시 시스템](/learn/l2-trace-rust/11-cache-system/)).

```rust
async fn get_chart_data(
    &self,
    request: Request<GetChartDataRequest>,
) -> Result<Response<GetChartDataResponse>, Status> {
    let req = request.into_inner();

    // 진입부 ~50 줄을 공통 헬퍼로 — 두 분석 RPC 가 동치
    let (parquet_paths, cache_path, fingerprint, trace_type, config) = self
        .resolve_request_meta(
            &req.source_bucket, &req.parquet_path,
            &req.parquet_paths, &req.trace_type,
        ).await?;

    // parquet fingerprint(ETag) + 요청 파라미터로 캐시 키 생성
    let cache_key = cache::chart_key(...);

    // try_get_with: hit 시 즉시 Arc<Response> 공유, miss 시 클로저 1회 실행 (single-flight)
    let entry_result = self.caches.chart.try_get_with(cache_key, async move {
        let reader = dispatch::select_reader();
        let (build_result, temp_file) = dispatch::build_chart_payload(
            reader, &trace_type, &config, &bucket,
            &parquet_paths, time_range, target_points,
            filter.as_ref(), &actions,
        ).await;
        if let Some(p) = temp_file.as_ref() { let _ = std::fs::remove_file(p); }
        let payload = build_result.map_err(|e| format!("chart build: {e}"))?;
        Ok::<_, String>(cache::ChartEntry {
            response: Arc::new(GetChartDataResponse {
                arrow_ipc: payload.arrow_ipc,
                total_events: payload.total_events,
                sampled_events: payload.sampled_events,
                dtoc: Some(proto_stats(&payload.dtoc)),
                ctod: Some(proto_stats(&payload.ctod)),
                ctoc: Some(proto_stats(&payload.ctoc)),
                qd:   Some(proto_stats(&payload.qd)),
                schema_version: payload.schema_version.to_string(),
            }),
        })
    }).await;
    let entry = entry_result.map_err(|e| Status::internal((*e).clone()))?;
    Ok(Response::new((*entry.response).clone()))
}
```

`grpc::dispatch::build_chart_payload` 가 어떤 reader path 갈지 결정 — 핸들러가 `select_reader()` 결과를 인자로 넘겨주고, dispatch 는 cleanup 용 임시 파일 경로(`Option<PathBuf>`) 도 함께 반환:

```rust
enum ReaderChoice { Duckdb, Async, Sync }

fn select_reader() -> ReaderChoice {
    #[cfg(feature = "stats-duckdb")]
    if duckdb_pool::is_duckdb_engine_enabled() { return ReaderChoice::Duckdb; }
    if std::env::var("TRACE_PARQUET_READER").as_deref() == Ok("async") {
        return ReaderChoice::Async;
    }
    ReaderChoice::Sync
}

pub async fn build_chart_payload(
    reader: ReaderChoice,
    trace_type: &str,
    config: &MinioConfig,
    bucket: &str,
    parquet_paths: &[String],
    /* ...time_range, target_points, filter, actions */
) -> (Result<ChartPayload, String>, Option<PathBuf>) {
    match reader {
        ReaderChoice::Duckdb => /* ufs/block/ufscustom match → DuckDB reader */,
        ReaderChoice::Async  => /* MinIO range-GET 기반 Arrow reader */,
        ReaderChoice::Sync   => /* spawn_blocking + 임시 파일 다운로드 (cleanup PathBuf 반환) */,
    }
}
```

핵심:

- **3개 path 분기 한 곳** — `#[cfg(feature)]` 가 dispatch 함수 안에 갇혀 핸들러는 깔끔
- **sync는 spawn_blocking + 임시 파일** — `parquet::ArrowReader`가 동기 + CPU-bound. cleanup 책임은 핸들러 (try_get_with 클로저 안)
- **async는 그대로 await** — `MinioParquetReader`가 자체 async, 임시 파일 없음
- **duckdb는 spawn_blocking + r2d2 pool + httpfs 직접 read** — DuckDB connection 은 sync API
- **single-flight** — 같은 키 동시 호출 시 빌더 1회만 실행, N-1 호출자는 같은 `Arc<Response>` 공유

`GetTraceStats` 도 동일 패턴 (`dispatch::build_stats_payload` + `caches.stats.try_get_with`).

응답 페이로드는 [l2-trace의 03-chart-arrow-ipc](/learn/l2-trace/03-chart-arrow-ipc/) 와 [8. Stats pipeline](/learn/l2-trace-rust/08-stats-pipeline/) 에서 자세히.

## StartRealtimeParsing — server-streaming + Cancel + panic watcher

```rust
async fn start_realtime_parsing(
    &self,
    request: Request<StartRealtimeParsingRequest>,
) -> Result<Response<Self::StartRealtimeParsingStream>, Status> {
    let req = request.into_inner();
    let job_id = Uuid::new_v4().to_string();

    let (tx, rx) = mpsc::channel(32);
    let cancel_token = CancellationToken::new();

    // 취소 토큰 등록 — StopParsing 이 발화시킴
    self.realtime_jobs.lock().await.insert(job_id.clone(), cancel_token.clone());

    let (notif_tx, mut notif_rx) = mpsc::channel(32);

    // 1) 파서 task — JoinHandle 보관해 watcher 가 결과 await
    let parser_handle = tokio::spawn(async move {
        run_realtime_parser(
            req.log_source, req.output_dir,
            req.window_seconds.unwrap_or(1),
            cancel_token, notif_tx,
        ).await
    });

    // 2) notification → ParquetFileReady 변환 (정상 흐름)
    let tx_for_stream = tx.clone();
    let job_id_for_task = job_id.clone();
    tokio::spawn(async move {
        while let Some(n) = notif_rx.recv().await {
            let msg = ParquetFileReady { /* ... */ };
            if tx_for_stream.send(Ok(msg)).await.is_err() { break; }
        }
    });

    // 3) parser 결과 watcher — Err / panic 둘 다 클라이언트로 전파
    let job_id_watch = job_id.clone();
    tokio::spawn(async move {
        match parser_handle.await {
            Ok(Ok(())) => {} // cancel 또는 EOF — 정상 종료
            Ok(Err(e)) => {
                let msg = format!("[Job: {}] 실시간 파싱 오류: {}", job_id_watch, e);
                let _ = tx.send(Err(Status::internal(msg))).await;
            }
            Err(join_err) => {
                let msg = if join_err.is_panic() {
                    format!("[Job: {}] 실시간 파서 panic: {}", job_id_watch, join_err)
                } else {
                    format!("[Job: {}] 실시간 파서 종료: {}", job_id_watch, join_err)
                };
                let _ = tx.send(Err(Status::internal(msg))).await;
            }
        }
    });

    Ok(Response::new(Box::pin(ReceiverStream::new(rx))))
}

async fn stop_parsing(
    &self,
    request: Request<StopParsingRequest>,
) -> Result<Response<StopParsingResponse>, Status> {
    let job_id = request.into_inner().job_id;

    // CancellationToken 발화 → run_realtime_parser 루프 break
    if let Some(token) = self.realtime_jobs.lock().await.remove(&job_id) {
        token.cancel();
    } else {
        return Err(Status::not_found("job not found"));
    }

    // 자동 머지가 끝날 때까지 잠시 대기 후 응답
    // (run_realtime_parser가 머지 후 is_final=true notification을 보냄)
    Ok(Response::new(StopParsingResponse {
        last_file_path: String::new(),
        total_events: 0,
        elapsed_seconds: 0.0,
    }))
}
```

핵심 세 가지:

- **CancellationToken** — `tokio_util::sync`. `cancel()` 호출 시 모든 `is_cancelled()` 체크 지점이 즉시 반응
- **2단계 채널** — 파서는 `mpsc::Sender<ParquetFileNotification>` 만 알면 됨 (gRPC와 무관). 서버 레이어가 proto 메시지로 변환. 파서 단위 테스트도 쉽게
- **Panic watcher** — `tokio::spawn` 의 `JoinHandle` 을 별도 task 가 `await` 하고 결과를 분기 (`Ok(Err)` / `is_panic()` / `Ok(())`). 옛날엔 parser panic 시 `eprintln!` 만 찍히고 stream 이 그냥 멈춰 클라이언트가 panic 사실을 모름 — 운영 진단 어려웠음. 이제 panic / 정상 에러 둘 다 `Status::internal` 로 즉시 전파

## ListFiles / GetJobStatus — 단순 unary

```rust
async fn list_files(
    &self,
    request: Request<ListFilesRequest>,
) -> Result<Response<ListFilesResponse>, Status> {
    let req = request.into_inner();
    let client = MinioAsyncClient::new(&MinioConfig {
        bucket: req.bucket,
        ..self.minio_config.clone()
    })?;
    let files = client.list_files(&req.prefix).await?;
    Ok(Response::new(ListFilesResponse { files }))
}
```

`GetJobStatus`도 비슷 — 그냥 in-memory `HashMap<job_id, JobState>` 조회.

## ReadParquet — Raw Data 페이지 + 키셋 무한 스크롤 + DuckDB 위임

처음 N행 streaming 으로 시작했지만, portal 의 Raw Data DataTable + 무한 스크롤 + 멀티 parquet 분석 요구가 추가되며 두 차례 확장되었음. 핸들러 본체는 320 → 110 줄로 줄고, 실제 read 는 `output/raw_rpc_duckdb.rs::query_raw_page` (DuckDB) 또는 `dispatch.rs::query_raw_page` (async/sync inmem fallback) 로 위임.

### 요청 필드 (확장 후)

```proto
message ReadParquetRequest {
  string source_bucket = 1;
  string source_parquet_path = 2;        // 단수 (호환). parquet_paths 비면 사용
  optional int32 max_records = 3;        // default 100
  optional FilterOptions filter = 4;

  // 페이지네이션 (offset 모드)
  optional int64 offset = 5;             // 음수는 0 클램프

  // 멀티 path
  repeated string parquet_paths = 6;     // set 시 source_parquet_path 무시

  // 시간 범위 (FilterOptions 와 별개 — chart/stats 패턴 호환)
  optional TimeRange time_range = 7;

  // 자동 감지 안 되는 옛 path 호환
  string trace_type = 8;                 // 빈 문자열이면 path 로 추론

  // 키셋 페이지네이션 (cursor_time 채우면 OFFSET 대신 keyset 모드)
  optional double cursor_time = 9;
  optional uint64 cursor_line_number = 10;  // tiebreak
}
```

응답 첫 메시지에만 `columns` (UI 컬럼 순서) 와 `schema_version` 이 채워지고, 마지막 row 이후엔 `next_cursor_time` / `next_cursor_line_number` 가 채워져 다음 페이지를 그대로 다시 보내면 됨. **둘 다 optional** — `time = 0.0` 이 유효한 cursor 일 수 있어 default 0 으로 "끝" 을 표현하면 trace 시작이 0.0 으로 normalize 된 케이스에서 무한 스크롤이 멈춰버림 (6f04f17 fix).

### Keyset pagination — OFFSET 누적 스캔 회피

OFFSET 모드: 깊은 페이지에서 매번 0..N 까지 스캔 후 버림 → O(skip + page).
Keyset 모드: `WHERE (sort_col > $t OR (sort_col = $t AND line_number > $ln)) ORDER BY (sort_col, line_number) LIMIT page` → 매 페이지가 항상 O(page).

`line_number` 가 tiebreak 인 이유: time 동값 row 가 여러 개일 때 페이지 경계에서 중복/누락이 발생하면 안 되므로, parquet 에 1-based 라인 번호로 채워둔 `line_number` 컬럼을 보조 키로 사용.

### DuckDB 위임 — `read_parquet([s3://...])` 한 큐리

```rust
// output/raw_rpc_duckdb.rs (요지)
pub fn query_raw_page(
    pool: &DuckDbPool,
    paths: &[String],
    trace_type: &str,
    filter: Option<&FilterOptions>,
    time_range: Option<TimeRange>,
    page: PageSpec, // offset OR keyset(cursor_time, cursor_line_number)
) -> Result<RawPage, Status> {
    // SELECT to_json(struct_pack(...)) AS record_json,
    //        sort_col AS cursor_time, line_number AS cursor_ln
    // FROM read_parquet([s3://...], union_by_name=True)
    // WHERE filter / time_range / (keyset 시) (sort_col > $t OR ...)
    // ORDER BY sort_col, line_number
    // LIMIT max_records [OFFSET offset]
}
```

- `to_json(struct_pack(...))` 로 in-memory serde 와 byte-level 호환 record_json 발급
- 옛 parquet 호환: `aligned`/`start_qd`/`end_qd`/`ctoc`/`ctod` 컬럼 부재 시 `has_*` 검사 + FALSE/0 리터럴 fallback (binder error 방지, 6f04f17 fix)
- `line_number` COALESCE 가 INT/UBIGINT 혼합 시 좁혀져 row.get mismatch → `::UBIGINT` 명시. `hwqid` 옛 UInt32 / 신규 Int32 통일 → `::INTEGER` cast

### inmem fallback + 정렬

DuckDB 비활성 환경 (또는 멀티 path 가 inmem 으로 떨어지는 경우) 은 `dispatch.rs::query_raw_page` 가 RecordBatch → row 변환 + 같은 정렬·필터·cursor 필터 적용. parquet 자체는 stats RPC 가속용으로 `(action, opcode, time)` 키로 저장돼 있으므로, 응답 직전 `time` (UFS/Block) 또는 `start_time` (UFSCUSTOM) 으로 재정렬해야 UI 가 "complete 만 가득" 같은 현상을 안 봄 (acc4942 fix).

### 임시 파일 cleanup

`?` early-return 시 다운로드된 임시 parquet 들이 정리되지 않던 버그 — `temp_files` 를 outer scope 로 끌어내고 `result.await` 후 정상/에러 모두에서 일괄 cleanup. download 시도 직전 push 로 부분 다운로드 잔여물도 정리 (c4357489 fix).

## GetRawLogLines — 원본 .log 라인 lazy 조회

ReadParquet row 의 `line_number` (1-based, 0 = 미상) 를 키로 원본 .log 의 해당 라인 ± 컨텍스트를 가져오는 RPC. raw log 뷰어 / 디버깅 용.

### 요청·응답

```proto
message GetRawLogLinesRequest {
  string upload_bucket = 1;             // 원본 .log 가 있는 MinIO 버킷
  string upload_path = 2;               // 원본 .log 경로
  repeated uint64 line_numbers = 3;     // 정렬 안 돼도 됨
  optional uint32 context_before = 4;   // default 0, max 50 (서버 클램프)
  optional uint32 context_after = 5;    // default 0, max 50
}

message RawLogBlock {
  uint64 target_line_number = 1;
  uint64 start_line_number = 2;         // target - context_before 또는 1 클램프
  repeated string lines = 3;            // 파일 범위 밖이면 빈 배열
}
```

### lazy single-pass 스캔

초기 구현은 `tokio::fs::read_to_string` 으로 `.log` 전체 적재 → 멀티 GB OOM 위험. 코드 리뷰 후 `read_raw_log_blocks` 헬퍼로 교체 — `BufReader::lines().next_line()` 단일 패스 스캔, target ± context 라인만 보관. **복수 target 도 한 번의 스캔**으로 처리. 단위 테스트 4건 (`grpc/server.rs::tests::raw_log_*`) 추가 (c4357489).

운영 의미: 1억 row, 수십 GB .log 환경에서도 메모리 사용이 page 캐시 + context window (≤ 100라인 × N target) 로 한정.

## ExportXlsx — 분할 xlsx + ZIP + Streaming Progress

ConvertToCsv 의 분할 정책을 그대로 따라 parquet → 1M row/file 분할 xlsx → (분할 시만) ZIP → MinIO 업로드. server-streaming progress 로 단계별 진행 메시지 송신.

### 요청 + 진행 stage

```proto
message ExportXlsxRequest {
  string source_bucket = 1;
  repeated string parquet_paths = 2;         // 멀티 path (시간순 정렬 호출자 책임)
  string trace_type = 3;
  optional FilterOptions filter = 4;
  optional TimeRange time_range = 5;
  string target_bucket = 6;
  string target_zip_path = 7;                // {user}/{jobId}/exports/{uuid}.zip
  optional uint32 file_row_limit = 8;        // default 1_000_000, max 1_048_575 (Excel sheet 한도)
  optional bool include_stats_sheet = 9;     // default true
  optional string filename_prefix = 11;      // default = trace_type
}

enum XlsxStage { XLSX_UNKNOWN; XLSX_DOWNLOADING; XLSX_FILTERING;
                 XLSX_WRITING; XLSX_ZIPPING; XLSX_UPLOADING;
                 XLSX_COMPLETED; XLSX_FAILED; }
```

### 진행률 단계 (실측 % 가이드)

| % | message 예시 | 의미 |
|---|---|---|
| 5–20 | `Downloaded n/m parquet …` | 멀티 path 다운로드 |
| 25 | `Reading parquet rows` | iter_*_from_parquet 시작 |
| 35 | `Read N {trace_type} rows` | 읽기 완료 |
| 42 → 48 | `Applying filters` → `Filtered N → M rows` | filter 적용 |
| 50 | `Sorting M rows by time` | parquet `(action, opcode, time)` 순서 → time 정렬 (e5b4677) |
| 55 → 80 | `Writing xlsx (M rows)` → `Wrote N file(s)` | chunk-local time sort + constant_memory write |
| 85 → 90 | `Compressing N files` → `Zipped to X bytes` | **단일 파일이면 skip** (a5733b2) |
| 93 | `Uploading {xlsx\|zip} to MinIO (X.XX MB)` | upload |
| 98 | `Generating download URL` | presigned URL |
| 100 | `Completed` | output_zip_path 채움 |

### 파일명 규약 (9e569ed)

`{filename_prefix}_{startTime}_{endTime}.xlsx` — chunk 의 첫/마지막 row 의 `time` (UFS/Block) 또는 `start_time` (UFSCUSTOM) 사용. 소수점 3자리(ms 정밀도), `.` → `_` 치환. chunk 1 = 가장 이른 시간, chunk N = 가장 나중 시간이 자연스럽게 단조증가.

### Streaming write — peak memory ≈ 1 chunk

핵심 구현 (`output/xlsx.rs::save_*_to_xlsx_split_streaming`):
- `iter_*_from_parquet` (`output/reader.rs`) 가 RecordBatch → row-by-row iterator
- `chunk(file_row_limit)` 단위 buffer + chunk-local time sort
- `rust_xlsxwriter` 0.94 `constant_memory` worksheet → 디스크 직접 flush
- `spawn_blocking + mpsc::blocking_send` 로 chunk progress forward

→ 1억 row · 수십 GB 입력에서도 peak memory 가 1 chunk (1M row) 분량으로 한정.

### 운영 함정 — 단일 파일은 ZIP 안 함

xlsx 자체가 zip 포맷이라 추가 압축 의미 없음 + 1M row 이내 (대부분 케이스) 는 파일 1개 → ZIP 으로 묶을 이유 없음. 분할이 일어난 경우만 ZIP. portal 이 보낸 `target_zip_path` 의 확장자를 결과에 맞춰 보정 (`.xlsx` 또는 `.zip`) 하고, 응답의 `output_zip_path` 가 실제 업로드된 path 입니다 (a5733b2).

또한 `target_client.upload_file(local, remote)` 시그니처에 인자 순서가 뒤바뀌어 있던 버그 (target_zip_path 가 local 자리에 들어가 "no such file") 도 같이 정정.

## RPC별 사용 시점

| 사용 주체 | 호출 RPC | 시나리오 |
|---|---|---|
| portal upload UI | `ProcessLogs` | 로그 파일 업로드 → @Async parsing |
| portal trace UI | `GetChartData`, `GetTraceStats` | 차트/표 렌더링 |
| portal trace UI (Raw Data) | `ReadParquet` (keyset cursor) | 무한 스크롤 DataTable |
| portal trace UI (Raw Log 뷰어) | `GetRawLogLines` | row 클릭 → 원본 .log ± 컨텍스트 표시 |
| portal trace UI (다운로드) | `ExportXlsx` | "xlsx 로 내보내기" 버튼 |
| portal trace UI | `ListFiles` | 디버깅/리스트 화면 |
| agent (Go) | `StartRealtimeParsing`, `StopParsing` | 라이브 trace 수집 |
| portal admin | `GetJobStatus` | 변환 진행률 폴링 |
| 운영자 CLI | `ConvertToCsv` | parquet → CSV 추출 |

`l2-trace`의 portal 흐름은 위 표에서 마지막 줄을 빼면 다 호출합니다.

## 클라이언트 — `--grpc-client`로 직접 호출

`grpc/client.rs`가 같은 서비스를 호출하는 CLI 클라이언트도 제공.

```bash
# 서버
./trace --grpc-server --port 50053

# 클라이언트 명령
./trace --grpc-client process \
    --server localhost:50053 \
    --source-bucket trace --source-path raw/job1.log \
    --target-bucket trace --target-path parquet/job1 \
    --log-type ufs

./trace --grpc-client list --server localhost:50053 --bucket trace --prefix parquet/

./trace --grpc-client realtime --server localhost:50053 \
    --source-path /tmp/trace.log --output-dir /tmp/realtime --log-type ufs --window 1

./trace --grpc-client stop --server localhost:50053 --job-id <UUID>
```

서버 디버깅이나 portal 없이 통계 확인할 때 유용.
