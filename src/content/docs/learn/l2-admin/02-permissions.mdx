## "17 권한이 평면인 이유"

역할(role) 기반 권한(RBAC)이 흔한 설계지만 MOVE는 **유저별 직접 부여**:

| 방식 | 장점 | 단점 |
|---|---|---|
| RBAC (roles + role_permissions + user_roles) | 공통 역할 재사용 | 역할 설계 필요 · DB join 3개 |
| **유저 직접 (현재)** | 단순 · 개별 예외 자연 | 권한 템플릿 부재 |

MOVE 맥락:

- 사용자 수 수십~수백 명 (엔터프라이즈급 수천이 아님)
- 권한 조합이 **유저마다 상당히 다름** (프로젝트 접근 권한 등)
- RBAC의 "역할 설계" 비용이 실질 이익 > **수작업 변경 부담**

**"작은 조직에선 RBAC 오버엔지니어링"** 의 전형. 조직 커지면 role 레이어 추가 가능.

## 17 권한 복원 프론트 렌더

```
GET /api/admin/permissions/defaults
 → [{key: 'menu:dashboard', label: '대시보드', category: 'menu'},
    {key: 'action:excel-export', label: 'Excel 내보내기', category: 'action'}, ...]

GET /api/admin/users/{id}/permissions
 → {'menu:dashboard': true, 'action:excel-export': false, ...}

// 프론트 렌더
{#each defaults as p}
  <Checkbox
    label={p.label}
    checked={permissions[p.key]}
    onChange={(v) => permissions[p.key] = v}
  />
{/each}

// 저장 시
PUT /api/admin/users/{id}/permissions
Body: {'menu:dashboard': true, 'action:excel-export': false, ...}
```

- **정의와 값이 분리** → 정의 변경이 값 마이그레이션 불필요
- prefix 기반 카테고리 필터로 UI 분할 (menu 섹션 vs action 섹션)

## Head Access — 권한의 두 번째 축

17 권한이 "기능 접근"이면 Head Access는 "**데이터 접근**":

- 유저 A: menu:slots=true 지만 HeadConnection {1,2,3}만 접근 가능
- 유저 B: menu:slots=true + HeadConnection 제한 없음

**슬롯 L2의 `filterByUserHeadAccess`**가 이 필터링을 구현. Admin UI에서 유저당 허용된 head id 리스트를 UPDATE.

**두 권한 축의 교차**:

| | menu:slots=true | menu:slots=false |
|---|---|---|
| head [1,2,3] 허용 | head 1/2/3의 slots만 접근 | menu 접근 차단 |
| head 전체 허용 | 모든 slots 접근 | menu 접근 차단 |

**기능 × 데이터** 이중 필터. 운영 복잡도 약간 상승이지만 세밀 제어 가능.

## 직접 해보기

- [ ] Admin UI에서 한 유저에게 menu:slots=true만 주고 action:slot-init=false로 → 슬롯 페이지는 보이지만 초기화 버튼은 비활성화인지
- [ ] `user_permissions` 테이블에서 특정 유저의 row 수가 17인지 (모든 키에 대해 저장되는지) 또는 명시한 것만 저장되는지
