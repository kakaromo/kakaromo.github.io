## 30초 요약

유저 자기 권한 신청 워크플로:

1. 유저: `POST /api/auth/request-access` — reason 적어 신청
2. `PermissionRequest` row 생성 (PENDING)
3. Admin: "대기 중" 목록에서 확인 → 권한 부여 UI 열기
4. Admin: 체크박스 설정 → "**권한 저장 & 승인**" 버튼
5. `PUT /api/admin/permission-requests/{id}/approve` — **비관적 락 + 3 write**(permissions + head access + status) 원자 처리

## 원자성의 중요성

이 3 write가 **하나라도 실패하면 전체 취소**:

- permissions 저장 → 성공
- head access 저장 → 실패 (네트워크 또는 DB)
- status 전이 → 실행 안 됨 (앞 실패로 rollback)

**그 결과**: 유저는 권한이 부분적으로만 적용된 상태가 되지 않음. 다시 승인 재시도 필요하지만 **일관성 유지**.

Spring의 `@Transactional`이 이 원자성을 제공. Controller가 트랜잭션 매니저 지정 없이 기본 `portalTransactionManager` 사용.

## 비관적 락 vs 낙관적 락

PermissionRequest에 **비관적 락(Pessimistic Write)** 사용:

| 방식 | 특성 | MOVE 적합성 |
|---|---|---|
| **비관적** | SELECT FOR UPDATE → 차단 | Admin 동시 승인 드물어 경합 비용 낮음 |
| 낙관적 | version 컬럼 → 충돌 시 재시도 | 재시도 로직 필요, Admin UX 복잡 |

**승인은 빈도 낮음** → 락 대기가 짧음 → 비관적이 적합. 슬롯 StateStore 같은 고빈도 쓰기에는 낙관적 또는 lock-free 구조 필요.

## 요청 생성 - `POST /auth/request-access`

유저 측 엔드포인트는 **Auth 컨트롤러**에 있음(AdminController 아님):

```java
@PostMapping("/request-access")
public Map<String, Object> requestAccess(@RequestBody Map<String, String> body,
                                          HttpSession session) {
    PortalUser user = (PortalUser) session.getAttribute("user");
    if (user == null) return Map.of("success", false, "error", "로그인 필요");
    String reason = body.get("reason");
    PermissionRequest req = PermissionRequest.builder()
        .userId(user.getId())
        .reason(reason)
        .status("PENDING")
        .build();
    permissionRequestRepository.save(req);
    return Map.of("success", true, "id", req.getId());
}
```

**관리 vs 사용자** 분리 — Admin은 `/api/admin/*`, 유저 신청은 `/api/auth/*`. 보안 경계가 URL 레벨에서 자연스럽게 그어짐.

## reject 경로 — 거절 승인 대칭

```java
@PutMapping("/permission-requests/{id}/reject")
public Map<String, Object> rejectPermissionRequest(@PathVariable Long id,
        @RequestBody Map<String, String> body) {
    PermissionRequest req = permissionRequestRepository.findByIdForUpdate(id).orElseThrow();
    if (!"PENDING".equals(req.getStatus())) return Map.of("success", false);
    req.setStatus("REJECTED");
    req.setReviewedAt(LocalDateTime.now());
    req.setReviewComment(body.get("comment"));
    return Map.of("success", true);
}
```

- 비관적 락 + PENDING 체크 동일
- **권한 변경 없음** — 유저 상태 그대로, request 상태만 REJECTED로
- `reviewComment`에 이유 — 유저에게 알려주는 정보

**승인 = 권한 변경 + 상태 전이**, **거절 = 상태 전이만**. 원자성 같고 범위 다름.
