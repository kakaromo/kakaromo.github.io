## 왜 18 탭 단일 페이지인가

대안과의 비교:

| 전략 | 장단점 |
|---|---|
| **단일 +page.svelte** (현재) | 전역 상태 공유 쉬움 · 검색/탭 전환 빠름 · **파일이 2652줄** |
| `/admin/users`, `/admin/servers` 등 분리 | 파일 작음 · 전환 시 데이터 재로드 · URL 공유 가능 |
| 각 탭이 별도 컴포넌트 | 파일 분할 · Props drilling · 컴포넌트 경계 설계 비용 |

MOVE 선택 이유:

- Admin 페이지는 **자주 쓰이는 탭 2-3개만** 실사용 (servers, users, permission-requests)
- 한 세션에 여러 탭을 빠르게 오가는 워크플로 (서버 설정 후 유저 권한)
- **Svelte 컴파일러가 조건부 렌더만 처리** — 런타임 비용 낮음
- 대신 **maintainability 악화** — 2652줄 파일은 grep 없이 특정 탭 찾기 어려움

**2652줄의 비용 인정**. 이 점이 후속 리팩토링 후보로 메모리에 있음.

## "God Controller"의 장단

**Bad smell로 보일 수 있지만**:

- 각 메서드 자체는 ~5줄로 간단
- 의존성 주입만 많음(15개), 로직 결합 없음
- REST 클라이언트에 `/api/admin/*` 네임스페이스 하나만 노출

**"God Controller = 자동으로 나쁜 것" 아님**. 로직이 얇고 책임이 "관리 엔드포인트 라우팅"으로 단일하면 수용 가능. 이건 Controller 라기보다 **API facade**.

## 거대 페이지의 실제 비용

개발 중 마주치는 어려움:

- **git blame**: 한 파일 blame이 너무 복잡해 어느 탭이 언제 추가됐는지 불명확
- **PR 리뷰**: 파일 diff 100+ 줄이 여러 탭 섞여 있으면 리뷰 어려움
- **충돌**: 다른 개발자가 다른 탭 수정 시 같은 파일 → rebase 충돌 빈번
- **에디터 성능**: VS Code의 syntax highlighter가 2652줄을 파싱하면서 약간 느려짐

**리팩토링 후보**: 탭별 컴포넌트 분리 (`AdminServersTab.svelte`, `AdminUsersTab.svelte`…). slots 페이지 리팩토링과 같은 패턴. 메모리에 이 의도가 기록됨.
