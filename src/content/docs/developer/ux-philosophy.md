MOVE의 프론트엔드는 **토스(Toss) 디자인 철학**을 기반으로 설계합니다. 복잡한 엔지니어링 도구이지만, 사용자가 생각하지 않아도 다음 행동이 자연스럽게 이어지는 경험을 목표로 합니다.

---

## 1. 핵심 원칙

### 한 화면에 하나의 목적

화면이 여러 역할을 하면 사용자가 "지금 뭘 해야 하지?"를 고민합니다. 각 뷰는 하나의 목적만 가집니다.

| 원칙 | 좋은 예 | 나쁜 예 |
|------|---------|---------|
| 뷰 분리 | Pre-Command 시트: main(등록/실행) → manage(편집/삭제) → edit(생성/수정) | 하나의 화면에 등록, 편집, 삭제, 실행 버튼이 모두 나열 |
| 단계별 진행 | 슬롯 선택 → 시트 열기 → 템플릿 선택 → 등록/실행 | 모든 옵션이 한 번에 노출 |

### 맥락을 유지하라

사용자가 지금 어디에 있고, 무엇을 하고 있는지 항상 보여줍니다.

| 패턴 | 구현 |
|------|------|
| **선택 상태 표시** | Pre-Command 시트 상단에 "Slot#3, Slot#5 선택됨" |
| **등록 상태 피드백** | ✅ 체크 아이콘 + primary 색상으로 즉시 구분 |
| **뒤로가기** | ← 버튼으로 이전 뷰 복귀, 현재 위치 명확 |
| **플로팅 카드** | 실행 중 진행 상황이 다른 작업을 가리지 않고 항상 보임 |

### 결과를 즉시 보여줘라

행동의 결과를 사용자가 찾아다니지 않도록 합니다.

| 패턴 | 구현 |
|------|------|
| **등록 즉시 반영** | 등록하면 슬롯 카드에 ⚡ 뱃지가 바로 표시 |
| **실행 진행** | 플로팅 카드에서 슬롯별/명령어별 실시간 상태 |
| **실패 자동 펼침** | 완료 시 실패한 슬롯이 자동으로 열려서 에러 즉시 확인 |
| **완료 토스트** | 카드를 닫으면 "3 성공 · 1 실패" 요약 토스트 |

### 불필요한 선택지를 줄여라

선택지가 많으면 결정 비용이 올라갑니다. 자동화할 수 있는 건 자동화합니다.

| 패턴 | 구현 |
|------|------|
| **adb -s 자동 삽입** | 사용자가 usbId를 기억할 필요 없음 |
| **SSH 대상 자동 결정** | setLocation에서 VM 이름 추출 |
| **상태별 자동 동작** | init 진입 시 등록된 명령어 자동 실행 |
| **메뉴 단순화** | 4개 서브메뉴 → 1개 항목 (시트에서 전부 처리) |

## 2. 컴포넌트 설계 패턴

### 시트(Sheet) — 슬라이드 패널

메인 콘텐츠 위에 오버레이되는 패널. 현재 페이지를 떠나지 않고 상세 작업을 수행합니다.

**사용 시점:**
- CRUD 작업 (생성, 편집, 삭제)
- 상세 정보 조회
- 설정 변경

**설계 규칙:**
- 시트 안에서 뷰 전환 (list → edit) 시 뒤로가기(←) 제공
- 시트 닫기(X 또는 외부 클릭)는 데이터를 변경하지 않음
- 저장/등록은 명시적 버튼 클릭으로만

```
┌─ Sheet ──────────────────────────┐
│ ← 제목                    [X] │  ← 헤더: 뒤로가기 + 닫기
│────────────────────────────────│
│                                │
│  (스크롤 가능한 본문)           │  ← 본문: flex-1 overflow-y-auto
│                                │
│────────────────────────────────│
│ [취소]              [저장] │  ← 푸터: 액션 버튼
└────────────────────────────────┘
```

### 플로팅 카드 — 비차단 진행 표시

장시간 실행되는 작업의 진행 상황을 메인 UI를 가리지 않고 표시합니다.

**사용 시점:**
- SSH 명령어 실행 (Pre-Command)
- 벤치마크 실행 (Agent)
- 파일 다운로드

**설계 규칙:**
- 우하단 고정 (`fixed bottom-4 right-4`)
- 접기/펼치기로 공간 조절
- 실행 중: 닫기 = 중단 확인, 완료 후: 닫기 = 결과 토스트
- 부모 컴포넌트에서 상태 관리, 카드는 순수 표시

**상태에 따른 프로그레스 바:**

| 상태 | 색상 | 동작 |
|------|------|------|
| 실행 중 | 파란색 | 진행률에 따라 채워짐 |
| 성공 완료 | 초록색 | 풀 바 |
| 실패 포함 | 빨간색 | 풀 바 |

### 슬롯 카드 — 정보 밀도 최적화

170×150px 카드에 상태, 제품 정보, 실행 상태를 압축합니다.

**시각 계층:**
1. **헤더** (최상단): Slot 번호 + 위치 + 뱃지(⚡메모📄)
2. **상태** (가장 눈에 띄게): 색상 + 아이콘 + 텍스트
3. **보조 정보**: 제품, FW, 파일시스템 (muted 색상)
4. **워터마크**: 활성 상태에서만 배경 아이콘 + shimmer bar

**뱃지 시스템:**

| 뱃지 | 위치 | 의미 |
|------|------|------|
| ⚡ | 헤더 우측 | Pre-Command 등록됨 |
| 📄 | 헤더 우측 | 메모 있음 |
| 🟢 | 헤더 우측 | 연결 상태 |

**툴팁**: hover 시 모든 상세 정보를 키-값 테이블로 표시 (Pre-Cmd 이름 포함)

### 컨텍스트 메뉴 — 깊이 최소화

우클릭 메뉴는 **2단계 이하**로 유지합니다.

| 원칙 | 설명 |
|------|------|
| 1단계 | 즉시 실행 가능한 명령 (Start Test, Stop Test, Terminal) |
| 2단계 | 서브메뉴 (Prepare Test > Set TR, Set TC, Pre-Command) |
| 금지 | 3단계 서브메뉴. 항목이 많으면 시트로 분리 |

**Pre-Command 사례**: 처음에는 Prepare Test > 사전 명령어 > 등록/해제/즉시 실행/관리 (3단계)였으나, 시트 통합으로 Prepare Test > Pre-Command (2단계)로 단순화.

## 3. 상호작용 패턴

### hover 효과

모든 인터랙티브 요소에 hover 피드백을 제공합니다.

```css
/* 카드형 요소 */
transition-all duration-150
hover:bg-accent/50 hover:border-primary/30 hover:shadow-sm
active:scale-[0.99]

/* 카드 내 제목 */
group-hover:text-primary transition-colors

/* 액션 버튼 (hover 시만 노출) */
opacity-0 group-hover:opacity-100 transition-opacity
```

### 토스트 메시지

행동의 결과를 간결하게 알립니다.

| 상황 | 타입 | 메시지 예시 |
|------|------|-------------|
| 등록 성공 | success | `3개 슬롯에 "tiotest 설치" 등록` |
| 해제 성공 | success | `Pre-Command 해제 완료` |
| 실행 완료 (성공) | success | `tiotest 설치: 3 성공` |
| 실행 완료 (실패 포함) | error | `tiotest 설치: 2 성공 · 1 실패` |
| 실행 중단 | info | `Pre-Command 실행이 중단되었습니다` |

### 빈 상태(Empty State)

데이터가 없을 때 다음 행동을 안내합니다.

| 요소 | 설명 |
|------|------|
| 설명 텍스트 | "등록된 명령어가 없습니다" |
| 예시 | 실제 사용 예시를 보여줘서 기능 이해 도움 |
| CTA 버튼 | "첫 명령어 만들기" — 바로 생성 뷰로 이동 |

## 4. 반응성(Reactivity) 설계

### Svelte 5 runes 주의사항

| 패턴 | 권장 | 비권장 |
|------|------|--------|
| 자식 → 부모 통신 | props 콜백 (`onExecute`, `onClose`) | `bind:this` + `export function` |
| 상태 공유 | 부모에서 `$state` 관리 → props 전달 | 자식 내부 `$state` + 외부 호출 |
| 배열 업데이트 | `array = [...array, item]` (새 참조) | `array.push(item)` (변경 감지 안 됨) |
| 객체 업데이트 | `obj = { ...obj, key: value }` (새 참조) | `obj.key = value` (깊은 변경 놓칠 수 있음) |

**Pre-Command 사례**: 처음에 `bind:this` + `export function handleEvent()`로 플로팅 카드를 업데이트했으나, Svelte 5에서 동작하지 않아 부모에서 `PreCommandProgress` 객체를 직접 관리하는 방식으로 변경.

### SSE 스트리밍 패턴

```typescript
// 부모에서 상태 직접 관리
let progress = $state<Progress>({ ... });

// SSE 이벤트 수신 시 새 객체 할당 (반응성 보장)
function onEvent(type, data) {
    progress = { ...progress, /* 업데이트 */ };
}

// 자식은 props로 수신 (순수 표시)
<FloatingCard {progress} />
```

## 5. 접근성 체크리스트

새로운 UI 컴포넌트를 만들 때 확인합니다:

- [ ] 모든 버튼에 `title` 또는 레이블 텍스트
- [ ] 아이콘만 있는 버튼에 의미를 전달하는 `title`
- [ ] disabled 상태에 `opacity-50` 시각 피드백
- [ ] hover/active 상태 전환 애니메이션 (`transition-colors`, `transition-all`)
- [ ] 긴 텍스트에 `truncate` (말줄임표)
- [ ] 스크롤 필요한 영역에 `overflow-y-auto` + `max-h-*`
- [ ] 다크 모드 대응 (Tailwind `dark:` 접두사)

## 6. 적용 사례: Pre-Command

Pre-Command 기능은 위 원칙들을 종합적으로 적용한 대표 사례입니다.

### Before (초기 설계)

```
문제점:
- 컨텍스트 메뉴 3단계: Prepare Test > 사전 명령어 > 등록/해제/즉시 실행/관리
- 등록과 실행이 별도 흐름
- 슬롯에 뭐가 등록되어 있는지 안 보임
- 실행 완료 후 결과 확인 불편
```

### After (토스 철학 적용)

```
개선:
- 메뉴 1개: Prepare Test > Pre-Command → 통합 시트
- 시트 안에서 등록 상태 확인 + 등록/해제 + 즉시 실행 + 관리
- 슬롯 카드 ⚡ 뱃지 + 툴팁으로 등록 상태 즉시 확인
- 플로팅 카드: 프로그레스 바 + 실패 자동 펼침 + 완료 토스트
- 빈 상태: 예시 명령어로 진입 장벽 감소
```

| 원칙 | 적용 |
|------|------|
| 한 화면 한 목적 | 시트 3단 뷰 (main/manage/edit) |
| 맥락 유지 | 선택된 슬롯 표시, 등록 상태 체크 |
| 즉시 결과 | ⚡ 뱃지, 실패 자동 펼침, 토스트 |
| 선택지 최소화 | adb -s 자동, SSH 대상 자동, 메뉴 단순화 |
